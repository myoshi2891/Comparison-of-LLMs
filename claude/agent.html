📋 CLAUDE.md に書くべきこと / 書かないべきこと
✅ 書くべき内容
  • プロジェクト概要（1〜3文）
  • 技術スタック・主要ライブラリ
  • サブエージェントのルーティングルール
  • 禁止コマンド・危険な操作の明示

  • モノレポ構造の説明
✗ 書かないべき内容
  • コードスタイル・フォーマットルール (linter に委ねる)
  • 特定タスクにしか使わない詳細手順
  • 長大なコードスニペット
  • @ファイル参照（毎回全体が埋め込まれる）
  • 「〜しないで」の禁止形のみの指示
  • 修正履歴・変更ログ

CLAUDE.mdMarkdown
# PROJECT: my-saas-app ## Overview Next.js 15 + Supabase + Stripe。マルチテナント SaaS。 本番環境は Vercel、DBは Supabase (PostgreSQL)。 ## Tech Stack - Frontend: Next.js 15 (App Router), TypeScript, Tailwind CSS - Backend: Supabase Edge Functions (Deno) - Auth: Supabase Auth + Row Level Security - Payment: Stripe Checkout / Webhooks ## Build & Test - Build: `pnpm build` - Test: `pnpm test` (Vitest) - Lint: `pnpm lint` (ESLint + Prettier — **linterに任せること**) - DB型生成: `pnpm supabase gen types` ## Sub-Agent Routing Rules **Parallel dispatch** (全条件を満たす場合のみ): - タスクが 3件以上で互いに独立 - 共有ファイルなし・状態が独立 - ドメイン境界が明確 (frontend / backend / db) **Sequential dispatch** (いずれかの条件で): - タスクに依存関係がある (B は A の出力が必要) - 共有ファイル / 共有状態がある - スコープが不明確な場合 ## Domain Boundaries - **frontend-agent** → app/, components/, styles/ のみ - **backend-agent** → supabase/functions/, lib/server/ のみ - **db-agent** → supabase/migrations/, schema.sql のみ ## Forbidden Operations - `supabase db reset` は絶対に実行しない (本番データ消去) - `--force` フラグは使わない; 代わりに `--dry-run` で確認 - .env.production は読み書き禁止 ## Key Docs - 詳細設計: docs/architecture.md - DB スキーマ: supabase/schema.sql - API仕様: docs/api-spec.md
💡
サブディレクトリ CLAUDE.md の活用 frontend/CLAUDE.md や backend/CLAUDE.md に分割すると、そのディレクトリで作業する際だけ追加コンテキストが読み込まれます。グローバル CLAUDE.md は薄く、局所設定は局所ファイルへ。
3サブエージェント定義 .claude/agents/*.md

サブエージェントは YAML フロントマター + システムプロンプト本文 で構成されます。メインエージェントの system prompt は継承されません。フロントマターで定義できるフィールドは以下の通りです。

フィールド必須値の例説明
name必須code-reviewerエージェント識別子（ハイフン区切り推奨）
description必須"PRのコードレビューを実施する"Claude がどのタスクでこのエージェントを呼ぶかの判断基準
tools任意Read, Glob, Grep許可ツール。省略するとメインと同じツールセットを継承
model任意sonnet / haiku / opus使用モデル。コスト最適化に活用
memory任意user / project / localMEMORY.md のスコープ
📄 実践的なサブエージェント定義テンプレート
.claude/agents/code-reviewer.mdMarkdown + YAML
--- name: code-reviewer description: "コードレビューを実施する。PR作成後・コード変更後に自動で呼び出される。 セキュリティ・パフォーマンス・型安全性・ベストプラクティスをチェックする。" tools: Read, Glob, Grep # Read-only: 書き込みは行わない model: sonnet # コスト効率重視 memory: project # プロジェクト固有のパターンを記憶 --- ## Role あなたはシニアエンジニアのコードレビュアーです。 コードの品質・セキュリティ・パフォーマンスを厳しくチェックします。 ## Review Checklist レビュー開始前に **必ず MEMORY.md を確認** し、過去のパターンを参照してください。 ### Security - [ ] SQL インジェクション / XSS リスクがないか - [ ] 認証・認可が適切に実装されているか - [ ] 秘密情報がハードコードされていないか ### Performance - [ ] N+1 クエリが発生していないか - [ ] 不要な再レンダリングがないか (React) - [ ] 重い処理が非同期化されているか ### Code Quality - [ ] 型定義が適切か (TypeScript) - [ ] エラーハンドリングが漏れていないか - [ ] テストが書かれているか ## Output Format 以下の形式でレポートを出力してください： ``` ## Code Review Report ### 🔴 Critical (要対応) ### 🟡 Warning (推奨対応) ### 🟢 Good (良い点) ``` ## Memory Management レビュー完了後、新しいパターンや繰り返し発生する問題を MEMORY.md に追記してください。 MEMORY.md が 200行を超えた場合は要約・整理してください。

--- name: pm-spec description: "新機能の要求仕様書を作成する。機能要件・非機能要件・受け入れ条件を定義し ステータスを READY_FOR_ARCH に設定する。" tools: Read, Write, Edit model: sonnet --- # PM Spec Agent のシステムプロンプト ...
.claude/agents/architect-review.mdYAML frontmatter
--- name: architect-review description: "pm-spec が READY_FOR_ARCH になった後に呼び出す。 プラットフォーム制約・コスト・パフォーマンスを考慮した設計を行い ADR を作成する。" tools: Read, Write, Edit, Bash model: opus # 複雑な設計判断はOpusを使用 ---
🚨
サブエージェントのツール権限は最小化せよ サブエージェントはメインエージェントのsystem promptを継承しません。toolsフィールドで明示的に権限を絞らないと、デフォルトで全ツールが有効になります。Read-onlyでよいエージェントには必ず tools: Read, Glob, Grep のみを指定してください。
4MEMORY.md — エージェントの学習・記憶ファイル

MEMORY.md はサブエージェントが「経験から学んだパターン」を蓄積するファイルです。200行を上限とし、それを超えたら要約・整理を行う設計が公式推奨です。LLMはステートレスなため、これが唯一の「記憶」となります。

.claude/memory/code-reviewer/MEMORY.mdMarkdown
# Code Reviewer Memory ## Recurring Patterns (繰り返し発生する問題) ### Security - [2026-10-10] `src/lib/db.ts`: ユーザー入力を直接SQLに結合していた → Prepared Statement を使うよう修正済 - [2026-10-12] `app/api/**/*.ts`: 認証チェックが middleware でのみ行われ、handler内で再検証なし → 二重チェック必須 ### Performance - [2026-10-08] Supabase の `.select('*')` が多用されている → 必要カラムのみ指定する慣習を徹底 - [2026-10-14] React Server Component で useClient が不必要に使われている → 'use client' は最小化 ### Code Conventions (このプロジェクト固有) - エラーは必ず `Result` 型でラップする (throw 禁止) - API route は `zod` でリクエストバリデーションが必須 - Server Action の命名は `action` プレフィックス (例: `actionCreateUser`) ## Known Good Patterns - `src/lib/auth.ts` の `withAuth()` HOF: 認証パターンの模範例 - `supabase/functions/stripe-webhook/`: エラーハンドリングの模範例
✅
MEMORY.md の管理ルール（エージェントのシステムプロンプトに必ず記述する） 「タスク開始前に MEMORY.md を参照せよ」「新パターン発見時に追記せよ」「200行超過時は要約せよ」の3つをサブエージェントの system prompt 内に必ず明示してください。

7絶対に避けるべき Anti-Patterns
✗ CLAUDE.md の Anti-Patterns
  • コードスタイル規約を書く（linterに委ねよ）
  • @ファイルで長大なドキュメントを埋め込む
  • 20件を超えるスラッシュコマンドを定義
  • 「〜しないこと」だけの禁止形指示（代替手段を示す）
  • MCP サーバーのトークンを 20k 超で使用
  • 修正履歴・変更ログの記録
✗ サブエージェント定義の Anti-Patterns
  • description が曖昧で呼び出し条件が不明確
  • tools フィールドを省略（全権限を継承してしまう）
  • 10件以上の並列エージェントを同時起動
  • 共有ファイルを触るエージェントを並列実行
  • MEMORY.md の上限管理をしない（肥大化）
  • 全エージェントに opus を使う（コスト爆発）

10 Agent Teams とは — サブエージェントとの根本的な違い
🧪 Experimental · 2026年2月 Opus 4.6 同時リリース

Agent Teams：複数の Claude Code セッションがチームとして協調する

Agent Teams は、複数の Claude Code インスタンスが独立したコンテキストウィンドウを持ちながら、共有タスクリスト・ダイレクトメッセージで連携する実験的機能です。サブエージェントが「親に報告するだけの孤立ワーカー」であるのに対し、Agent Teams のテイムメイトはお互いに直接通信し、タスクを自律的にクレームし、仮説を議論しながら作業します。デフォルト無効（フィーチャーフラグ）。Anthropic は 16 並列インスタンスで 10万行の C コンパイラをビルドする実証実験を行いました（コスト: $20,000 / 2週間）。

🔹 Sub-Agents（既存）
  • 単一セッション内で動作
  • 親エージェントにのみ結果を報告
  • テイムメイト間の直接通信なし
  • 親が仲介しないと情報共有不可
  • タスク完了後は存在しない
  • コスト: 通常セッションと同等
  • 適用: 独立タスクの並列実行、ファイル検索、コードレビューなどの単一ドメイン作業
🟢 Agent Teams（新機能）
  • 各テイムメイトが独立したセッション
  • テイムメイト間で直接メッセージ可能
  • 共有タスクリストからタスクを自律クレーム
  • リードなしに相互調整できる
  • ユーザーが個別テイムメイトに直接話せる
  • コスト: 通常の 3〜4倍（テイムメイト数 × コンテキスト）
  • 適用: 競合仮説デバッグ、クロスレイヤー実装、並列リサーチ

11Agent Teams の有効化と settings.json 設定
⚠️
デフォルト無効・実験的機能 Agent Teams はデフォルトで無効。有効にするには settings.json または環境変数で CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1 を設定します。セッション再開・タスク調整・シャットダウンの挙動に既知の制限があります。必ずリードからクリーンアップを実行してください（テイムメイトから実行するとリソースが不整合になります）。
⚙️ ~/.claude/settings.json — Agent Teams 設定
~/.claude/settings.jsonJSON
{ "env": { "CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS": "1" // 環境変数でも設定可: export CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1 }, // テイムメイトの表示モード // "auto" = tmux 使用中は split panes、それ以外は in-process（デフォルト） // "split" = tmux / iTerm2 の分割ペイン（要 tmux or iTerm2 + it2 CLI） "hooks": { // ── TeammateIdle: テイムメイトがアイドル直前に実行 ────────────── "TeammateIdle": [ { "hooks": [ { "type": "command", "command": "bash -c 'if [ -f ./dist/output.js ]; then echo OK; else echo \"Build artifact missing\" >&2; exit 2; fi'" // exit 2 → フィードバックを送りテイムメイトを継続 // exit 0 → アイドルに移行させる } ] } ], // ── TaskCompleted: タスク完了マーク直前に実行 ────────────────── "TaskCompleted": [ { "hooks": [ { "type": "command", "command": "bash -c 'if ! pnpm test 2>&1; then echo \"Tests failing — fix before closing task\" >&2; exit 2; fi'" // exit 2 → 完了をブロックしフィードバック送信 // exit 0 → タスクを completed に遷移させる } ] } ] } }
💡
ショートカットキー（In-process モード） Shift+↑/↓ でテイムメイトを切り替え、Enter で選択、Esc で中断、Ctrl+T でタスクリストを表示。Shift+Tab で Delegate Mode ON/OFF を切り替えられます（ON にするとリードは実装せず調整専任になります）。
12Agent Teams 向け CLAUDE.md 設計 — 最重要ポイント
✅
テイムメイト全員が CLAUDE.md を自動ロードする 各テイムメイトは起動時に CLAUDE.md・MCP サーバー・Skills を自動読込します。リードの会話履歴は継承されません。CLAUDE.md の品質がそのままチーム全体のパフォーマンスを決定します。 CLAUDE.md が曖昧だと、各テイムメイトが独自にコードベースを探索し直すため、トークンコストが 3〜4倍に膨れ上がります。
📄 Agent Teams 向け CLAUDE.md テンプレート（追記部分）
CLAUDE.md — Agent Teams セクション追加版Markdown
# PROJECT: my-saas-app ## Overview ... (既存セクション省略) ## ─── Agent Teams Configuration ──────────────────────────────────── ## Agent Teams — File Ownership Rules 競合防止のため、テイムメイトは以下の境界を厳守すること: | Teammate Role | 書き込み可能なパス | 読み取り | |---|---|---| | frontend-teammate | app/, components/, styles/ | 全体 | | backend-teammate | supabase/functions/, lib/server/ | 全体 | | db-teammate | supabase/migrations/ | 全体 | | test-teammate | tests/, **/*.test.ts | 全体 | 同一ファイルへの同時書き込みは絶対に行わないこと。 担当外ファイルを変更したい場合はリードに報告し指示を仰ぐこと。 ## Agent Teams — Task "Done" Definition テイムメイトがタスクを "completed" にマークする前に必ず確認すること: - [ ] `pnpm test` がすべて PASS している - [ ] `pnpm lint` にエラーがない - [ ] 担当ファイルのみが変更されている - [ ] リードへの完了レポートに変更ファイル一覧・テスト結果を含める ## Agent Teams — Communication Protocol - リードへの報告形式: "DONE: [タスクID] [変更ファイル一覧] [テスト結果]" - 他テイムメイトへのブロック報告: "BLOCKED: [依存タスクID] を待機中" - 型定義など共有 API が変わった場合は関係テイムメイトに SendMessage ## Agent Teams — Plan-Before-Implement Rule リスクの高い変更（DB スキーマ変更・API 破壊的変更）は実装前に必ずリードに計画を提出し承認を得ること。 承認前はファイルを Read-only 扱いとし、実装を開始しないこと。 ## Agent Teams — Forbidden - チームクリーンアップをテイムメイトから実行することは禁止（必ずリードから） - 担当外ディレクトリへの書き込み禁止 - `supabase db reset` の実行禁止
⚠️
コンテキスト効率化 — スポーンプロンプトに詳細を含める テイムメイトはリードの会話履歴を持ちません。タスク固有の詳細情報はスポーンプロンプト（TaskCreate の description フィールド）に含めてください。CLAUDE.md には普遍的ルールだけを書き、タスク固有の詳細はタスクファイルに持たせます。
13タスクリストファイル設計 — ~/.claude/tasks/{team}/*.json

Agent Teams のタスクは ~/.claude/tasks/{team-name}/ 配下の JSON ファイルとして管理されます。タスクは pending → in_progress → completed の3状態を遷移し、ファイルロックによる競合防止が組み込まれています。description フィールドが実質的にテイムメイトへのプロンプトとなるため、内容の精度がパフォーマンスを左右します。

📋 タスクファイル設計パターン（ベストプラクティス）
~/.claude/tasks/auth-team/1.json（タスク定義例）JSON
{ "id": "1", "subject": "JWT 処理を src/auth/jwt.ts に分離", "description": " ## タスク概要 monolithic な src/auth.ts から JWT 署名・検証ロジックを分離し、 src/auth/jwt.ts に独立モジュールとして実装する。 ## 担当ファイル（書き込み可能） - src/auth/jwt.ts（新規作成） - src/auth/index.ts（export 更新のみ） ## 実装仕様 - JWT_SECRET は process.env から読み取る（ハードコード禁止） - 関数: signToken(payload) / verifyToken(token) - エラーは Result 型で返す（throw 禁止） ## 完了条件 - pnpm test auth が PASS - 型エラーなし (tsc --noEmit) - src/auth.ts の既存テストが引き続き PASS ## 依存関係 - このタスクが完了後に T2（session-teammate）がアンブロック - 完了時に session-teammate に SendMessage で型定義変更を通知 ", "status": "pending", "owner": null, "dependencies": [], "blockedBy": [] }
✅ タスク設計 ベストプラクティス
  • 1タスク = 1テイムメイトが担当できる単位
  • description に「担当ファイル」「完了条件」「依存関係」を記載
  • 1テイムメイトあたり 5〜6タスクが最適（多すぎず少なすぎず）
  • 依存関係を dependencies フィールドで明示（自動アンブロック）
  • 完了時の通知先テイムメイトを記述（型変更・API変更時）
  • タスクサイズは「明確な成果物」が出せる粒度に
✗ タスク設計 Anti-Patterns
  • description が短すぎる（テイムメイトが再探索し無駄にトークン消費）
  • 担当ファイルを明示しない（別テイムメイトと衝突する）
  • 完了条件を書かない（テイムメイトが勝手に判断）
  • 粒度が大きすぎる（1タスクで全機能実装など）
  • 同一ファイルを複数タスクで書き込み対象にする
  • 依存関係を手動で管理（ファイルロックを信頼する）
14Agent Teams Hooks 設計 — 品質ゲートの自動化

Claude Code は Agent Teams 向けに TeammateIdle・TaskCompleted の2つのフックイベントを提供します（v2.1.33+）。フックは命令的（Imperative）であり、シェルコマンドを実行してその結果（終了コード）で Claude の動作を制御します。14種類あるフックイベント全体（SessionStart・PreToolUse 等）と組み合わせることで、チーム全体に一貫した品質ポリシーを強制できます。

🪝 全 Agent Teams 対応フック一覧
フックイベント発火タイミングexit 0exit 2主な用途
TeammateIdle テイムメイトがアイドル直前 アイドルに移行 フィードバックして継続 ビルド成果物チェック・次タスクの自動アサイン
TaskCompleted タスク完了マーク直前 completed に遷移 完了ブロック＋フィードバック テスト必須・Lint チェック・受け入れ条件検証
SubagentStart サブエージェント起動時 起動継続 起動キャンセル 不正なエージェントタイプのブロック
SubagentStop サブエージェント終了時 終了継続 フィードバック送信 成果物の確認・クリーンアップ
🪝 実践的な hooks 設定テンプレート（品質ゲート）

{ "env": { "CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS": "1" }, "hooks": { // ── TeammateIdle: アイドル前にビルドチェック ────────────────── "TeammateIdle": [ { "hooks": [ { "type": "command", "timeout": 10000, // TypeScript コンパイルエラーがないかチェック "command": "bash -c 'if ! npx tsc --noEmit 2>&1; then echo \"TypeScript errors found\" >&2; exit 2; fi'" } ] } ], // ── TaskCompleted: タスク完了前にテスト実行（品質ゲート）──── "TaskCompleted": [ { "hooks": [ { "type": "command", "timeout": 60000, // テストが通らない限りタスクを完了させない "command": "bash -c 'if ! pnpm test --run 2>&1; then echo \"Tests must pass before task completion\" >&2; exit 2; fi'" } ] } ], // ── PreToolUse: 機密ファイルへの書き込みブロック ──────────── "PreToolUse": [ { "matcher": "Write", "hooks": [ { "type": "command", // .env ファイルへの書き込みを全テイムメイトでブロック "command": "bash -c 'if echo \"$CLAUDE_TOOL_INPUT\" | grep -q \".env\"; then echo \".env file modification blocked\" >&2; exit 2; fi'" } ] } ] } }
15Agent Teams のユースケース別プロンプトパターン
🔍
競合仮説デバッグ（最強のユースケース）
複数テイムメイトが独立して異なる仮説を検証し、互いに反論し合うことで最速で根本原因を特定。逐次的な調査では最初に発見した理論に引きずられる「アンカリング」を防げます。
"アプリが1メッセージ後に終了するバグがある。
5人のテイムメイトを起動して異なる仮説を調査させ、
互いに理論を反証し合うよう指示して。
findings.md に共同コンセンサスをまとめて。"
🏗️
クロスレイヤー実装
フロントエンド・バックエンド・DBマイグレーション・テストを担当テイムメイトが並列実装。Backend テイムメイトが API 型定義を完成させたら、Frontend テイムメイトに SendMessage で直接通知。
"認証機能を実装して。
Frontend / Backend / DB-migration / Test の
4テイムメイトを起動。Sonnet を各テイムメイトに使用。
DB テイムメイトから始めて、完了後に他をアンブロックして。"
🔬
並列 QA スウォーム
複数テイムメイトが同時にアプリの異なる側面（パフォーマンス・セキュリティ・アクセシビリティ・E2E）を検証。alexop.dev 実績: 5エージェントで3分でブログ QA 完了。
"localhost:3000 で動くアプリの QA をして。
3テイムメイトを起動: セキュリティ / パフォーマンス / アクセシビリティ。
各自 curl でページを検証して findings をまとめて。"
📐
Plan → Team の2ステップパターン（推奨）
まず Plan モード（安価・読み取り専用）で計画を立て、レビュー後に Agent Teams を起動（高コスト・並列実装）。計画をチェックポイントにしてトークン無駄遣いを防ぐのが鉄則。
Step 1: "auth.ts のリファクタリング計画を立てて。
jwt/session/middleware に分割するプランを見せて。
（実装はしないこと）"

Step 2（計画確認後）:
"この計画を Agent Teams で並列実装して。
各ファイルを別テイムメイトに担当させて。"
🚨
Agent Teams Anti-Patterns（コスト・品質両面）
• いきなり大タスクで起動：Plan モードで計画を確認してから Team を起動すること
• 同一ファイルを複数テイムメイトが編集：必ずファイルオーナーシップを CLAUDE.md に明記
• 依存関係のあるタスクを並列実行：dependencies フィールドを使って自動順序管理
• 全テイムメイトに Opus を使う：リードに Opus、テイムメイトに Sonnet でコスト最適化
• テイムメイトからチームクリーンアップを実行：必ずリードから実行（リソース不整合になる）
• CLAUDE.md が曖昧：全テイムメイトが独自に探索し直し、トークンが3倍以上消費される
