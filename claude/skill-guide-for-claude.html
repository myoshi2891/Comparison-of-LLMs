01

SKILL.md とは何か？

SKILL.md は、Claude Code に「専門的なスキル（能力）」を追加するための設定ファイルです。 一言でいうと — 「Claudeに特定のタスクのプロの手順書を渡す仕組み」です。

💡

Claude Code はセッションをまたいで記憶を保持しません。毎回「このプロジェクトではPDFはこう作る」「コードレビューはこの観点でやる」と説明し直すのは非効率です。
SKILL.md を使うと一度書けば何度でも再利用できます。

なぜ SKILL.md が必要なのか？

♻️
再利用性
一度書けば何度でも繰り返し使える。毎回同じ説明をする必要がなくなる。
🤖
自動読み込み
Claude が状況を判断し、適切なスキルを自動的に読み込んで実行する。
📦
バンドル対応
スクリプトやテンプレート、参考資料をスキルにまとめて同梱できる。
🌐
オープンスタンダード
チームで共有可能。複数のAIツール間でも互換性があるオープン仕様。

従来の方法 vs SKILL.md

❌ 従来の方法（非効率）

毎回 Claude に手順を説明し直す。セッションが変わるたびに同じ指示を繰り返す。コンテキストが毎回リセットされる。

✅ SKILL.md を使う方法

一度書いたら自動で読み込まれる。チームで共有できる。スクリプトも同梱できる。

03

トークン経済と3層アーキテクチャ

なぜ単一の巨大なプロンプトではなく、ディレクトリとファイル群による階層的な構造なのか？ その答えが「プログレッシブ・ディスクロージャー」という設計思想です。

🧠

情報を無秩序にコンテキストへ詰め込むと「アテンション・バジェット（注意力の予算）」が枯渇し、 モデルの推論精度が著しく低下します。SKILL.md はこれを防ぐため、必要な情報を必要なタイミングでのみ読み込む3層構造を採用しています。

3層読み込みレベル

レベル 構成要素 トークン消費 読み込みタイミング
Level 1 メタデータ YAMLフロントマター（nameとdescription） ~100 tokens/スキル Claude Code 起動時・常時
Level 2 インストラクション SKILL.md のマークダウン本文（推奨500行未満） ~5000 tokens スキルがトリガーされた瞬間のみ
Level 3 参照ファイル 外部ドキュメント・実行スクリプト 実質無制限 エージェントが必要と判断した時のみ
sequenceDiagram participant User as 👤 ユーザー participant C as ⚡ Claude Code participant L1 as Level 1 (YAML) participant L2 as Level 2 (本文) participant L3 as Level 3 (スクリプト) Note over C,L1: セッション開始フェーズ C->>L1: 全スキルのメタデータをスキャン L1-->>C: [name] と [description] のみ登録（~100 tokens/スキル） User->>C: "READMEを作成してください" Note over C: 推論: ユーザー意図 ↔ description を照合 C->>L2: readme-writer スキルの本文を読み込み L2-->>C: 指示内容をコンテキストに展開 C->>L3: references/style.md を読み込み L3-->>C: スタイルガイドラインを返す C->>L3: scripts/validate.sh を実行 L3-->>C: 実行結果のみを返す（コードは不要） C-->>User: ガイドラインに準拠したREADMEを出力
⚡

この設計の最大の利点はスケーラビリティです。 100個のスキルを追加しても、トリガーされないスキルのコストはほぼゼロ。 メインの会話コンテキストを圧迫しません。

04

SKILL.md の基本構造

SKILL.md は 2つのパート で構成されます。

block-beta columns 1 A["--- YAMLフロントマター ---\nname: スキル名\ndescription: いつ使うかの説明\nallowed-tools: [Read, Write]\n---"] B["# マークダウン本文（手順書）\n\n## Step 1: 〇〇をする\n## Step 2: 〇〇をする\n## 出力フォーマット\n..."] style A fill:#1c2a1c,stroke:#3fb950 style B fill:#1c2530,stroke:#58a6ff

実際のファイル例：コードレビュースキル

~/.claude/skills/code-review/SKILL.md 
--- ← YAMLフロントマター（Level 1 · 常時読み込み）
name: code-review
description: >
  コードのセキュリティ・バグ・パフォーマンス・スタイルを
  レビューする。「レビューして」「確認して」「チェックして」
  と言われたときに使用する。
allowed-tools: [Read, Grep, Glob]
--- ← ここまでがフロントマター

# コードレビュースキル
↑ マークダウン本文（Level 2 · トリガー時のみ読み込み）

## レビュー観点

1. **セキュリティ**: SQLインジェクション、XSS をチェック
2. **バグ**: NULL参照、境界値、例外処理を確認
3. **パフォーマンス**: N+1クエリ、不要なループを検出
4. **スタイル**: プロジェクトの命名規則に従っているか確認

## 出力フォーマット

- 🔴 重大: すぐに修正が必要
- 🟡 警告: できれば修正を推奨
- 🟢 提案: コード品質向上のヒント
⚠️

--- で囲まれたYAMLフロントマターは必須です。これがないと Claude はスキルを認識できません。 また、name と description は最低限必要なフィールドです。

06

スキルが呼び出される仕組み

スキルが実行されるまでの流れを見てみましょう。

sequenceDiagram participant U as 👤 ユーザー participant CC as ⚡ Claude Code participant SK as 📚 Skills一覧 participant SM as 📄 SKILL.md U->>CC: 「このコードをレビューして」 CC->>SK: 利用可能なスキルを確認 SK-->>CC: [code-review, deploy, api-docs, ...] Note over CC: description とユーザーの意図を照合 CC->>SM: code-review/SKILL.md を読み込む SM-->>CC: レビュー手順・観点・出力形式を返す CC->>U: 手順に従ったコードレビュー結果

2つの呼び出し方法

🤖
方法①：Claude が自動判断（推奨）
description を読んで、ユーザーの意図と一致すると判断したら自動で実行。 ユーザーは何も意識する必要がない。 
ユーザー: 「このPDFを解析して要約してくれ」
Claude:  (自動でpdfスキルを検出・実行)
⌨️
方法②：スラッシュコマンドで手動
ユーザーが直接スキル名をコマンドとして入力して実行する。確実に特定のスキルを動かしたい時に使う。 
ユーザー: /code-review src/auth/login.ts
Claude:  (code-reviewスキルを直接実行)

invocation フィールドで制御する

YAML frontmatter 
# Claude が自動判断（デフォルト）
invocation: automatic

# /deploy と入力したときのみ実行（破壊的操作に推奨）
invocation: explicit
# または
disable-model-invocation: true

08

ステップバイステップ：最初のスキルを作る

「コードを分かりやすく説明する」スキルを実際に作ってみましょう。

1
ディレクトリを作成する
個人スキルとして作成（全プロジェクトで使える）
Terminal 
$ mkdir -p ~/.claude/skills/explain-code
2
SKILL.md ファイルを作成する
スキルのメインファイルを作成します
Terminal 
$ touch ~/.claude/skills/explain-code/SKILL.md
3
YAMLフロントマターを書く
name と description を必ず記入します。 description はトリガーワードを具体的に含めることが最重要です。
SKILL.md — フロントマター部分 
---
name: explain-code
description: >
  コードの動作をビジュアル図解と例えを使って説明する。
  「どうやって動いてる？」「わかりやすく説明して」
  「教えて」と言われたときに使用する。
---
4
本文（指示書）を書く
手順・出力形式・ルールを Markdown で記述します
SKILL.md — 本文部分 
# コード説明スキル

## 説明プロセス

1. **アナロジー（例え話）で始める**
   日常生活の何かに例えてから説明する
   例：「このコードは郵便局のようなもので...」

2. **図を描く（ASCII アート）**
   コードの流れ・構造・関係性を視覚化する

3. **ステップバイステップで解説する**
   コードを上から順に、何が起きているか説明

4. **「落とし穴」を1つ挙げる**
   よくある誤解や注意すべき点を1つ教える

## 重要なルール

- 専門用語を使う場合は必ず初出時に定義する
- 説明は会話的なトーンで行う

動作確認する
Claude Code を起動して自動・手動の両方でテストします
Terminal 
$ claude   # Claude Code を起動

# 自動呼び出しをテスト
> 「このコードはどうやって動いているの？」

# または直接呼び出し
> /explain-code src/utils/parser.ts
6
完成！ディレクトリ構造を確認
最終構造 
~/.claude/skills/explain-code/
└── SKILL.md     ✅ 完成！

YAMLフロントマター 全フィールド解説

graph TD FM["--- YAMLフロントマター ---"] FM --> N["📛 name\nスキル名・/slash-commandの名前になる\n最大64文字、小文字・数字・ハイフンのみ\n例: name: code-review"] FM --> D["📝 description\n★最重要★ いつ使うかをClaudeが判断する\n最大1024文字。具体的なトリガーワードを含める\n三人称視点で記述"] FM --> AT["🔒 allowed-tools\nスキルで使えるツールを制限\n例: [Read, Write, Bash, Grep, Glob]"] FM --> INV["⚡ invocation\nautomatic: Claude自動判断（デフォルト）\nexplicit: /コマンドのみ実行"] FM --> DIS["🚫 disable-model-invocation\ntrue: 自動発動を完全に禁止\n破壊的操作スキルに推奨"] FM --> CTX["🔀 context\nfork: 独立したコンテキストで実行\ninherit: メイン会話を引き継ぐ（デフォルト）"] FM --> AG["🤖 agent\ngeneral-purpose（デフォルト）\nExplore / Plan / カスタム"] FM --> DEP["📦 dependencies\n必要なソフトウェアを明示\n例: python>=3.8"] style FM fill:#1c2a1c,stroke:#3fb950

完全な設定例

SKILL.md — YAMLフロントマター完全版 
---
name: security-audit          # /security-audit コマンドになる
description: >
  コードのセキュリティ監査を行う。
  「セキュリティをチェックして」「脆弱性を探して」
  「セキュリティ監査して」と言われたときに使用する。
  OWASP Top 10 の観点でレビューする。

allowed-tools:              # セキュリティ上、最小権限を推奨
  - Read
  - Grep
  - Glob

invocation: automatic       # Claude が自動判断
context: fork               # メインのコンテキストを汚染しない
agent: general-purpose      # デフォルトのエージェント
---
⚠️

name フィールドの制約：
最大64文字。小文字・数字・ハイフンのみ使用可能。 anthropic や claude などのシステム予約語は使用不可。 命名は動名詞スタイルが推奨（例：processing-pdfs, reviewing-code）

10

引数と動的コンテキストの注入

SKILL.md は静的な手順書だけではありません。 ユーザーからの動的な入力値や、実行時のシステム状態をリアルタイムで取り込む強力な機能があります。

引数の受け取り方

SKILL.md — 引数処理 
---
name: explain-code
description: コードを解説する
---

# コード解説スキル

以下のファイルを解説してください: $ARGUMENTS
# ↑ /explain-code src/main.py と実行すると "src/main.py" に置換される

# 複数の引数も使用可能
対象ファイル: $1   # 最初の引数
比較対象:     $2   # 2番目の引数

動的コンテキストの事前注入（強力な機能）

🚀

バッククォート構文（! `command`）を使うと、スキルが実行される「前」に シェルコマンドを実行し、その結果をプロンプトに自動で埋め込めます。 エージェントが自分でコマンドを実行して調べる手間が完全に省けます。

SKILL.md — 動的コンテキスト注入 
## 実行コンテキストの動的取得

エージェントは以下の確定したデータに基づいて分析を開始すること。

現在のOS情報:
! `uname -a`
# ↑ スキル実行前にコマンドが走り、結果が自動挿入される

最新のコミット:
! `git log -1 --oneline`

解析対象ファイル: $1
❌ 従来の方法

Claude に「git log を確認して」と指示 → Claude が自分でコマンド実行 → 結果を見て分析開始。 余分な推論ステップとレイテンシが発生する。

✅ 動的注入を使う

スキル起動時に自動でコマンド実行 → 結果がプロンプトに埋め込まれた状態でClaudeが受け取る。 即座に分析を開始できる。

リソースファイルのバンドル

スキルには指示書だけでなく、スクリプトや参考資料もまとめられます。

実践例：PDF処理スキルの構造

ディレクトリ構造 
~/.claude/skills/pdf-processor/
├── SKILL.md                  ← 指示書（Level 2）
├── scripts/
│   ├── extract_text.py       ← テキスト抽出（Level 3）
│   └── summarize.py          ← 要約スクリプト（Level 3）
├── references/
│   └── output_format.md      ← 出力形式の仕様書（Level 3）
└── assets/
    └── report_template.html  ← レポートテンプレート（Level 3）
SKILL.md — スクリプトを参照する書き方 
---
name: pdf-processor
description: PDFファイルを処理して要約・抽出を行う。
allowed-tools: [Bash, Read, Write]
---

# PDF処理スキル

## 処理手順

1. `scripts/extract_text.py` を実行してテキストを抽出する
   # → Claude は Bash で自動実行し、結果のみを受け取る

2. `references/output_format.md` の形式に従って結果を整理する
   # → 必要なタイミングでのみ読み込む（コンテキスト節約）

3. `assets/report_template.html` を使ってレポートを生成する

## 重要な注意事項

- 大きなPDFは分割して処理する（1回あたり最大50ページ）
- 画像が含まれる場合は OCR の精度について言及する
⚠️

参照ファイルは必ず 1階層（One level deep） にとどめること。 深くネストされた参照（AがBを参照し、BがCを参照）は情報が不完全になるリスクがあります。 100行を超える参照ファイルには冒頭に目次（Table of Contents）を設置してください。

12

ベストプラクティス

✅ description の書き方が最重要

Claude がスキルを自動選択するかどうかは、description の質で決まります。 エージェントは総じてスキルを起動したがらない傾向があるため、やや強引に記述することが推奨されています。

❌ 悪い例（曖昧すぎる）
description: コードに関する作業をする
✅ 良い例（トリガーワードを含む）
description: >
  コードのセキュリティ・パフォーマンス・バグを包括的にレビューする。
  「レビューして」「確認して」「チェックして」「監査して」
  「問題点を探して」という言葉をトリガーに使用する。
  Pull Request のレビューにも使用する。

✅ 「自由度（Degrees of Freedom）」の制御

タスクタイプ 自由度 記述スタイル 例
複数アプローチが許容されるタスク High Freedom 高レベルなガイドラインのみ テキスト要約、コードレビュー
好ましいフォーマットがあるタスク Medium Freedom テンプレートやパラメータを提供 定型レポート生成

確認項目 説明

よくあるミスと対処法

ミス① スキルが自動で呼ばれない

❌ よくある原因
  • description が曖昧・抽象的
  • SKILL.md の配置が間違っている
  • スキルが設定で無効化されている
  • name に使えない文字が含まれる
✅ 解決策
  • トリガーワードを具体的に追記
  • skills/スキル名/SKILL.md の構造を確認
  • Settings → Skills で有効化を確認
  • /スキル名 で手動呼び出しテスト

ミス② 同名スキルの競合

競合の例 
~/.claude/skills/deploy/SKILL.md     ← 個人スキル
.claude/skills/deploy/SKILL.md       ← プロジェクトスキル（こちらが優先）
# 優先順位: enterprise > personal > project
# 意図しない上書きの場合はスキル名を変更すること

ミス③ CLAUDE.md にすべてを書きすぎる

❌ よくあるミス

CLAUDE.md に詳細な手順書や長大なドキュメントを書いてしまっている。 毎回コンテキストを大量消費し、会話の精度が低下する。

✅ 正しい分担

CLAUDE.md には短い参照のみ書く。 詳しい手順は skills/my-procedure/SKILL.md に移動する。 トリガーされない限りコンテキストを消費しない。

ミス④ description に XMLタグを使う

YAML 
# ❌ XMLタグは description に使えない
description: <task>コードをレビューする</task>  ← エラー

# ✅ プレーンテキストで記述
description: コードのセキュリティ・バグを包括的にレビューする。

エンタープライズ展開とプロビジョニング

Team/Enterprise プランでは、管理者が検証済みのスキルを 組織全体に一括展開（プロビジョニング）できます。

1
スキルのパッケージ化
スキルディレクトリを ZIP 圧縮する。ルートがディレクトリ名になるよう注意。 
$ zip -r team-reviewer.zip team-reviewer/
# 解凍結果: team-reviewer/SKILL.md ← この構造が必須
2
アップロードと展開
管理者が Organization settings → Skills から ZIP ファイルをアップロードする。即座に全ユーザーに展開される。
3
デフォルトステータスの設定
全員に適用するスキルは「デフォルト有効」、一部のユーザーのみが使う専門的・破壊的スキルは「デフォルト無効」で展開する。
