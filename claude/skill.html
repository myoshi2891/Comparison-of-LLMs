01.AI仕様駆動開発の全体像とファイル構成

Section 02

02.CLAUDE.md — プロジェクト永続メモリ

Claude Codeが起動時に自動読み込みする最重要ファイル。プロジェクトの「永続的な脳」として機能し、セッションをまたいでコンテキストを維持します。ディレクトリ階層に応じた読み込み優先度があります。

📋
CLAUDE.md
プロジェクト永続メモリ / 最優先読み込みファイル

読み込み優先度（下 → 上が優先）

パススコープ用途
~/.claude/CLAUDE.md グローバル 個人の全プロジェクト横断設定
/project-root/CLAUDE.md プロジェクト チーム共有のプロジェクト規約（最重要）
CLAUDE.local.md 個人/プロジェクト 個人設定（.gitignore自動追加）
src/CLAUDE.md サブディレクトリ モジュール別補足（オンデマンド読み込み）

ベストプラクティス構成テンプレート

CLAUDE.md 
# Project: [プロジェクト名]

## 🎯 Project Overview
# 1〜2文でプロダクトの目的を記述。Claudeが全判断の基準にする。
ECサイトのバックエンドAPI。Go製マイクロサービス構成。

## 🛠 Build & Test Commands
# 最重要: Claudeが即座に実行できるコマンドを列挙
- Build: `go build ./...`
- Test:  `go test ./... -race`
- Lint:  `golangci-lint run`
- Dev:   `docker-compose up -d`

## 📐 Code Style
# Claudeが守るべきコーディング規約を明記
- Go modules使用。CommonJSは使わない
- エラーハンドリング: errors.Wrap()を必ず使用
- テスト: table-driven testsを標準とする
- コメント: 英語で書く

## 🏗 Architecture
# 主要ファイル・パターンをポインタで示す
- 状態管理: internal/store/ 参照
- APIルーティング: cmd/server/main.go
- DB: PostgreSQL (マイグレーション: migrations/)

## 🔗 Imports
# @import構文で詳細ドキュメントを参照させる
@docs/design.md
@docs/api-conventions.md

## ⚠️ Constraints
# 絶対に破ってはいけないルール
- DBのmigrations/フォルダへの直接書き込み禁止
- secrets/配下ファイルは読み取りのみ
⚠️
アンチパターン：過剰記述
CLAUDE.mdが長すぎると、重要なルールが「ノイズ」に埋もれてClaudeが無視します。公式ドキュメントは「Claudeがすでに正しくできることは書かない。Hooksで強制できるなら削除する」と推奨しています。目安は500行以内。
💡
@import構文でファイルを参照可能。@docs/git-instructions.md のように書くと、その内容がコンテキストに注入されます。相対パスは「そのファイルからの相対」になります。
Section 03

03.spec.md — 仕様書の起点

SDDの出発点となるファイル。アプリケーションの「何を作るか」を人間とClaudeが合意するための設計図。ここが明確であるほど、後続のrequirements.md・design.md・tasks.mdの品質が高まります。

📝
spec.md
仕様書の起点 / プロダクト概要

2つのアプローチ

🤖 AI生成 → 人間レビュー

「TODOアプリ（優先度・日付・タグ機能付き）のspecを作成して」と依頼し、Claudeが草案を作成。人間が仮定の修正・エッジケースの追加を行う。速度重視。

✍️ 手動作成 → AI補完

自分でspec.mdを書き込み、Claudeに「このspecの抜け漏れを指摘して」と依頼。品質重視。複雑なドメイン知識を持つ場合に適する。

docs/spec.md — テンプレート 
# [機能名] 仕様書

## 概要
# 何を・なぜ・誰のために作るかを2〜4文で記述
ユーザーが商品を事前予約できるプリオーダーシステム。
在庫切れ状態でも購入確定できることでCVRを向上させる。

## 主要機能
# ユーザー視点で「何ができるか」を箇条書き
- 商品詳細ページ: 画像・名称・説明・数量選択
- チェックアウト: ログイン→配送選択→決済
- マイページ: 注文履歴・ステータス確認

## 非機能要件（制約）
# パフォーマンス・セキュリティ・技術スタック制約
- 同時接続1,000件対応（Redis在庫予約使用）
- 決済はStripe APIを使用
- レスポンスタイム p99 < 500ms

## 依存ライブラリ・外部サービス
# Claudeが「なぜこれを使うか」を知るためのメモ付き
- Stripe: 決済処理（SCA対応）
- Redis: 在庫ロック（原子的操作）
- PostgreSQL: 永続ストレージ

## 「完了」の定義
# Acceptance Criteria — テスト可能な形で記述
- [ ] プリオーダー完了後、確認メールが届く
- [ ] 在庫0状態でもカートに追加できる
- [ ] 管理画面で注文一覧が確認できる
ℹ️
重要: spec.mdは実装中も継続的に更新します。「コードが変わったらspecも変える」という規律がプロジェクトの健全性を保ちます。

04.requirements.md — 要求定義

spec.mdをClaudeが解析し、機能要件・非機能要件・システムスコープに分解した詳細ドキュメント。ステークホルダーと実装チームの「契約書」として機能します。

📋
requirements.md
機能要件 / 非機能要件 / 受け入れ基準

生成プロンプト例

Claude Codeへの指示
claude -p "docs/spec.md を解析し、機能要件と非機能要件に分解した
requirements.md を生成してください。
各要件にはID(REQ-001形式)と受け入れ基準を付与してください。"
docs/requirements.md — テンプレート 
# 要求定義書

## 機能要件（Functional Requirements）

### FR-001: プリオーダー登録
**概要**: 在庫切れ商品をカートに追加できる
**優先度**: P0 (Must Have)
**受け入れ基準**:
- 在庫0の商品でも「予約する」ボタンが表示される
- カート追加後、確認画面で"予約注文"と表示される
- ユーザー未ログイン時はログイン画面にリダイレクト

### FR-002: 決済処理
**概要**: Stripe経由でクレジットカード決済を実行
...

## 非機能要件（Non-Functional Requirements）

### NFR-001: パフォーマンス
- 商品ページ読み込み: p99 < 500ms
- 在庫ロック処理: < 100ms（Redis使用）

### NFR-002: セキュリティ
- 決済情報はサーバー側で保持しない（Stripe Token使用）
- CSRF対策必須

## スコープ外（Out of Scope）
- モバイルアプリ対応（今フェーズはWeb only）
- 複数通貨対応
💡
要件に優先度（P0/P1/P2）とID（FR-001形式）を付けることで、tasks.mdでのトレーサビリティを確保できます。

05.design.md / architecture.md — 技術設計書

requirements.mdをもとにClaudeが生成するシステムアーキテクチャ・API設計・データモデルの詳細設計書。Mermaid記法のシーケンス図やER図を含めることで視覚的な理解を促します。

🏛
design.md / architecture.md
システム設計 / API / データモデル

設計書生成プロンプト例

Claude Codeへの指示
claude -p "requirements.md に基づいてdesign.mdを作成してください。
マイクロサービスアーキテクチャを提案し、
各サービスのAPIエンドポイント、データモデル（DDL）、
チェックアウトフローのシーケンス図（Mermaid形式）を含めてください。"
docs/design.md — テンプレート 
# 技術設計書

## アーキテクチャ概要
マイクロサービス構成。API Gateway → User Service / Order Service / Inventory Service

## サービス一覧

### Order Service
**責務**: 注文の作成・ステータス管理
**API Endpoints**:
- POST /orders       — 注文作成
- GET  /orders/{id}  — 注文詳細
- PUT  /orders/{id}/confirm — 在庫確保後の確定

## データモデル（DDL）
```sql
CREATE TABLE orders (
  id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id     UUID NOT NULL REFERENCES users(id),
  status      VARCHAR(20) CHECK (status IN ('pending','confirmed','cancelled')),
  created_at  TIMESTAMPTZ DEFAULT NOW()
);
```

## シーケンス図（チェックアウト）
```mermaid
sequenceDiagram
  Client->>API Gateway: POST /checkout
  API Gateway->>Inventory Service: ロック在庫
  Inventory Service-->>API Gateway: OK
  API Gateway->>Payment Service: Stripe課金
  Payment Service-->>API Gateway: 決済完了
  API Gateway->>Order Service: 注文確定
  Order Service-->>Client: 201 Created
```

## 技術スタック
| レイヤー | 技術 | 選定理由 |
|---------|------|---------|
| API     | Go + chi | 軽量・高速 |
| DB      | PostgreSQL | ACID準拠 |
| Cache   | Redis | 原子的在庫ロック |
ℹ️
architecture.mdは実装を進めながら人間とClaudeが共同編集します。「3〜4タスク完了後にClaudeにコードレビューとarchitecture.md更新を依頼する」サイクルが効果的です（[9]）。

06.tasks.md — タスク分解と追跡

design.mdからClaudeが生成する実装の「道路地図」。各タスクは独立してテスト可能な粒度に分解され、チェックボックスで進捗を追跡します。セッション間でClaudeの「記憶喪失」を防ぐ最重要ファイルです。

✅
tasks.md
実装タスクチェックリスト / 進捗追跡
docs/tasks.md 
# 実装タスクリスト
# 各タスクは単独でテスト可能な粒度にする

## Phase 1: 基盤構築
- [x] Task 1: User Serviceの/users/registerエンドポイント作成 + 単体テスト
- [x] Task 2: JWTミドルウェア実装
- [ ] Task 3: PostgreSQLマイグレーション（orders, inventory テーブル）

## Phase 2: コアロジック
- [ ] Task 4: Inventory Serviceの在庫ロック（Redis SETNX）
  **依存**: Task 3完了が前提
  **Ref**: FR-001, NFR-001
- [ ] Task 5: Order Service — 注文作成エンドポイント
- [ ] Task 6: Stripe決済フロー統合

## Phase 3: 検証
- [ ] Task 7: E2Eテスト（チェックアウトフロー全体）
- [ ] Task 8: 負荷テスト（k6, 1,000同時接続）

## ブロッカー
# Claudeが詰まった問題を記録。後続セッションへの引き継ぎ情報
- Task 4: Redis Luaスクリプトのタイムアウト処理要調査

Claudeへのタスク実行指示パターン

Claude Codeへの指示
design.md のアーキテクチャを把握してください。
tasks.md のTask 3を実行: 「PostgreSQLマイグレーション作成」
Goで実装し、単体テストも作成してください。テストはすべてパスさせること。
完了後にgit commitしてください。
💡
サブエージェント活用: implement @docs/tasks.md - use the task tool, each task should only be done by a subagent, do a commit before continuing と指示することで、各タスクを独立したコンテキストウィンドウで処理し、コンテキスト汚染を防げます（[1]）。
⚠️
コンテキスト管理: tasks.mdとarchitecture.mdに重要事項が記載されていれば、コンテキストクリア（/clear）しても大丈夫。セッションをまたぐ情報はこれらのファイルに書き出しておきます（[9]）。

07.MEMORY.md — 自動メモリシステム

Claude Codeが自動的に書き込むメモリファイル。人間が書くCLAUDE.mdとは異なり、Claudeが作業中に発見したパターン・学習・洞察を記録します（2025年後半から段階的ロールアウト）。

🧠
MEMORY.md
Claude自動生成メモリ / インデックスファイル

📍 保存場所

~/.claude/projects/<project>/memory/MEMORY.md
gitリポジトリルートで一意のディレクトリが決定される。

⚡ 読み込み仕様

MEMORY.mdの先頭200行のみが毎セッション自動注入。200行超の詳細はトピックファイルに分割。

📁 トピックファイル

debugging.md, api-conventions.md 等のファイルはオンデマンド読み込み。indexとして参照される。

✏️ 手動編集も可

/memory コマンドでファイルセレクターが開き、人間が直接編集可能。# ショートカットでその場追記。

~/.claude/projects/myapp/memory/MEMORY.md — 構造例 
# Project Memory Index
# このファイルは先頭200行のみ毎回読み込まれる

## Current State
- バージョン: v1.2.0（2025-12-01リリース）
- 直近の変更: RedisキャッシュをValkey 8に移行済み
- 進行中: GraphQL APIのスキーマ更新（design.md参照）

## Tech Stack（Quick Reference）
- Runtime: Go 1.23
- DB: PostgreSQL 16（接続: DATABASE_URL環境変数）
- Cache: Valkey 8（Redis互換）

## Topic Files
# 詳細はオンデマンドで読み込まれるトピックファイルを参照

## User Preferences

- テスト: TDD必須（まずテストを書く）

09.SKILL.md — 再利用ナレッジ

.claude/skills/配下に置くドメイン知識・繰り返しワークフローのカプセル化ファイル。チームの「ベストプラクティスのライブラリ」として機能します。

🎓
SKILL.md
再利用ナレッジ / チームノウハウ
.claude/skills/database-migration/SKILL.md 
# Database Migration Skill

## いつ使うか
DBスキーマ変更を伴う実装タスクで使用。

## 手順
1. `migrations/`配下に `YYYYMMDD_description.up.sql` を作成
2. ロールバック用 `.down.sql` も必ず作成
3. `make migrate-test` でステージング適用確認
4. PRにマイグレーション内容を記載

## 禁止事項
- 既存マイグレーションファイルの編集
- NULL制約の後付け（データがある場合）

## コード例
```sql
-- 20241201_add_preorder_flag.up.sql
ALTER TABLE products ADD COLUMN is_preorder BOOLEAN DEFAULT FALSE;
CREATE INDEX idx_products_preorder ON products(is_preorder) WHERE is_preorder = TRUE;
```
💡
SKILL.mdは関連性があるときだけClaudeが自動適用、または /skill-name で明示的に呼び出せます。プロジェクト固有のものも、チーム共有のグローバルなものも作成可能です。
Section 10

10.横断ベストプラクティス 10則

01

CLAUDE.mdは500行以内に

重要ルールが埋もれないよう厳選。Hooksで強制できるルールはCLAUDE.mdから削除する。

02

Spec First, Code Second

コードを一行も書く前にspec.md→requirements.md→design.md→tasks.mdを準備。

03

specを生きたドキュメントに

コードが変わったらspec/design.mdも更新。「コードとspecの乖離」がプロジェクト腐敗の原因。

04

タスクは独立・テスト可能に

tasks.mdの各タスクはそれ単体でテストできる粒度に。依存関係は明示する。

05

コンテキストを意識して管理

重要情報はファイルに書き出しておく。/clearしてもtasks.md+design.mdで再現可能にする。

06

@import構文を活用

CLAUDE.mdで@docs/design.mdなど参照。大きな情報はCLAUDE.mdに直書きせず参照する。

07

サブエージェントでコンテキスト分離

各タスクを独立したコンテキストで処理。メインセッションのノイズを防ぐ。

08

コードレビューを3〜4タスクごとに

「design.mdに基づいてTask 1〜3のコードをレビューし、必要ならdesign.mdを更新して」

09

CLAUDE.local.mdで個人設定を分離

チームに見せたくない個人設定はCLAUDE.local.mdへ。.gitignoreに自動追加される。

10

Hooksで強制・CLAUDE.mdは指針

「必ずやること」はHooksで確実化。CLAUDE.mdは「指針」、Hooksは「強制」。
