1 グローバル設定を作る 所要時間: 5分

まず ~/.codex/AGENTS.md を作成し、自分のすべてのプロジェクトに共通するルールを書きます。これが「あなたの作業スタイルの設定ファイル」です。

~/.codex/AGENTS.md（グローバル設定）Markdown
# Global Working Agreements ## Language コメントとコミットメッセージは日本語で書くこと。 ## Code Style - インデント: 2スペース（TypeScript / JavaScript） - 関数名はキャメルケース、定数はスネークケース大文字 ## General Rules - テストなしでコードをコミットしない - TODO コメントは残さない（実装するか Issue を立てる）
💡
ポイント：ここにはプロジェクトに関係ない「個人の作業習慣」のみ書く。プロジェクト固有の内容は次ステップへ。
2 プロジェクトルートに AGENTS.md を作る 最重要ファイル

プロジェクトの /init コマンドで自動生成するか、手動で作成します。テストコマンドの記述が最重要です。Codex はこれを見てテストを自動実行します。

./AGENTS.md（プロジェクトルート）Markdown
# PROJECT: my-app ## Overview Next.js 15 + Supabase の SaaS アプリ。 ## ⚠ Build & Test（必ず実行すること） - テスト: `pnpm test` ← 変更後は必ずこれを実行 - ビルド: `pnpm build` - Lint: `pnpm lint` ← PR 前に必ず実行 ## Architecture - frontend/ → Next.js ページ・コンポーネント - backend/ → API ルート・サーバーサイド処理 - db/ → Supabase マイグレーション ## Forbidden - `.env.production` の読み書き禁止 - `db reset` は絶対に実行しない
⚠️
32 KiB 上限に注意：このファイルが肥大化したら、詳細手順を SKILL.md（Step 5）に分離してください。テストコマンドと禁止事項を優先的に上に書くこと。
3 サービス別サブディレクトリに AGENTS.md を追加する サブエージェント対応

マイクロサービス・モノレポ構成の場合、サービスごとにルールが異なります。各サブディレクトリの AGENTS.md にはそのサービス固有のルールだけを書き、ルートと重複させないのが原則です。

./services/payments/AGENTS.md（サービス固有）Markdown
# Payments Service # ルートの AGENTS.md に加えて、このファイルのルールが適用される ## Service-Specific Commands - テスト: `make test-payments` ← このサービスは make を使う - 型生成: `pnpm stripe:types` ## Domain Constraints - このディレクトリ内のファイルのみ変更すること - Stripe API は v2 のみ使用（v1 API は廃止済み） - PCI-DSS: カード番号をログに出力しない ## Sub-agent Handoff このサービスのタスク完了後は AGENT_TASKS.md を更新し、 PM エージェントに transfer_to_project_manager で引き渡すこと。
4 config.toml でサブエージェントロールを定義する マルチエージェント

サブエージェントを使う場合は .codex/config.toml で各ロールのモデル・サンドボックス・承認ポリシーを定義します。ロール定義なしにサブエージェントをスポーンしてはいけません（Anti-Pattern）。

.codex/config.toml（最小構成）TOML
# 最小限のマルチエージェント設定 [features] multi_agents = true [agents] max_threads = 4 # 同時実行上限（デフォルト: 6） max_depth = 1 # 孫エージェントは禁止（推奨） # 軽量探索ロール（コード読み専用） [agents.roles.explorer] description = "コードベース探索・ファイル検索専門。実装しない。" config_file = ".codex/roles/explorer.toml" # レビューロール（書き込み禁止） [agents.roles.reviewer] description = "PR前のコードレビュー専門。コードは変更しない。" config_file = ".codex/roles/reviewer.toml"
🚨
サブエージェントは承認を継承しない：各ロールの config.toml で必ず approval_policy = "never" を指定すること。未設定のまま実行すると承認待ちで処理がブロックされます。
5 専門タスクを SKILL.md にスキル化する コンテキスト節約

AGENTS.md が肥大化してきたら、繰り返し実施する専門タスク（DBマイグレーション・コードレビュー・テスト生成など）を .agents/skills/ 配下の SKILL.md に分離します。スキルは該当タスクのときだけロードされるため、コンテキストウィンドウを節約できます。

.agents/skills/db-migration/SKILL.md（例）Markdown + YAML frontmatter
--- name: db-migration description: | DB スキーマ変更・マイグレーションファイル作成のスキル。 以下の場合にトリガーする: - テーブルの追加・変更・削除を依頼された - マイグレーションファイルの作成を求められた 以下の場合はトリガーしない: - シードデータの挿入 - アプリケーションレベルのデータ変換 --- ## Instructions 1. `migrations/` に `YYYYMMDD_description.up.sql` を作成 2. ロールバック用 `.down.sql` を必ず同時作成 3. チェック: `python scripts/check_migration.py --env staging` 4. 本番適用は人間レビュー後のみ
6 PM 駆動ワークフローを組む Agents SDK

複数エージェントを使う場合、Project Manager エージェントに REQUIREMENTS.md と AGENT_TASKS.md を生成させるのが Codex 公式推奨パターンです。各サブエージェントはこの2ファイルを「唯一の真実の情報源」として実装します。

✅ PM 駆動パターン（推奨）
  • PM が REQUIREMENTS.md を生成
  • PM が AGENT_TASKS.md を生成（役割分担を明記）
  • 各エージェントは AGENT_TASKS.md のみを参照して実装
  • 完了後に PM が成果物を確認・統合
✗ 直接指示パターン（非推奨）
  • PM なしで各エージェントが直接タスクを受け取る
  • エージェント間の依存関係が暗黙的になる
  • どのエージェントが何を実装したか追跡できない
  • 成果物の整合性確認が困難
7 /init + /review で品質を維持する 継続的改善

設定が完了したら、定期的に以下のスラッシュコマンドで品質を維持します。

コマンド タイミング 効果
/init プロジェクト開始時・大きなリファクタ後 コードベースを再探索して AGENTS.md の雛形を再生成
/review PR 作成前 変更差分をコードレビュースキルでチェック
/experimental マルチエージェント初回有効化時 TUI から multi_agents フラグを ON にする

AGENTS.md / AGENTS.override.md — コンテキスト設計

ℹ️
Codex 固有：1ディレクトリ1ファイルのみ採用 + 32 KiB 上限 AGENTS.override.md → AGENTS.md → fallback の順でチェックし、最初にヒットしたファイルのみ読む（同一ディレクトリでのマージはしない）。全体の合計が 32 KiB を超えるとそれ以降を切り捨てる。config.toml の project_doc_max_bytes = 65536 で引き上げ可能。
📋 AGENTS.md に書くべきこと / 書かないべきこと
✅ 書くべき内容
  • プロジェクト概要・目的（2〜3文）
  • ビルド・テスト・Lint コマンド（必須）
  • テスト実行の要求（「変更後は必ずテストを実行」）
  • PR メッセージのフォーマット・規約
  • サブエージェント委譲ルール
  • 禁止操作・危険コマンドの明示
  • コーディング規約（コンパクトに）
  • 重要ドキュメントへのパス参照
✗ 書かないべき内容
  • 32 KiB を超える内容（スキルに分割）
  • MCP / モデル設定（config.toml に分離）
  • 長大なコードスニペット
  • 特定タスク専用の詳細手順（SKILL.md へ）
  • 機密情報・APIキー
  • 修正履歴・変更ログの蓄積
📄 実践的な AGENTS.md テンプレート（サブエージェント対応版）
AGENTS.md (project root)Markdown
# PROJECT: my-saas-app ## Overview Next.js 15 + Supabase + Stripe のマルチテナント SaaS。 本番: Vercel Edge / DB: Supabase (PostgreSQL) / AI: OpenAI Responses API。 ## Build & Test (CRITICAL: always run after changes) - Build: `pnpm build` - Test: `pnpm test` — コード変更後は必ず実行すること - Lint: `pnpm lint` — PR前に必ず実行すること - DB types: `pnpm supabase gen types` - E2E: `pnpm test:e2e` (Playwright) ## Multi-Agent Dispatch Rules Parallel spawn 条件（すべて満たす場合のみ）: - タスクが独立していて互いに依存しない - 異なるドメイン (frontend / backend / db) に閉じている - 同一ファイルへの書き込みが発生しない Sequential / single-agent: - 依存関係がある（B に A の出力が必要） - 共有ファイルへの書き込みが発生する ## Domain Boundaries - frontend → app/, components/, styles/ のみ - backend → supabase/functions/, lib/server/ のみ - database → supabase/migrations/, schema/ のみ ## PR Message Format ``` ## Summary [変更の概要 2〜3文] ## Changes - [変更点1] - [変更点2] ## Tests - [テスト内容] ``` ## Forbidden Operations - `supabase db reset` は絶対に実行しない - `.env.production` の読み書き禁止 - 新しい本番依存関係の追加は確認を求めること
⚡ AGENTS.override.md — ホットスワップ型上書きパターン
services/payments/AGENTS.override.mdMarkdown
# Payments Service — Emergency Override # このファイルが存在する間、payments/ での作業は下記ルールのみ適用される # 通常の AGENTS.md はこのディレクトリでは読まれない ## Active Incident Protocol 本番障害対応中。以下のルールを最優先で守ること: - テストコマンド: `make test-payments`（通常の `pnpm test` ではなく） - API キーのローテーションは セキュリティチャンネルに通知後のみ実施 - Stripe Webhook の変更は `--dry-run` フラグで確認後に実行 - すべての変更を INCIDENT_LOG.md に記録すること # 障害解消後はこのファイルを削除して通常の AGENTS.md に戻す

🔑 使いどころ：特定サービスで一時的に異なるルールを適用したい場合（本番障害対応・セキュリティレビュー中・テストモードなど）に、AGENTS.md を編集せずにホットスワップできます。完了後は削除するだけで元に戻ります。

💡
/init コマンドで AGENTS.md を自動生成する /init をプロジェクトルートで実行すると、Codex がコードベースを探索して AGENTS.md の雛形を自動生成します。生成されたファイルをレビューして編集し、コミットすることで次回から継続的に利用できます。
2.5

サブエージェント向け AGENTS.md 設計パターン詳解

役割別・依存関係・ハンドオフの設計

サブエージェント開発において、AGENTS.md は単なる「ルール書き」ではありません。各エージェントが自律的に動くための「コンテキスト・コントラクト」です。設計の要点は「そのエージェントが何を知るべきか」と「何を知るべきでないか」の両方を明示することです。

🗂 役割別 AGENTS.md 設計マトリクス
エージェントロール AGENTS.md に書くべき内容 参照すべきファイル ドメイン境界
PM Project Manager 成果物ファイル名・フォーマット・品質基準・ゲート条件 ユーザー要件のみ 全ディレクトリへの読み書き
FE Frontend Developer フレームワーク規約・コンポーネント設計ルール・スタイルガイド REQUIREMENTS.md, AGENT_TASKS.md, design_spec.md frontend/ のみ
BE Backend Developer API設計規約・エラーハンドリング規則・認証フロー REQUIREMENTS.md, AGENT_TASKS.md backend/, lib/server/ のみ
DB DB Migrator マイグレーション命名規則・ロールバック必須ルール・本番適用禁止 REQUIREMENTS.md の DB スキーマ定義のみ migrations/, schema/ のみ
QA Tester テストフレームワーク・カバレッジ要件・テスト命名規則 TEST.md, REQUIREMENTS.md tests/, *.spec.ts のみ
SEC Security Reviewer チェックリスト（OWASP Top 10 等）・報告フォーマット 差分ファイルのみ（Read-Only） 書き込み禁止
📋 PM エージェント用 AGENTS.md テンプレート
docs/pm/AGENTS.md（PM エージェント専用）Markdown
# Project Manager Agent ## Role あなたは PM エージェントです。タスクを受けたら下記の3ファイルを必ず作成し、 各専門エージェントにハンドオフしてください。 ## Required Output Files（ゲート: これが存在しないと次フェーズに進めない） 1. REQUIREMENTS.md — 製品目標・機能要件・制約・技術スタック 2. AGENT_TASKS.md — 各エージェントへの具体的指示（曖昧さゼロ） 3. TEST.md — 受け入れ基準・テスト計画（Tester 向け） ## AGENT_TASKS.md フォーマット（必ず守ること） ``` ## [ロール名] - 担当: [何をするか 1〜2文] - 成果物: [ファイルパス・形式を明示] - 参照: [参照すべきファイル] - 制約: [やってはいけないこと] ``` ## Handoff Gate（以下を確認してからハンドオフ） - [ ] 全成果物ファイルが存在する - [ ] テストが全件パスしている - [ ] domain boundary の侵犯がない - [ ] AGENT_TASKS.md の全チェックボックスが完了している ## Forbidden - 成果物ファイルなしでサブエージェントを起動しない - 依存関係のあるタスクを同時並列スポーンしない
🔀 ハンドオフ設計のベストプラクティス
✅ 良いハンドオフ設計
  • transfer 関数のみを使う（メッセージ直渡し禁止）
  • 引き継ぎ先ファイルのパスを AGENTS.md に明記
  • ゲート条件（チェックボックス）を AGENTS.md に定義
  • 完了後は AGENT_TASKS.md のチェックを更新
  • 依存タスクは順次スポーン（直列）
✗ 悪いハンドオフ設計
  • ハンドオフ先をメッセージで直接渡す
  • ゲート条件なしで並列スポーン
  • AGENTS.md に引き継ぎ情報を書かない
  • 完了確認なしで次フェーズに進む
  • max_depth を 3 以上に設定する
ハンドオフ AGENTS.md — design_spec 待機パターンMarkdown
## Handoff Protocol (Frontend / Backend 共通) # このファイルが存在することを確認してから実装を開始すること **依存ファイル確認（実装開始前にチェック）:** - [ ] REQUIREMENTS.md が存在する - [ ] AGENT_TASKS.md が存在し、自分の担当セクションが記載されている - [ ] design_spec.md が存在する（Frontend は必須） # 上記ファイルが存在しない場合は実装を開始せず PM に確認すること ## Completion Handoff 実装完了後は以下を実施してから transfer_to_project_manager を呼ぶ: 1. テストを実行し全件パスを確認 2. AGENT_TASKS.md の自分のチェックボックスを完了にする 3. 成果物ファイルのパス一覧を引き継ぎメモに記載
🏗 ドメイン分離パターン — サブディレクトリ AGENTS.md の活用
🔑
原則：サブエージェントは自分のドメイン外を知らなくてよい。Frontend エージェントはDB スキーマを知る必要がなく、Backend エージェントはUIコンポーネント名を知る必要はありません。AGENTS.md で「このエージェントが見るべきものと見てはいけないもの」を明示的に定義することで、エージェントの誤動作を防ぎます。
モノレポ構成でのドメイン分離例File Tree
my-app/ ├── AGENTS.md # ← 全エージェント共通（概要・テストコマンド） ├── frontend/ │ ├── AGENTS.md # ← Frontend エージェント専用ルール │ │ # 「frontend/ のみ変更可」と明記 │ └── components/ ├── backend/ │ ├── AGENTS.md # ← Backend エージェント専用ルール │ │ # 「backend/ のみ変更可」と明記 │ └── api/ ├── services/ │ └── payments/ │ ├── AGENTS.md # ← Payments サービス固有ルール │ └── AGENTS.override.md # ← 本番障害時の緊急上書き（平時は不要） └── .codex/ ├── config.toml # ← マルチエージェントロール定義 └── roles/ ├── explorer.toml └── reviewer.toml

SKILL.md — 遅延ロード型スキルの設計

Codex 独自の概念。Claude / Gemini にはない

SKILL.md は AGENTS.md と根本的に違います。 AGENTS.md は毎回フルロードされますが、SKILL.md は タスクに合致すると判断されたときのみ フル内容が読み込まれます（プログレッシブ・ディスクロージャー）。これによりコンテキストウィンドウを節約しながら、豊富な専門知識を持てます。

⚠️
description フィールドが「スキルのAPIドキュメント」 Codex が「このスキルを使うかどうか」を判断するのは description のみです（スキルメタデータ段階ではフル内容は読まない）。description には「いつ使うべきか・いつ使わないべきか」の境界を明確に書いてください。
.agents/skills/code-review/SKILL.mdMarkdown + YAML frontmatter
--- name: code-review description: | コードレビューを実施するスキル。以下の場合にトリガーする: - PR を作成する前にレビューしてほしい - セキュリティ・パフォーマンス・型安全性を確認してほしい - コードの品質チェックを依頼された場合 以下の場合はトリガーしない: - 単純なファイル作成・移動 - ドキュメントのみの変更 - 設定ファイルの軽微な変更 --- ## Role あなたはシニアエンジニアのコードレビュアーです。 提供されたコードをセキュリティ・パフォーマンス・品質の観点でレビューします。 ## Review Checklist ### 🔴 Security（セキュリティ） - SQLインジェクション / XSS / CSRF リスク - 認証・認可の適切な実装（エンドポイントごとの検証） - 秘密情報のハードコードなし - 入力バリデーション（zod スキーマ必須） ### 🟡 Performance（パフォーマンス） - N+1 クエリ問題の有無 - 不要な再レンダリング（React.memo / useMemo / useCallback） - 重い同期処理の非同期化 ### 🟢 Code Quality（品質） - TypeScript strict モードへの準拠 - エラーハンドリングの網羅 - テストカバレッジ（変更部分に対するテスト） - 命名規則の一貫性 ## Output Format ``` ## Code Review Report — {filename} ### 🔴 Critical (必ず修正) ### 🟡 Warning (推奨修正) ### 🟢 Good (良い点) ### 📋 Summary ```

config.toml — マルチエージェントロール定義

Codex のマルチエージェントワークフローでは、各サブエージェントの定義は config.toml の [agents] セクションで行います。各ロールには 専用の config.toml ファイル（config_file）を指定でき、それによってモデル・sandbox・承認ポリシーをロールごとに設定できます。

🔍
explorer（組み込み）
コードベースの探索・検索専用。Haiku相当の軽量モデルで並列実行。Read-only sandbox。
model = "gpt-4.1-mini"
sandbox = "read-only"
🖊️
default（組み込み）
汎用コーディングエージェント。メインのタスク実行ロール。workspace-write sandbox。
model = "gpt-5.3-codex"
sandbox = "workspace-write"
📋
reviewer（カスタム例）
コードレビュー専用。Read-only で never approval。高精度モデルで品質確認。
model = "gpt-5.4"
approval_policy = "never"

# .codex/config.toml — マルチエージェントワークフロー設定 # Codex がこのプロジェクトを trusted と判断した場合のみロードされる model = "gpt-5.3-codex" # プロジェクトデフォルトモデル approval_policy = "on-request" # インタラクティブ実行向け # ── プロジェクトドキュメント設定 ────────────────────────── project_doc_max_bytes = 65536 # 64 KiB に引き上げ（大型プロジェクト向け） project_doc_fallback_filenames = ["TEAM_GUIDE.md", "CLAUDE.md"] # AGENTS.md がないディレクトリでは上記をフォールバックとして使用 # ── マルチエージェント設定 ──────────────────────────────── [features] multi_agents = true # 実験的: TUIで /experimental から有効化 [agents] max_concurrent = 8 # 同時実行エージェント上限（デフォルト: 無制限） max_depth = 2 # スポーン最大深度（root=0, 子=1, 孫=2） # ── ロール定義 ──────────────────────────────────────────── [agents.roles.explorer] # 組み込みロールのオーバーライド description = "コードベース探索・検索専門。実装は行わない。" config_file = ".codex/roles/explorer.toml" [agents.roles.reviewer] # カスタムロール: コードレビュー専用 description = """ コードレビューを実施する専門エージェント。 以下の場合に呼び出す: - コード変更が完了した後 - PR作成前のセキュリティ・品質チェック - テストカバレッジ確認 Read-only: コードの変更は行わない。 """ config_file = ".codex/roles/reviewer.toml" [agents.roles.db-migrator] # DBマイグレーション専用（慎重な承認ポリシー） description = "DBスキーマ変更とマイグレーション専門。services/db/ のみ操作可能。" config_file = ".codex/roles/db-migrator.toml" # ── MCP サーバー設定 ───────────────────────────────────── [mcp_servers.filesystem] command = "npx" args = ["-y", "@modelcontextprotocol/server-filesystem", "./src"] [mcp_servers.github] command = "npx" args = ["-y", "@modelcontextprotocol/server-github"] env = { GITHUB_TOKEN = "${GITHUB_TOKEN}" }

Agents SDK マルチエージェント — PM駆動ファイル生成パターン

OpenAI Agents SDK を使ったマルチエージェント開発では、Project Manager エージェントが司令塔となり、REQUIREMENTS.md・AGENT_TASKS.md といったMarkdownファイルを自動生成してチームメンバーエージェント（Designer / Frontend / Backend / Tester）に引き渡します。これが Codex の最大の特徴です。

Agents SDK — PM駆動マルチエージェント実装例Python
from agents import Agent, Runner from codex_mcp import get_codex_mcp_server RECOMMENDED_PROMPT_PREFIX = """ あなたはソフトウェア開発チームの専門エージェントです。 引き継ぎには transfer 関数のみを使い、メッセージで直接渡さないこと。 """ # Codex を MCP サーバーとして使用（ファイルI/O を担当） codex_mcp = get_codex_mcp_server( approval_policy="never", sandbox="workspace-write" ) # ── 各専門エージェント定義 ──────────────────────────────── frontend_agent = Agent( name="Frontend Developer", instructions=RECOMMENDED_PROMPT_PREFIX + """ あなたはフロントエンド開発者です。 AGENT_TASKS.md と REQUIREMENTS.md のみを真実の情報源として実装してください。 担当ディレクトリ: /frontend 成果物: - index.html — メインページ構造 - styles.css — スタイルシート - main.js — インタラクション 実装完了後は transfer_to_project_manager を呼び出すこと。 """, model="gpt-5.3-codex", mcp_servers=[codex_mcp], ) backend_agent = Agent( name="Backend Developer", instructions=RECOMMENDED_PROMPT_PREFIX + """ あなたはバックエンド開発者です。 AGENT_TASKS.md に記述されたエンドポイントのみを実装してください。 担当ディレクトリ: /backend 成果物: - server.js — API実装 (GET /health, GET/POST /scores) - package.json — start スクリプトを含める 実装完了後は transfer_to_project_manager を呼び出すこと。 """, model="gpt-5.3-codex", mcp_servers=[codex_mcp], ) # ── Project Manager（司令塔）──────────────────────────────── pm_agent = Agent( name="Project Manager", instructions=RECOMMENDED_PROMPT_PREFIX + """ あなたは PM です。タスクを受け取ったら以下の3ファイルをプロジェクトルートに作成してください: 1. REQUIREMENTS.md — 製品目標・対象ユーザー・主要機能・制約 2. AGENT_TASKS.md — 各エージェントへの具体的な指示 3. TEST.md — テスト計画（Tester エージェント向け） ファイル作成後、各チームメンバーに transfer で引き渡し、 成果物の確認を行い、品質基準を満たしたら完了とする。 """, model="gpt-5.4", mcp_servers=[codex_mcp], handoffs=[frontend_agent, backend_agent], )
🔍
Agents SDK 組み込みトレーシング（2026年〜） Agents SDK には built-in tracing が統合されており、generation step・tool calls・handoffs・guardrails・カスタムイベントトレーシング・trace dashboard・trace grading をワンストップで行えます（Agents SDK tracing ドキュメント）。
一方、spawn_agents_on_csv による CSV からのエージェントファンアウトと ETA 付き進捗追跡は Codex CLI（rust-v0.105.0+）固有の機能です（Codex CLI リリースノート）。
🚨
Sub-agents は親の承認を継承しない サブエージェントは親のサンドボックスポリシーを継承しますが、非インタラクティブな承認で実行されます。承認が必要なアクションをサブエージェントが試みると、そのアクションは失敗してエラーが親ワークフローに返されます。各ロールの config.toml で明示的に approval_policy = "never" を指定するのがベストプラクティスです。

💡
プロファイル機能でコスト最適化 config.toml の [profiles] で複数の設定プロファイルを定義し、codex --profile deep-review のように切り替えられます。通常開発は gpt-4.1、セキュリティレビューは gpt-5.4 + reasoning_effort=high のように使い分けることでコストを大幅に削減できます。

絶対に避けるべき Anti-Patterns

✗ AGENTS.md の Anti-Patterns
  • 空ファイルを作る（Codex がスキップするため無効）
  • 32 KiB を超える内容を1ファイルに詰める
  • MCP設定・モデル設定を記述（config.toml に分離）
  • 特定タスク専用の詳細手順（SKILL.md に分離）
  • 修正履歴・変更ログの蓄積
  • 機密情報・API キーの記述
✗ マルチエージェントの Anti-Patterns
  • config.toml でロールを定義せずにスポーン
  • サブエージェントの approval_policy を未設定のまま使う
  • SKILL.md の description を曖昧にする（誤発動）
  • max_depth を深くしすぎる（デフォルト 1 推奨）
  • 全ロールに gpt-5-pro を使う（コスト爆発）
  • AGENTS.override.md を削除せず放置（永続的な上書き）

まとめ：各ファイルの役割と設計原則

ファイル 読者 設計原則 アンチパターン
~/.codex/AGENTS.md 全プロジェクト共通 個人の Working agreements のみ。プロジェクト固有内容は書かない プロジェクト固有ルールを混在
AGENTS.md (root) メインエージェント（常時ロード） 32 KiB 以内。概要・テストコマンド・PR規約・禁止操作 空ファイル・大量スニペット・config設定
subdir/AGENTS.md そのディレクトリ作業時のみ サービス固有ルール。rootと重複させない rootと同じ内容の重複記述
AGENTS.override.md エージェント（一時的に優先） 緊急時・一時上書き専用。完了後は必ず削除 通常運用で使い続ける
.agents/skills/*/SKILL.md エージェント（遅延ロード） description を明確に。いつ使う/使わないかを記述 曖昧な description・巨大な1ファイル
config.toml Codex ランタイム モデル・ロール・MCP・sandbox を集中管理 APIキーの直接記述
REQUIREMENTS.md
