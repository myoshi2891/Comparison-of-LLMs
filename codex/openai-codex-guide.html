OpenAI Codex とは何か？
まず全体像を把握しよう

OpenAI Codex は、ソフトウェア開発に特化したAIコーディングエージェントです。2025年5月にクラウドエージェントとして登場し、2026年3月現在ではGPT-5.4を搭載した本番利用可能な開発インフラへと進化しています。単なるコード補完ではなく、リポジトリ全体を理解・操作できるエージェントです。

✦ できること
  • ✓
    機能追加・コード生成
  • ✓
    バグ検出・修正
  • ✓
    コードレビュー・説明
  • ✓
    リファクタリング・テスト生成
  • ✓
    PRの自動提案
  • ✓
    大規模なコードベース探索
✦ 利用方法 (3経路)
🌐
Codex App (Web/Cloud)
ブラウザから並列タスク実行
💻
Codex CLI (Terminal)
ローカル環境でエージェント実行
🔧
IDE Extension
VS Code等に統合
💡
プラン確認：ChatGPT Plus / Pro / Business / Edu / Enterprise に含まれています。追加料金なしで利用可能です（API利用はOpenAI APIレートに準じます）。
1
セットアップ
Codex CLI を5分でインストールする
1
Node.js のインストール確認
Codex CLI は npm 経由でインストールします。まず Node.js が入っているか確認。
bash
# バージョン確認 (v18以上推奨)
node --version
npm --version
2
Codex CLI のインストール
npm でグローバルインストールします。macOS/Linux で動作。Windows は WSL 推奨。
bash
# グローバルインストール
npm install -g @openai/codex

# または npx で直接実行 (インストール不要)
npx @openai/codex
3
認証 (初回サインイン)
初回起動時に ChatGPT アカウントまたは API キーで認証します。
bash
# ターミナルを開いて実行 → ブラウザでサインインを促される
codex

# API キーで使う場合
export OPENAI_API_KEY="sk-..."
codex
4
バージョンアップデート
定期的にリリースがあるので最新版を保つことが重要です。
bash
# 最新版に更新
npm update -g @openai/codex
⚠️
Windows ユーザーへ：Windows のネイティブサポートは実験的段階です。最良の体験のためには WSL (Windows Subsystem for Linux) ワークスペースでの使用を強く推奨します。
2
モデル選択
タスクに応じて最適なモデルを使い分ける

2026年3月現在、Codex では以下のモデルが利用可能です。適切なモデル選択がコスト効率と品質に直結します。

gpt-5.4
★ 推奨
コーディング・推論・エージェントワークフローを統合したフラッグシップモデル。GPT-5.3-Codexの業界最高のコード生成能力を内包。ほとんどのタスクはこちらから始めよう。
gpt-5.4-mini
軽量・高速版。CRUD操作、コード探索、大ファイルレビューなど軽めのタスク向け。gpt-5.4の30%のリソース消費なので長時間作業に最適。
2x速い・低コスト
gpt-5.3-codex-spark
リアルタイムコーディングに最適化した研究プレビューモデル。ほぼ即時のコードイテレーションが可能。ChatGPT Pro ユーザー向け。
Pro限定
gpt-5.3-codex
コーディング特化モデルの前世代フラッグシップ。Responses API でも利用可能。GPT-5.4に後継されたが現在も有効。
モデルの切り替え方法
bash / config
# CLI起動時にモデル指定
codex -m gpt-5.4

# 実行中に /model コマンドで切り替え
/model gpt-5.4-mini

# config.toml で永続設定 (~/.codex/config.toml)
[model]
default = "gpt-5.4"
💚
使い分けの鉄則：複雑な設計・アーキテクチャ判断・最終レビューには gpt-5.4、コードベース探索・大量ファイルのレビュー・サブエージェントには gpt-5.4-mini を使うことでリソースを節約できます。
3
プロンプト設計のベストプラクティス
「何をしたいか」を正確に伝える4つの要素
ℹ️
Codex は完璧なプロンプトがなくても動作しますが、明確なプロンプト = より信頼性の高い結果につながります。特に大規模コードベースでは効果絶大です。
✦ 良いプロンプトの4要素（公式ガイドより）
🎯
1. コンテキスト (Context)
関連するファイル・フォルダ・ドキュメント・エラー情報を伝える。@filename でファイルをメンション可能。
📋
2. タスク (Task)
何を実装・修正・説明してほしいかを具体的に書く。「なんとなく直して」より「〇〇という理由でXXを修正して」が効果的。
✅
3. 完了条件 (Done criteria)
「何をもって成功とするか」を定義する。テストが通るか、特定の出力が得られるかなど。
🚧
4. 制約 (Constraints)
やってほしくないこと・守るべきルールを明示。「既存のAPIを変更しない」「依存関係を増やさない」など。
✦ 悪い例 vs 良い例
❌ 悪い例 ✅ 良い例
「バグを直して」 「@src/auth.ts の JWT 検証で、トークン期限切れ時に500エラーが返る問題を修正して。401を返すようにすること。既存のミドルウェア構造は変えないで。」
「テスト書いて」 「@src/utils/parser.ts のすべての公開関数にJestのユニットテストを追加して。境界値とエラーケースをカバーすること。既存のテストパターン (@tests/) に合わせて。」
「レスポンシブにして」 「@components/Dashboard.tsx をモバイル対応に。ブレークポイントは 768px。Tailwind のみ使用。既存のデザインシステムのクラス名を変えないこと。」
実践例：プロンプトテンプレート
# コンテキスト
@src/api/users.ts と @src/types/user.ts を参照してください。
Node.js + Express + TypeScript プロジェクトです。

# タスク
ユーザー作成エンドポイント (POST /api/users) に
メールアドレスの重複チェックを追加してください。

# 完了条件
- 重複時は 409 Conflict を返す
- npm test が通ること

# 制約
- データベーススキーマは変更しない
- 新しい依存パッケージを追加しない

AGENTS.md の活用
Codexにプロジェクトのルールを「記憶」させる最強の方法

AGENTS.md は README.md のようなテキストファイルです。リポジトリのルートや特定ディレクトリに配置することで、Codex がタスク開始前に自動的に読み込みます。「毎回プロンプトに書くルール」を一度定義すれば、以後すべてのタスクに適用されます。

読み込み優先順位（高 → 低）
~/.codex/AGENTS.override.md グローバル一時オーバーライド
~/.codex/AGENTS.md グローバルデフォルト設定
/repo-root/AGENTS.md プロジェクト共通設定
/subdir/AGENTS.override.md サブディレクトリ固有ルール
AGENTS.md テンプレート（コピーして使おう）
# AGENTS.md

## プロジェクト概要
- Node.js + Express + TypeScript の REST API
- データベース: PostgreSQL (Prisma ORM)
- テストフレームワーク: Jest

## コーディング規約
- コメントは日本語で記述すること
- 関数の引数が3つ以上の場合はオブジェクト型を使う
- any 型は使用禁止。unknown を使うこと
- エラーハンドリングは必ず Result 型でラップする

## テスト・ビルドコマンド
- テスト実行: `npm test`
- Lint チェック: `npm run lint`
- ビルド: `npm run build`
- ⚠️ PRを開く前に必ず lint を実行すること

## 禁止事項
- package.json への新規依存追加は確認を取ること
- データベースのスキーマ変更は migration ファイルを作成すること
- console.log のコミットは禁止（logger を使うこと）
サブディレクトリへのオーバーライド例
# services/payments/AGENTS.override.md
## Payments サービス固有ルール
- テストコマンド: `make test-payments` (npm test ではなく)
- API キーのローテーション時は必ずセキュリティチャンネルに通知
- PCI-DSS 準拠のため、カード情報はログに出力しないこと
💚
プロのヒント：AGENTS.md にテストコマンドを明記することが最も重要です。Codex は「何をもって完了か」を知るためにテストを実行します。テスト方法が分からないとコードを書いて終わりになってしまいます。
5
実践ワークフロー
1日の開発ルーティンに Codex を組み込む
✦ プロが実践するモーニングルーティン

WorkOS のエンジニアチームが実践する方法：朝のコーヒーを飲む前に、3〜5個のタスクを Codex に並列実行させる。コーヒーを飲み終わる頃にはレビュー待ちのPRが積まれている。

bash — 朝のバッチタスク例
# タスク1: TypeScript エラー修正
codex "@src/onboarding/validator.ts の TypeScript エラーをすべて修正して。型安全性を保つこと"

# タスク2: スキーマ更新対応
codex "@api/webhooks.ts を新しいイベントスキーマに対応させて。@docs/schema-v2.md を参照"

# タスク3: テストカバレッジ向上
codex "@src/components/AdminDashboard.tsx のエラーバウンダリを改善して"

# ↑ これらが並列実行される間に他の作業を進められる
✦ Approval モード（安全設定）の理解
モード コマンド 動作 推奨場面
suggest codex 変更前に毎回確認 初めてのリポジトリ・重要な変更
on-request codex --local ローカル書き込みは許可、外部は確認 日常的な開発作業
auto-edit --full-auto ほぼ自動承認 CI/CD・信頼済みワークフロー
✦ コードレビューを Codex に任せる
bash
# コミット前にレビュー実行
codex /review

# または exec コマンドでスクリプト化
codex exec "git diff HEAD~1 をレビューして、潜在的なバグとセキュリティ問題を報告して"

上級テクニック
MCP・Skills・Automations・画像入力
🔌
MCP連携
Model Context Protocol

外部ツール（GitHub, Slack, Jira など）をCodexに接続。コードと外部サービスをシームレスに連携できます。 

# config.toml でMCPサーバーを設定
[[mcp_servers]]
name = "github"
command = "npx @modelcontextprotocol/server-github"
⚡
Skills（スキル）
繰り返しワークフローを定義

繰り返し実行するワークフローをスキルとして定義。チームで共有することでオンボーディングも効率化。 

# ~/.agents/skills/add-tests.md
## タスク
指定ファイルのユニットテストを追加する
## 手順
1. 既存テストのパターンを確認
2. 境界値・エラーケースを網羅
3. npm test で確認
🤖
Automations
定期実行・スケジューリング

Codex Appの「Automations」タブで定期実行を設定。CHANGELOG更新・依存関係チェックなどを自動化できます。

💡
Skills = 方法を定義、Automations = スケジュールを定義、という分担が最適です。
🖼️
画像入力
スクリーンショット・デザインから実装

UI のスクリーンショットやデザインモックを入力として渡すと、それをベースにコードを生成できます。 

# 画像を添付してプロンプト実行
codex "このデザインを実装して" \
  -i ./mockup.png
✦ CI/CD での自動化例（GitHub Actions）
yaml — .github/workflows/codex-changelog.yml
jobs:
  update_changelog:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Update CHANGELOG via Codex
        run: |

          codex exec "前回タグからのコミットをもとに CHANGELOG を更新して"
7
やること・やってはいけないこと
公式ベストプラクティスまとめ
✅ DO — やるべきこと
  • ✓
    AGENTS.md で永続的なルールを管理する
  • ✓
    タスクは1スレッド=1タスクに分ける（プロジェクト全体を1スレッドにしない）
  • ✓
    テスト環境を整備してから Codex に渡す
  • ✓
    Codex の作業と並行して自分の作業を進める
  • ✓
    生成されたコードは必ず人間がレビューする
  • ✓
    軽いタスクには mini モデルを使ってコスト節約
  • ✓
    スクショ・デザインモックを画像入力として活用
❌ DON'T — やってはいけないこと
  • ✗
    プロンプトに毎回同じルールを書く（AGENTS.md に移す）
  • ✗
    Codex の作業を1ステップずつ監視する（並列活用できない）
  • ✗
    テスト環境がないまま複雑な実装タスクを渡す
  • ✗
    レビューなしで本番環境にマージする
  • ✗
    1スレッドにプロジェクト全体を扱わせる
  • ✗
    --yolo フラグを本番コードに使う（サンドボックスを無効化）
