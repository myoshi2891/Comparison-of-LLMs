01.OpenAI Codexとは — Cloud版 vs CLI版 vs Agents SDK

OpenAI Codexは2025年5月にリリースされたクラウドベースのソフトウェアエンジニアリングエージェントです。かつての「コード補完モデル」とは全く別物で、現行最新モデルのGPT-5.3-Codex（GPT-5.2-Codex比25%高速、SWE-Bench ProでSoTA。軽量版GPT-5.3-Codex-Sparkは1000+ tokens/secのリアルタイム向け）を搭載し、GitHubリポジトリを自律的に操作・PR提案まで行います。

☁️ Cloud Codex

専用Codex App（Web・デスクトップ、2026年3月Windows版リリース）から利用。クラウドサンドボックスでタスクを並列実行。GitHub連携でPRを自動作成。ChatGPTサイドバーからも引き続き利用可。

💻 Codex CLI

ターミナルで動作するオープンソースエージェント（npm install @openai/codex）。ローカルリポジトリを直接操作。TUI（テキストUI）とHeadlessモードを持つ。Fast modeがデフォルト有効、音声入力（スペースキー長押しで録音・文字起こし）対応。

🤖 Agents SDK

Python/TypeScriptでマルチエージェントパイプラインを構築するSDK。Codex MCP経由でCodex CLIをサブエージェントとして呼び出す。並列SDD実装を実現。

03.AGENTS.md — プロジェクト永続メモリ（オープン標準）

AGENTS.mdはClaude CodeのCLAUDE.mdに相当するCodexの最重要ファイルです。Codexはタスクを開始する前に必ずAGENTS.mdを読み込み、コーディング規約・テストコマンド・PR手順などのコンテキストを取得します。公式システムメッセージにも「AGENTS.mdに記載されたテストを全件実行する」と明記されています。

ベストプラクティス完全テンプレート

AGENTS.md — プロジェクトルート 
# AGENTS.md
# READMEのエージェント版。Codexはタスク開始前に必ずこのファイルを読む。

## Project Overview
# 1〜2文で目的を記述。Codexが全判断の基準とする。
ECサイトのバックエンドAPI（Go製マイクロサービス）。
プリオーダー・決済・在庫管理の3サービスで構成。

## Repository Structure
# Codexがコードベースを理解するための地図
- cmd/       — サービスエントリーポイント
- internal/  — ビジネスロジック（外部公開不可）
- pkg/       — 再利用可能パッケージ
- migrations/— DBマイグレーション（直接編集禁止）
- docs/      — 仕様書群（spec / requirements / design / tasks）

## Build & Test Commands
# ★最重要: codex-1のシステムメッセージがここに書かれたテストを全件実行する
- Build:    `go build ./...`
- Test all: `go test ./... -race -timeout 120s`
- Lint:     `golangci-lint run`
- Coverage: `go test ./... -coverprofile=coverage.out && go tool cover -html=coverage.out`
- Dev:      `docker compose up -d`

## Code Style
- Go 1.23 modules。CGO無効（CGO_ENABLED=0）
- エラーハンドリング: `fmt.Errorf("context: %w", err)` 形式
- テスト: table-driven tests 必須
- コメント: GoDoc形式（英語）
- パッケージ名: 単数形・短縮なし

## Architecture Constraints
- サービス間通信: gRPCのみ（REST禁止）
- ORM使用禁止（pgx v5 raw SQL のみ）
- グローバル変数の新規追加禁止
- panic() の使用禁止（エラーを返す）

## PR Instructions
# Codex がPRを作成する際の形式を指定（公式機能）
タイトル形式: `[feat/fix/refactor] 短い説明（英語）`
必須セクション:
- Summary: 変更内容の1段落サマリー
- Testing Done: 実行したテストと結果
- Breaking Changes: 破壊的変更があれば記載

## Forbidden Actions
# Codexに絶対にさせてはいけないこと
- migrations/ フォルダへの直接書き込み（docs/SKILL.md の db-migration スキル使用）
- .env ファイルの作成・変更
- 本番DBへのDELETE/DROP（必ず人間確認）
- APIキーのハードコード
⚠️
32KiBの上限を意識せよ
AGENTS.mdは全ディレクトリ分を連結した合計がデフォルト32KiB（~/.codex/config.tomlのproject_doc_max_bytesで変更可）を超えると以降のファイルが切り捨てられます。重要な情報は上に、詳細はSKILL.mdへ分離するか、上限を65536（64KiB）に増やしてください（[1]）。
ℹ️
AGENTS.mdはオープン標準なのでClaude Codeでもそのまま使える
Claude CodeはCLAUDE.mdに@AGENTS.mdでインポートすることで、AGENTS.mdを唯一の真実のソースとして活用できます。ツールを乗り換えても書き直し不要（[10]）。

# Database Migration Skill

## Goal
PostgreSQLスキーマ変更をプロジェクト標準マイグレーション手順で安全に実行する。

## Instructions
1. `migrations/` に `YYYYMMDD_HHMMSS_description.up.sql` を作成
2. ロールバック用 `.down.sql` を必ず同時作成
3. 整合性チェック: `python scripts/run_migration.py --check --env staging`
4. 人間がレビュー後に: `python scripts/run_migration.py --apply`
5. `docs/AGENT_TASKS.md` の対象タスクをチェック済みにする

## Examples
# Few-shot examples で精度を向上させる

**Input**: "usersテーブルにlast_login_atカラムを追加して"
**Output**:
```sql
-- 20250920_143022_add_last_login_at_to_users.up.sql
ALTER TABLE users ADD COLUMN last_login_at TIMESTAMPTZ;
CREATE INDEX CONCURRENTLY idx_users_last_login ON users(last_login_at);
```

## Constraints
- 既存マイグレーションファイルを絶対に編集しない
- NULL制約の後付けはデータ移行計画なしに行わない  
- `DROP TABLE` は人間確認なし禁止
- 本番環境への直接適用は承認が必要

06..prompt.md — 再利用タスクプロンプト

.prompt.mdはAGENTS.mdとは役割が異なる「タスク特化型の再利用プロンプト」です。AGENTS.mdが「プロジェクトの永続的コンテキスト」なのに対し、.prompt.mdは「特定タスクを毎回同じ品質で実行するための手順書」です（[10]）。

⚡
.prompt.md
.github/ または .codex/ 配下 — 再利用タスクプロンプト
AGENTS.mdを補完 タスク特化・再利用
AGENTS.md との違い

AGENTS.mdはプロジェクトレベルの「常識」を提供。.prompt.mdは「テストケースを生成する」「PRの説明文を書く」など特定タスクの実行手順を再利用可能な形で定義。

SKILL.md との違い

SKILL.mdはフロントマターによる意味的トリガーを持つ。.prompt.mdはより単純な形式で、手動で呼び出す「タスクテンプレート」。IDE統合での直接実行を想定。

.github/generate-tests.prompt.md 
# Generate Unit Tests
# このプロンプトを実行すると、選択ファイルのテストを自動生成する

以下の指示に従って `$SELECTED_FILE` の単体テストを生成してください：

1. テストフレームワーク: Go の `testing` パッケージ + `testify/assert`
2. パターン: table-driven tests（全関数に適用）
3. カバレッジ目標:
   - 正常系: 全パスをカバー
   - 境界値: nil/空/最大値
   - エラー系: 全エラーパス
4. モック: `mockgen` でインターフェースをモック（外部依存の場合）
5. ファイル名: `{original_name}_test.go`

テストが全件パスすることを確認してから終了すること。

08.config.toml — Codex設定ファイル

⚙️
config.toml
~/.codex/config.toml — グローバル設定
~/.codex/config.toml — 完全設定例 
# ~/.codex/config.toml
# 変更後は Codex の再起動が必要

[project_docs]
# AGENTS.md の代替ファイル名（既存プロジェクトのファイルを読み込む）
project_doc_fallback_filenames = ["TEAM_GUIDE.md", ".agents.md", "AI_INSTRUCTIONS.md"]

# 読み込み上限（デフォルト 32768 = 32KiB）
project_doc_max_bytes = 65536   # 64KiBに拡張（大規模プロジェクト向け）

[[skills.config]]
# 特定スキルを無効化（削除せず）
path = "/path/to/skill/SKILL.md"
enabled = false
ℹ️
project_doc_fallback_filenamesを設定すると、各ディレクトリでAGENTS.override.md → AGENTS.md → TEAM_GUIDE.md → .agents.mdの順で検索します。既存プロジェクトのドキュメントファイルをCodexに読み込ませるための仕組みです（[1]）。

09.agents/openai.yaml — スキルUIメタデータ

SKILL.mdのフロントマターだけでは表現しきれないCodex App向けのUIメタデータ・呼び出しポリシー・ツール依存関係を定義するYAMLファイルです。Codex AppのスキルセレクターUIのアイコン・表示名・自動呼び出し設定などを制御します。

.agents/skills/db-migration/agents/openai.yaml 
# Codex App UIメタデータ
display_name: DB Migration
icon: 🗄️
category: database

invocation:
  policy: explicit   # explicit: ユーザーが明示的に呼び出す場合のみ
                         # implicit: Codexが自動判断で呼び出す場合も許可
  shortcut: $migrate  # プロンプトでのショートカット名

tool_dependencies:
  # このスキルが必要とするツール（未存在の場合は警告）
  - shell
  - python3

SDD パイプライン — ファイル生成順序とゲート条件 
# Agents SDK × Codex MCP — SDD ファイル依存グラフ

Step 1: PM Agent が生成（ゲート: なし）
  ├── REQUIREMENTS.md   → 製品要件・制約
  ├── AGENT_TASKS.md    → 各役割への指示
  └── TEST.md           → 受け入れ基準

Step 2: Designer Agent（ゲート: AGENT_TASKS.md 存在確認）
  └── /design/design_spec.md  → DOM構造・UI仕様

Step 3: 並列実行（ゲート: design_spec.md 存在確認）
  ├── Frontend Agent  → /frontend/{index.html, styles.css, main.js}
  └── Backend Agent   → /backend/{server.go, handlers/}

Step 4: Tester Agent（ゲート: frontend + backend 完了確認）
  ├── /tests/TEST_PLAN.md  → 手動チェックリスト
  └── /tests/test.sh       → 自動テストスクリプト

Step 5: PM Agent 最終確認
  └── TEST.mdの受け入れ基準と全成果物の整合性チェック
⚡
ゲート条件（Handoff Gating）がSDDの鍵
公式サンプルコードではdesign_spec.mdが存在することを確認してから並列ハンドオフと明示されています。AGENT_TASKS.mdに「依存ファイルが存在しない場合は停止」と書くことで、前フェーズ未完了での実装開始を防ぎます（[5]）。
🔍
Agents SDK 組み込みトレーシング（2026年〜）
Agents SDKにはbuilt-in tracingが統合されており、マルチエージェントフローの可視化・デバッグ・評価・ファインチューニングをワンストップで行えます。またspawn_agents_on_csvでCSVからエージェントをファンアウトしETA付き進捗追跡が可能です。サブエージェントへのニックネーム付与と子スレッド承認プロンプト表示により、複雑なパイプラインの追跡も容易になっています。
Section 11

11.横断ベストプラクティス 10則

01

AGENTS.mdにテストコマンドを必ず書く

CodexのシステムメッセージはAGENTS.mdに書かれたテストを「全件実行」する。コマンドが未記載だとテストがスキップされる。

02

32KiB上限を意識してファイルを分割

重要なルールはAGENTS.mdの上部へ。詳細な知識はSKILL.md・references/へ分離。上限超過で切り捨てられる。

03

AGENTS.mdはオープン標準を活かす

Claude CodeのCLAUDE.mdに@AGENTS.mdでインポートする。1ファイルで全ツールに対応し「ツール乗り換え問題」を解消。

04

AGENT_TASKS.mdに「あいまいさゼロ」で書く

「成果物のファイル名・パス・形式」を全て明記。「後でデザイナーのDOMを参考にして」ではなく「/design/design_spec.mdのDOM構造を参照」。

05

AGENTS.override.mdを一時的にのみ使う

使い終わったら即削除。永続的ルール変更はAGENTS.mdを直接編集する。override.mdの.gitignore追加を忘れない。

06

SKILL.mdのdescriptionに境界を明記

「Use when...」だけでなく「Do NOT use when...」も書く。暗黙マッチングの誤爆を防ぐnegative examplesが精度向上の鍵。

07

PRインストラクションをAGENTS.mdに書く

Codex Cloudが自動PR生成する際のタイトル形式・必須セクション・レビュアー設定をAGENTS.mdのPR Instructionsセクションで制御。

08

Forbidden Actionsで安全柵を作る

「migrations/への直接書き込み禁止」「.envの変更禁止」を明示。Codexは明示的な禁止を強く尊重する。

09

ゲート条件付きハンドオフで品質を担保

Agents SDKパイプラインでは「design_spec.mdが存在することを確認してから実装開始」のようなゲート条件を各エージェントに持たせる。

10

$skill-installerで公式スキルを活用

openai/skillsの公式カタログとコミュニティのスキル集（40,000+プロジェクト）を積極的に活用。スクラッチで書く前に既存スキルを確認。
