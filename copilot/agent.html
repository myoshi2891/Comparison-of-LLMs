📋 copilot-instructions.md に書くべきこと / 書かないべきこと
✅ 書くべき内容
  • プロジェクト概要・目的（2〜3文）
  • 技術スタック・主要バージョン
  • ビルド・テスト・Lint コマンド
  • コーディング規約（コンパクトに）
  • PR メッセージのフォーマット
  • サブエージェント委譲ルール
  • 禁止操作・危険コマンドの明示
  • ペルソナの定義（「新人開発者向けに」等）
✗ 書かないべき内容
  • 特定パスにしか使わない詳細ルール（*.instructions.md に分離）
  • 専門的な手順（SKILL.md に分離）
  • 長大なコードスニペット（コンテキスト浪費）
  • 機密情報・API キー
  • 修正履歴・変更ログ
  • ツール設定（MCP は別ファイルへ）

.github/instructions/*.instructions.md — パスマッチ型指示

2025年7月〜 GA

*.instructions.md は YAML フロントマターの applyTo でパスグロブを指定することで、特定のファイル・ディレクトリを操作しているときのみ自動適用されます。さらに excludeAgent プロパティで特定のエージェント（coding-agent・code-review）から除外できます。

.github/instructions/frontend.instructions.mdMarkdown + YAML
--- applyTo: "src/components/**,src/app/**" # この指示は src/components/ と src/app/ への変更時のみ適用される # excludeAgent: "code-review" # コードレビューから除外する場合 --- ## Frontend Development Rules ### Component Structure - React Server Components (RSC) を優先し、'use client' は最小限に - コンポーネントは 150行以下に収める - Props は明示的な TypeScript インターフェースを定義する - CSS は Tailwind ユーティリティクラスのみ（インラインスタイル禁止） ### Naming Conventions - コンポーネントファイル: PascalCase.tsx (例: UserCard.tsx) - ページファイル: page.tsx (Next.js App Router 規約) - フック: use プレフィックス (例: useUserProfile) ### DO THIS / NOT THIS ✅ 良い例: ```tsx // Server Component（デフォルト） export default async function UserList() { const users = await getUsers() return
    {users.map(u => )}
} ``` ❌ 悪い例: ```tsx // 不要な 'use client' の使用 'use client' export default function UserList() { ... } // fetchをuseEffectで行っている ```
.github/instructions/testing.instructions.mdMarkdown + YAML
--- applyTo: "**/*.test.ts,**/*.test.tsx,**/*.spec.ts,tests/**" excludeAgent: "code-review" # Copilot code review はこの指示を読まない（テスト専用ルールを汚染しない） --- ## Testing Standards ### Unit Tests (Vitest) - テストは AAA (Arrange / Act / Assert) 形式で記述 - モックは vi.mock() を使用、過剰なモックは避ける - テスト名は「[条件] [状態] [期待結果]」形式 ### E2E Tests (Playwright) - ページオブジェクトパターンを使用 (tests/pages/ に配置) - data-testid 属性でセレクタを指定（クラス名ベースは禁止） - 各テストは独立して実行できること（beforeEach で状態をリセット）
💡
excludeAgent で指示の適用先をコントロールする（2025年11月〜） excludeAgent: "code-review" を設定するとコードレビューに、excludeAgent: "coding-agent" を設定すると Copilot Coding Agent に指示が適用されなくなります。レビュー専用・実装専用の指示を分離するのに有効です。

.github/agents/*.agent.md — カスタムエージェント完全ガイド

🤖 .agent.md とは — 「専門チームメイト」を定義するファイル

*.agent.mdは特定のペルソナ・ツールセット・指示・ハンドオフを持つ専門エージェントを定義するMarkdownファイルです。定義したエージェントは @エージェント名 で呼び出せ、GitHub.com（Coding Agent）・VS Code・JetBrains・Eclipse・Xcodeのすべてで動作します。

エージェントは「常に同じ指示とコンテキストを繰り返し与える」必要をなくし、セキュリティレビュー専門・テスト専門・ドキュメント専門など、役割ごとに特化した振る舞いを一度定義するだけで再利用できます。

📁

そのリポジトリのすべてのコラボレーターが利用可能。git管理でチーム共有。

⚠️

4-3. ステップバイステップ作成ガイド（7ステップ） NEW

01
エージェントの「単一責任」を決める
まず「このエージェントが何をするか / 何をしないか」を1文で言えるか確認します。「テスト生成 + コードレビュー + ドキュメント作成」を1エージェントにまとめるのはアンチパターン。1エージェント = 1専門領域に分割してください。

✅ 良い例:「テストカバレッジを向上させる。本番コードは変更しない」
❌ 悪い例:「開発全般をサポートする」
02
ファイルを作成する
VS Codeでは Copilot Chat の「エージェント選択ドロップダウン → Configure Custom Agents... → Create new custom agent」から作成。または手動でファイルを作成してください。
TERMINAL
# プロジェクト（リポジトリ）レベルのエージェント mkdir -p .github/agents touch .github/agents/security-reviewer.agent.md # 個人レベル（VS Code設定から作成推奨） # Create new custom agent → "User" を選択
03
description を丁寧に書く（最重要）
description はエージェントが自動選択・手動選択される唯一の判断材料です。「Focuses on X without doing Y」パターンが最も効果的です。2,500以上のリポジトリ分析から、descriptionが曖昧なエージェントほど誤呼び出しが多いことが判明しています。
description のベストプラクティスYAML
✅ 明確な description（推奨） description: 'Security reviewer focused on OWASP Top 10 vulnerabilities. Reads code and generates audit reports. Does NOT modify any files.' ❌ 曖昧な description（アンチパターン） description: 'コードを分析するエージェント'
04
ツールを最小権限で設定する
tools フィールドを省略すると全ツールが使用可能になります（便利だが危険）。必要最小限のツールのみ許可してください。特に本番DBや本番環境へのアクセスツールは慎重に扱います。
ツール設定パターンYAML
# Read-Only エージェント（セキュリティ審査・レビュー用） tools: [read_file, grep_search, list_directory] # Write限定エージェント（docs/のみ書き込み可） tools: [read_file, create_file, edit_file] # → 本文で "Write: /docs/ のみ" と明示的に制約を追加 # サブエージェントを呼び出す場合は 'agent' ツールが必須 tools: [read_file, search, agent] # MCPサーバーのツールを個別許可（完全なツール名はサーバー名/ツール名） tools: [read_file, sentry/get_issue_details, sentry/get_issue_summary]
05
本文（プロンプト）を構造化して書く
本文は最大30,000文字のMarkdownです。Role → Instructions → Constraints → Output Format の4セクション構成が推奨されます。
本文の推奨構成Markdown

handoffs は GitHub.com では無視されます。 VS Code・JetBrains・Eclipse・Xcodeのみ有効です。GitHub.com（Copilot Coding Agent）のIssue処理フローでは使用できません。

4-5. Subagents（サブエージェント）— 並列・分離実行 NEW

Subagentsはメインエージェントから独立したコンテキストで実行される子エージェントです。Handoffsが「ユーザーが操作する順次遷移」であるのに対し、Subagentsは「エージェントが自律的に委任する並列実行」です。

🔄 Handoffs vs Subagents — 使い分け
観点HandoffsSubagents
制御ユーザーが各ステップを承認エージェントが自律委任
実行順次（ユーザー操作後）並列実行可能
コンテキスト前のセッションを引き継ぐ完全に独立・隔離
適した場面人間の承認が必要な作業調査・分析の並列実行
必要ツール不要（UIメカニズム）tools: [agent]が必須
ℹ️
Subagentsの主なユースケース
  • コードレビューを複数の視点（セキュリティ・パフォーマンス・可読性）で並列実行
  • 実装前の技術調査（複数ライブラリの比較など）
  • 大規模コードベースの並列分析（デッドコード・重複コード等）
  • コンテキスト節約：サブエージェントは結果のみを返すため、メインエージェントのコンテキストが汚染されない

4-8. .agent.md ベストプラクティス 12則 NEW

01

descriptionは「Focuses on X, does NOT do Y」形式で

2,500リポジトリの分析から、明確なdescriptionを持つエージェントほど誤呼び出しが少ない。50〜150文字で「使う場面」と「使わない場面」を両方書く。

02

ツールは最小権限の原則を徹底する

tools省略=全ツール許可は危険。セキュリティ審査エージェントは read_file+grep のみ。本番DB操作ツールは原則許可しない。必要に応じて人間確認ステップを本文に組み込む。

03

本文Constraintsセクションで「絶対に行わないこと」を明記

「ソースコードを変更しない」「本番DBには書き込まない」「テストを削除しない」などをConstraintsセクションに明示。LLMの非決定性に対する安全網として機能する。

04

handoffsはsend:false（手動承認）を原則にする

send:trueの自動送信は完全自動化できるが、予期しない変更につながるリスクがある。重要な処理の前は send:false でユーザー確認を挟む設計が安全。CI/CD用途のみsend:true。

05

サブエージェント専用は user-invocable: false にする

plannerから呼び出されるimplementerなど、直接呼び出し不要なエージェントは @メニューに表示しない。coordinatorがagents[]で明示すれば上書き可能。エコシステムをクリーンに保てる。

06

File Boundariesを本文に必ず書く

「書き込みは /tests/ のみ」「/docs/ 以外は読み取り専用」のように書き込み範囲を明示する。toolsでの制限に加え、本文でも自然言語で制約を再確認するダブルチェック体制が重要。

07

MCPシークレットはcopilot環境から取得する

mcp-servers の env に直接トークンを書かない。リポジトリ Settings → Copilot → Coding Agent の「copilot」環境に登録し、${{ secrets.COPILOT_MCP_* }} で参照する。gitにシークレットを含めない。

08

handoffsはGitHub.com Coding Agentでは動作しない

handoffsとargument-hintはIDE専用機能（VS Code・JetBrains・Eclipse・Xcode）。GitHub.com上でIssueに割り当てる「Copilot Coding Agent」では無視される。プラットフォーム別の設計が必要。

09

modelはIDEで使用、GitHub.comでは無視

model指定はVS Code等のIDEカスタムエージェント専用。GitHub.comのCopilot Coding AgentはOrg/Repoの設定に従う。配列指定でフォールバックも設定可能: ['Claude Opus', 'GPT-5']

10

agentsでサブエージェントホワイトリストを制限する

agents[]を省略するとdisable-model-invocation:falseの全エージェントがサブエージェント対象になる。予期しない呼び出しを防ぐため、coordinatorエージェントは使用するサブエージェントを明示的に列挙する。

11

エージェントはgit管理してデフォルトブランチに置く

GitHub.com Copilot Coding AgentはデフォルトブランチのエージェントのみIssueトリアージUIに表示する。PR中のエージェントはIDE経由では使えるが、GitHub.comのAgentsパネルには表示されない。

12

Output Formatセクションで出力を標準化する

「テスト完了後は作成ファイル一覧・Pass/Fail数・カバレッジ変化を報告する」のように出力形式を明示する。チームメンバーがエージェントの成果物を予測・評価できるようになる。

📊 Good vs Anti Pattern — .agent.md 設計比較
✅ ベストプラクティス
  • 1エージェント = 1専門領域（単一責任）
  • descriptionに「Use for: ... / Does NOT:...」両方記載
  • toolsを最小権限で明示
  • 本文にConstraints・File Boundaries・Output Format
  • handoffs は send:false（手動承認）を原則
  • サブエージェント専用は user-invocable: false
  • MCPシークレットはcopilot環境変数で管理
  • エージェントをgit管理・デフォルトブランチに配置
✗ アンチパターン
  • description が曖昧・短すぎる（1行）
  • tools 省略（全ツール許可のまま）
  • 本番DBアクセスツールを制限なく許可
  • 「何でもやる汎用エージェント」を1つ作る
  • handoffs を send:true にして確認なく自動実行
  • MCPシークレットをYAMLに直書きしてgitにコミット
  • copilot-instructions.mdと全く同じ内容を重複記述
  • ファイル名に日本語・スペースを使用
💡
copilot-instructions.mdとの位置関係： エージェントのプロンプトはシステムプロンプト内で copilot-instructions.md の下に配置されます。つまり全体のルールは常に尊重されつつ、エージェントがその振る舞いを拡張・上書きできます。エージェント固有のルールを copilot-instructions.md に書く必要はありません（重複・コンテキスト汚染の原因になります）。

絶対に避けるべき Anti-Patterns

✗ copilot-instructions.md
  • ファイルを .github/ 以外の場所に置く（読み込まれない）
  • 特定パス専用ルールを記述（*.instructions.md へ）

  • 長大なコードスニペット（コンテキスト圧迫）
  • 機密情報・API キーの記述
  • 指示が互いに矛盾する記述
✗ *.agent.md / SKILL.md
  • description が曖昧（「何でもやります」エージェント）
  • ファイル境界を設定しないエージェント（無制限書き込み）
  • SKILL.md で applyTo なしに常時ロード状態にする
  • *.agent.md を .github/agents/ 以外に置く
  • excludeAgent を設定せず指示が混在する
  • agent.md ファイル名に許可外文字（スペース等）を使う

まとめ：各ファイルの役割と設計原則

ファイル配置場所ロード方式設計原則アンチパターン
copilot-instructions.md .github/ 直下（必須） 常時 普遍的なルールのみ。プロジェクト概要・スタック・禁止操作 パス固有ルール・専門手順の混在
*.instructions.md .github/instructions/ applyTo applyTo でパスを絞る。excludeAgent でエージェント除外 applyTo なしの全体適用
*.agent.md .github/agents/ @呼び出し description 明確化・ツール最小化・境界（書き込み先）を明示 曖昧なdescription・書き込み先無制限
SKILL.md .github/skills/*/ 遅延ロード description で「いつ使う/使わない」を明確に記述 description が曖昧・常時ロード設定
*.prompt.md .github/prompts/ /コマンド 繰り返しタスクの再利用化。mode と tools を明示 1回しか使わないタスクを定義
*.chatmode.md .github/chatmodes/ モード切替 特定ペルソナ（DBA・セキュリティ等）のツールセット定義 過剰なツール許可
AGENTS.md プロジェクトルート クロスツール Claude/Gemini/Codex と共通の普遍ルール。ツール固有は別ファイル ツール固有構文の混在
