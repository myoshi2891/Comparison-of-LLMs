.github/copilot-instructions.md — 常時ロード型グローバル指示

ℹ️
必ず .github/ 直下に配置する copilot-instructions.md はワークスペースルートの .github/ フォルダ内に置く必要があります。他の場所に置いても読み込まれません。この1ファイルが Chat・Agent Mode・Code Review・Copilot Coding Agent のすべてで常時ロードされます。
📋 copilot-instructions.md に書くべきこと / 書かないべきこと
✅ 書くべき内容
  • プロジェクト概要・目的（2〜3文）
  • 技術スタック・主要バージョン
  • ビルド・テスト・Lint コマンド
  • コーディング規約（コンパクトに）
  • PR メッセージのフォーマット
  • サブエージェント委譲ルール
  • 禁止操作・危険コマンドの明示
  • ペルソナの定義（「新人開発者向けに」等）
✗ 書かないべき内容
  • 特定パスにしか使わない詳細ルール（*.instructions.md に分離）
  • 専門的な手順（SKILL.md に分離）
  • 長大なコードスニペット（コンテキスト浪費）
  • 機密情報・API キー
  • 修正履歴・変更ログ
  • ツール設定（MCP は別ファイルへ）
.github/copilot-instructions.mdMarkdown
# PROJECT: my-saas-app ## Overview Next.js 15 App Router + Supabase + Stripe のマルチテナント SaaS。 本番: Vercel Edge Runtime / DB: Supabase (PostgreSQL) / AI: GitHub Models。 ## Tech Stack - Frontend: Next.js 15 (App Router), TypeScript 5.x, Tailwind CSS - Backend: Supabase Edge Functions (Deno 2.x) - Auth: Supabase Auth + RLS (Row Level Security) - Payment: Stripe Checkout / Webhooks - Testing: Vitest (unit) + Playwright (E2E) ## Persona シニアフルスタックエンジニアとして振る舞うこと。 コードを変更する前に必ず計画を要約すること。 ## Build & Test (CRITICAL: run after every change) - Build: `pnpm build` - Test: `pnpm test` — 変更後は必ず実行すること - E2E: `pnpm test:e2e` (Playwright) - Lint: `pnpm lint` — PR前に必ず実行すること ## Coding Standards - TypeScript strict モード必須 - エラーは Result 型でラップ（throw 禁止） - Supabase クエリは必要カラムのみ .select('col1, col2') - React Server Components を優先し 'use client' を最小化 ## Sub-Agent Dispatch (for Copilot Coding Agent) @test-agent → テスト作成・実行・失敗分析 @docs-agent → ドキュメント生成・更新 (docs/ のみ) @security-agent → セキュリティ監査（コード変更なし） ## PR Message Format ``` ## Summary [2〜3文の概要] ## Changes - [変更点] ## Tests - [テスト内容] ``` ## Forbidden Operations - `supabase db reset` は絶対に実行しない - `.env.production` の読み書き禁止

.github/instructions/*.instructions.md — パスマッチ型指示

2025年7月〜 GA

*.instructions.md は YAML フロントマターの applyTo でパスグロブを指定することで、特定のファイル・ディレクトリを操作しているときのみ自動適用されます。さらに excludeAgent プロパティで特定のエージェント（coding-agent・code-review）から除外できます。

.github/instructions/frontend.instructions.mdMarkdown + YAML
--- applyTo: "src/components/**,src/app/**" # この指示は src/components/ と src/app/ への変更時のみ適用される # excludeAgent: "code-review" # コードレビューから除外する場合 --- ## Frontend Development Rules ### Component Structure - React Server Components (RSC) を優先し、'use client' は最小限に - コンポーネントは 150行以下に収める - Props は明示的な TypeScript インターフェースを定義する - CSS は Tailwind ユーティリティクラスのみ（インラインスタイル禁止） ### Naming Conventions - コンポーネントファイル: PascalCase.tsx (例: UserCard.tsx) - ページファイル: page.tsx (Next.js App Router 規約) - フック: use プレフィックス (例: useUserProfile) ### DO THIS / NOT THIS ✅ 良い例: ```tsx // Server Component（デフォルト） export default async function UserList() { const users = await getUsers() return
    {users.map(u => )}
} ``` ❌ 悪い例: ```tsx // 不要な 'use client' の使用 'use client' export default function UserList() { ... } // fetchをuseEffectで行っている ```
.github/instructions/testing.instructions.mdMarkdown + YAML
--- applyTo: "**/*.test.ts,**/*.test.tsx,**/*.spec.ts,tests/**" excludeAgent: "code-review" # Copilot code review はこの指示を読まない（テスト専用ルールを汚染しない） --- ## Testing Standards ### Unit Tests (Vitest) - テストは AAA (Arrange / Act / Assert) 形式で記述 - モックは vi.mock() を使用、過剰なモックは避ける - テスト名は「[条件] [状態] [期待結果]」形式 ### E2E Tests (Playwright) - ページオブジェクトパターンを使用 (tests/pages/ に配置) - data-testid 属性でセレクタを指定（クラス名ベースは禁止） - 各テストは独立して実行できること（beforeEach で状態をリセット）
💡
excludeAgent で指示の適用先をコントロールする（2025年11月〜） excludeAgent: "code-review" を設定するとコードレビューに、excludeAgent: "coding-agent" を設定すると Copilot Coding Agent に指示が適用されなくなります。レビュー専用・実装専用の指示を分離するのに有効です。

.github/agents/*.agent.md — カスタムエージェント定義

2025年11月〜 GA · 2,500以上のリポジトリで採用

*.agent.md は 特定のペルソナ・ツールセット・指示を持つ専門エージェントを定義します。定義したエージェントは @test-agent・@docs-agent のように @ プレフィックスで呼び出せます。YAML フロントマターで 使用ツール・MCP サーバー・モデルを設定できます。

🧪
test-agent
@test-agent
テスト作成・実行・失敗分析の専門家。tests/ ディレクトリのみ書き込み。ソースコード変更不可。
tools: [run_tests, read_file]
Write: /tests/ のみ
📖
docs-agent
@docs-agent
技術文書の生成・更新専門。src/ を読み取り docs/ のみに書き込む。コードの変更は行わない。
tools: [read_file, create_file]
Write: /docs/ のみ
🔒
security-agent
@security-agent
セキュリティ監査専門。コードを読んでレポートのみ出力。コードの変更は一切行わない。
tools: [read_file, grep]
Write: なし（Read-only）
.github/agents/test-agent.agent.mdMarkdown + YAML
--- name: test-agent description: | QAエンジニアペルソナ。以下の場合に呼び出す: - 新機能のテストを作成する - テストを実行して失敗を分析する - テストカバレッジを向上させる 以下の場合は呼び出さない: - ソースコードの実装 - ドキュメントの作成 model: claude-sonnet-4-6 # ↑ IDEカスタムエージェント専用（VS Code / JetBrains / Eclipse / Xcode）。GitHub.comでは無視される tools: - read_file - create_file - run_terminal_command - grep_search # mcpServers: # MCP サーバーも個別に設定可能 # playwright: # Playwright MCP でブラウザテスト # type: npm # instructions: | 制約: /tests/ ディレクトリのみに書き込むこと。ソースコードは変更しない。 --- ## Role あなたは経験豊富な QA ソフトウェアエンジニアです。 このリポジトリのテストを作成・実行・分析します。 ## Testing Standards - テストフレームワーク: Vitest（ユニット）+ Playwright（E2E） - テストは AAA（Arrange / Act / Assert）形式で記述 - data-testid でセレクタを指定（クラス名ベースは禁止） - 各テストは独立して実行可能であること ## File Boundaries - 書き込み: /tests/ ディレクトリのみ - 読み取り: プロジェクト全体 - ソースコードの変更は絶対に行わない - 失敗しているテストを削除しない ## Commands - テスト実行: `pnpm test` - E2E 実行: `pnpm test:e2e` - カバレッジ: `pnpm test:coverage` ## Output Format テスト完了後は以下の形式で報告: - 作成したテストファイル一覧 - 実行結果（Pass/Fail 件数） - カバレッジ変化
⚠️
description が「いつ@エージェントを呼ぶか」の唯一の判断材料 Copilot が「このエージェントを使うべきか」を判断するのは description の内容だけです。「いつ使う・いつ使わない」の境界を具体的に記述してください。2,500以上のリポジトリ分析から、description が明確なエージェントほど誤呼び出しが少ないことが分かっています。

.github/skills/*/SKILL.md — 遅延ロード型スキル（Open Standard）

2025年12月〜 · VS Code + Copilot CLI + Coding Agent 対応

Agent Skills は プロンプトに関連する場合のみ自動ロードされる「専門知識パッケージ」です。copilot-instructions.md が常時ロードされるのに対して、SKILL.md はタスクマッチ時のみロードされます（プログレッシブ・ディスクロージャー）。Anthropic の Claude Code と同じフォーマットのオープンスタンダードであり、anthropics/skills リポジトリのスキルをそのまま使えます。

✅
Claude Code の .claude/skills/ と互換性あり .claude/skills/ に配置したスキルは GitHub Copilot が自動的に認識します。スキルはクロスツールで動作するオープンスタンダードです。github/awesome-copilot や anthropics/skills リポジトリのコミュニティスキルも利用できます。
.github/skills/playwright-testing/SKILL.mdMarkdown + YAML frontmatter
--- name: playwright-testing description: | Playwright を使った E2E テストを作成・実行するスキル。 以下の場合にトリガーする: - ブラウザベースのテストを作成してほしい - E2E テストを実行・デバッグしてほしい - Playwright の設定・セットアップが必要 以下の場合はトリガーしない: - ユニットテスト（Vitest）の作成 - API テスト（supabase functions test） --- # Playwright E2E Testing ## Setup インストール: `pnpm add -D @playwright/test` 設定: playwright.config.ts（プロジェクトルート） ブラウザインストール: `pnpm playwright install chromium` ## Page Object Pattern すべての E2E テストはページオブジェクトパターンを使用する: ```typescript // tests/pages/LoginPage.ts export class LoginPage { constructor(private page: Page) {} async login(email: string, password: string) { await this.page.getByTestId('email').fill(email) await this.page.getByTestId('password').fill(password) await this.page.getByTestId('submit').click() } } ``` ## Commands - 全 E2E 実行: `pnpm playwright test` - UI モード: `pnpm playwright test --ui` - トレース記録: `pnpm playwright test --trace on` ## Test Structure Rules - data-testid 属性でセレクタを指定（クラス名は変わりやすいため禁止） - beforeEach でテスト状態をリセット - 各テストは独立して実行可能なこと（テスト間の依存禁止） # スキルディレクトリには追加リソースも含められる # ./test-template.ts — テストテンプレート（参照可能） # ./setup-guide.md — 詳細セットアップガイド

絶対に避けるべき Anti-Patterns

✗ copilot-instructions.md
  • ファイルを .github/ 以外の場所に置く（読み込まれない）
  • 特定パス専用ルールを記述（*.instructions.md へ）
  • 専門手順・コマンドを詰め込む（SKILL.md へ）
  • 長大なコードスニペット（コンテキスト圧迫）
  • 機密情報・API キーの記述
  • 指示が互いに矛盾する記述
✗ *.agent.md / SKILL.md
  • description が曖昧（「何でもやります」エージェント）
  • ファイル境界を設定しないエージェント（無制限書き込み）
  • SKILL.md で applyTo なしに常時ロード状態にする
  • *.agent.md を .github/agents/ 以外に置く
  • excludeAgent を設定せず指示が混在する
  • agent.md ファイル名に許可外文字（スペース等）を使う

まとめ：各ファイルの役割と設計原則

ファイル配置場所ロード方式設計原則アンチパターン
copilot-instructions.md .github/ 直下（必須） 常時 普遍的なルールのみ。プロジェクト概要・スタック・禁止操作 パス固有ルール・専門手順の混在
*.instructions.md .github/instructions/ applyTo applyTo でパスを絞る。excludeAgent でエージェント除外 applyTo なしの全体適用
*.agent.md .github/agents/ @呼び出し description 明確化・ツール最小化・境界（書き込み先）を明示 曖昧なdescription・書き込み先無制限
SKILL.md .github/skills/*/ 遅延ロード description で「いつ使う/使わない」を明確に記述 description が曖昧・常時ロード設定
*.prompt.md .github/prompts/ /コマンド 繰り返しタスクの再利用化。mode と tools を明示 1回しか使わないタスクを定義
*.chatmode.md .github/chatmodes/ モード切替 特定ペルソナ（DBA・セキュリティ等）のツールセット定義 過剰なツール許可
AGENTS.md プロジェクトルート クロスツール Claude/Gemini/Codex と共通の普遍ルール。ツール固有は別ファイル ツール固有構文の混在
