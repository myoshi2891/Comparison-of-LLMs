.github/copilot-instructions.md 
# Copilot Instructions
# このファイルはCopilotへの全リクエストに自動添付される。
# "2ページ以内・簡潔に"が鉄則（公式推奨）。

## Project Overview
ECサイトのバックエンドAPI（Go + gRPC マイクロサービス）。
プリオーダー・決済・在庫管理の3サービスで構成。

## Repository Structure
- `cmd/`        — サービスエントリーポイント
- `internal/`   — ビジネスロジック（外部公開不可）
- `migrations/` — DBマイグレーション（直接編集禁止）
- `docs/`       — 仕様書群（features/{name}/spec|plan|tasks.md）

## Build & Test
# Coding Agentが実行するコマンドを明示する
- Build:    `go build ./...`
- Test all: `go test ./... -race -timeout 120s`
- Lint:     `golangci-lint run`
- Dev:      `docker compose up -d`

## Code Standards
- Go 1.23。CGO_ENABLED=0
- エラー: `fmt.Errorf("context: %w", err)` 形式
- テスト: table-driven tests 必須
- ORM使用禁止（pgx v5 raw SQLのみ）
- サービス間通信: gRPCのみ（REST禁止）

## PR Instructions
# Coding AgentがPRを作成する際の形式
タイトル: `[feat/fix/refactor] 短い説明（英語）`
必須セクション: Summary / Testing Done / Breaking Changes

## Forbidden
- `migrations/` への直接書き込み（SKILL.md db-migration を使用）
- `.env` の作成・変更
- APIキーのハードコード
- `panic()` の使用（エラーを返す）
⚠️
「2ページ以内」の鉄則
公式ドキュメントは「短く自己完結した記述（short, self-contained statements）にすること」を強調しています。外部ファイルへの参照リクエストや長大な指示は機能しません。詳細なルールは.instructions.md・SKILL.mdへ分離してください（[4]）。
💡
Copilotに自動生成させる
初回PR作成時にCopilotがinstructionsファイル生成リンクをコメントします。また@workspace Generate custom instructions for this repository based on the current codebaseとCopilot Chatで依頼すれば即座に草案が生成されます（[3]）。

04..instructions.md — パス特化型ルール（最重要追加機能）

2025年7月にリリースされたパス特化型カスタム指示。YAMLフロントマターのapplyToプロパティでglob パターンを指定し、対象ファイルを編集するときのみ自動注入されます。テスト・スタイル・インフラ・DBなど領域ごとにルールを分離でき、コンテキスト汚染を防げます（[5]）。

フロントマター仕様

フロントマター全プロパティ
---

                                     # 複数パターンはカンマ区切り

excludeAgent: "code-review"       # 任意: "code-review" or "coding-agent"
                                     # 指定したエージェントには非適用
---

# Spec Generation Prompt

`.specify/memory/constitution.md` を読み、プロジェクトの原則を理解してください。

ユーザーが説明した機能について、以下の構造で仕様書を生成してください：

# Database Migration Skill

## Instructions
1. `migrations/` に `YYYYMMDD_HHMMSS_description.up.sql` を作成
2. ロールバック用 `.down.sql` を必ず同時作成
3. 整合性チェック: `python scripts/run_migration.py --check`
4. 人間レビュー後に適用: `--apply`
5. `features/.../tasks.md` の対象タスクをチェック済みに

## Constraints
- 既存マイグレーションファイルを絶対に編集しない
- `DROP TABLE` は人間確認なしに禁止
- NULL制約後付けはデータ移行計画なしに行わない

13.MCPサポート — .vscode/mcp.json 完全ガイド

エージェントフックはエージェントセッションの主要タイミングでカスタムコマンドを自動実行する機能です。実装前のLint、コミット前のセキュリティスキャン、PR作成前のテスト実行など、チームのポリシーをエージェントワークフローに自動組み込みできます（[21]）。

pre_tool_call フック

ツール呼び出し前に実行。ファイル変更前のバリデーション、特定ファイルへの書き込み禁止チェック、コーディング規約の事前確認に使用。

post_tool_call フック

ツール呼び出し後に実行。変更ファイルに対する自動Lint・フォーマット適用、テスト実行トリガー、変更ログ自動更新に使用。

session_end フック

エージェントセッション終了時に実行。セキュリティスキャン（SAST）の実行、テストカバレッジ確認、PR作成前の最終検証チェックリストに使用。

# .github/agent-hooks/post-edit.yaml
name: "Post-Edit Quality Gate"
trigger: post_tool_call
conditions:
  - tool: edit/editFiles
    file_pattern: "**/*.go"

steps:
  - name: "Lint"
    run: "golangci-lint run ${changed_files}"
  - name: "Test"
    run: "go test ./... -run TestUnit"
    on_failure: "pause_agent"  # 失敗時はエージェントを一時停止

# → テストが失敗するとエージェントが停止してエラーを報告
# → 人間が確認後に「続けて」と指示できる

15.横断ベストプラクティス 12則（2026年3月更新版）

01

copilot-instructions.mdは「2ページ以内」

詳細は.instructions.md・SKILL.mdに分離。長い指示はCopilotが無視する原因になる。Referencesパネルで参照確認必須。

02

AGENTS.mdを「唯一の真実のソース」に

Copilot Coding AgentはAGENTS.mdを自動読み込みするため、copilot-instructions.mdへの明示的インポートは不要。4ツール間でルールを二重管理しない。ツール乗り換え時の書き直しゼロ。

03

applyToで「コンテキストの汚染」を防ぐ

Goファイル編集時にはGoルールのみ、テストファイル編集時にはテストルールのみを注入。全ルールを常時注入しない。

04

Spec Kitで「バイブコーディング」と決別

constitution.md → specify → plan → tasks の順序を厳守。Spec Kitの一貫性チェックでspec/plan/tasksの食い違いを自動検出。

05

.prompt.mdのモデルをタスク特化で選ぶ

設計フェーズにclaude-opus-4-6、実装フェーズにgpt-4.1。プロンプトごとの最適モデル選択はCopilot独自の強み。

06

Coding Agentには「完了条件」を明記

GitHubのIssueテキストは「AIプロンプト」として機能する。受け入れ基準・変更ファイル・テスト要件を明示することがベストプラクティス。

07

.chatmode.mdと.agent.mdを使い分ける

「アーキテクト目線で会話したい」→ chatmode。「ファイルを実際に操作させたい」→ agent。ペルソナか実行委任かで判断。

08

SKILL.mdのdescriptionに境界を書く

「Use when...」「Do NOT use when...」の両方を記述。全4ツール共通規格なのでClaude CodeのSKILL.mdをそのまま移植できる。

09

Referencesパネルで動作確認を習慣化

Copilotのレスポンス下部のReferencesパネルに使用ファイルが表示される。リストにないファイルは無効か構文エラーがある。

10

constitution.mdを「生きた文書」として保守

技術スタック変更・アーキテクチャ決定変更があれば即constitution.mdを更新。コードとconstitutionの乖離がプロジェクト腐敗の根本原因。

11

🆕 MCPサーバーは.vscode/mcp.jsonでプロジェクト管理

APIキーは${input:変数名}でVS Codeシークレットストレージに保管。mcp.jsonをgitにコミットする場合は平文シークレット不可。GitHub MCP ServerはCopilot CLIにビルトイン（設定不要）なので真っ先に活用する。Business/Enterprise管理者はOrgポリシーでドメイン制限が必要（[18]）。

12

🆕 複雑タスクはプランモードを起点にする

5ファイル以上に影響する変更はShift+Tabでプランモードへ切り替え。Copilotの質問に答えて計画を確定してから実装を開始する。内部テストではSWE-benchで成功率15%向上・タスク完了20%増加の実績。エージェントフックとの組み合わせでCI/CDを自動化（[20], [21]）。
