ベストプラクティス完全テンプレート

.github/copilot-instructions.md 
# Copilot Instructions
# このファイルはCopilotへの全リクエストに自動添付される。
# "2ページ以内・簡潔に"が鉄則（公式推奨）。

## Project Overview
ECサイトのバックエンドAPI（Go + gRPC マイクロサービス）。
プリオーダー・決済・在庫管理の3サービスで構成。

## Repository Structure
- `cmd/`        — サービスエントリーポイント
- `internal/`   — ビジネスロジック（外部公開不可）
- `migrations/` — DBマイグレーション（直接編集禁止）
- `docs/`       — 仕様書群（features/{name}/spec|plan|tasks.md）

## Build & Test
# Coding Agentが実行するコマンドを明示する
- Build:    `go build ./...`
- Test all: `go test ./... -race -timeout 120s`
- Lint:     `golangci-lint run`
- Dev:      `docker compose up -d`

## Code Standards
- Go 1.23。CGO_ENABLED=0
- エラー: `fmt.Errorf("context: %w", err)` 形式
- テスト: table-driven tests 必須
- ORM使用禁止（pgx v5 raw SQLのみ）
- サービス間通信: gRPCのみ（REST禁止）

## PR Instructions
# Coding AgentがPRを作成する際の形式
タイトル: `[feat/fix/refactor] 短い説明（英語）`
必須セクション: Summary / Testing Done / Breaking Changes

## Forbidden
- `migrations/` への直接書き込み（SKILL.md db-migration を使用）
- `.env` の作成・変更
- APIキーのハードコード
- `panic()` の使用（エラーを返す）
⚠️
「2ページ以内」の鉄則
公式ドキュメントは「短く自己完結した記述（short, self-contained statements）にすること」を強調しています。外部ファイルへの参照リクエストや長大な指示は機能しません。詳細なルールは.instructions.md・SKILL.mdへ分離してください（[4]）。
💡
Copilotに自動生成させる
初回PR作成時にCopilotがinstructionsファイル生成リンクをコメントします。また@workspace Generate custom instructions for this repository based on the current codebaseとCopilot Chatで依頼すれば即座に草案が生成されます（[3]）。

04..instructions.md — パス特化型ルール（最重要追加機能）

2025年7月にリリースされたパス特化型カスタム指示。YAMLフロントマターのapplyToプロパティでglob パターンを指定し、対象ファイルを編集するときのみ自動注入されます。テスト・スタイル・インフラ・DBなど領域ごとにルールを分離でき、コンテキスト汚染を防げます（[5]）。

📐
.instructions.md
.github/instructions/*.instructions.md — パスマッチで自動注入
applyTo: glob July 2025 GA excludeAgent対応

フロントマター仕様

フロントマター全プロパティ
---
applyTo: "glob-pattern"          # 必須: どのファイルに適用するか
                                     # 複数パターンはカンマ区切り
description: "Brief description"  # 任意: 何のルールかの説明
excludeAgent: "code-review"       # 任意: "code-review" or "coding-agent"
                                     # 指定したエージェントには非適用
---

4つの実践テンプレート（領域別分割パターン）

frontend.instructions.md

applyTo: "src/**/*.tsx, src/**/*.ts"

## Frontend Standards
- コンポーネント: 関数型のみ（クラス禁止）
- スタイル: Tailwind CSS優先。
  CSS Modulesは動的スタイルのみ
- インライン `style={}` 禁止

## Testing Standards
- RTL使用（enzyme系API禁止）
- `data-testid` でセレクタを定義
- 外部依存はすべてモック
- ユーザー視点の振る舞いをテスト
- 実装詳細をテストしない
infra.instructions.md
---
applyTo: "infra/**/*.tf"
description: "Terraform規約"
---

## Terraform Standards
- リソース命名: `{env}-{service}-{type}`
- 全リソースに `Environment` タグ必須
- `terraform fmt` 実行後にコミット
- シークレットは AWS SSM / Vault参照
- ハードコードIP禁止
migrations.instructions.md
---
applyTo: "migrations/**/*.sql"
excludeAgent: "coding-agent"
---

## Migration Standards
- ファイル名: YYYYMMDD_description.sql
- `.up.sql` と `.down.sql` を必ず両方作成
- `DROP TABLE` は人間確認なしに禁止
- NULL制約後付けは移行計画を先に書く
💡
excludeAgent の活用法
excludeAgent: "coding-agent"を設定すると、Copilot Coding AgentにはそのルールファイルがPRとして届かず、Copilot Chatでの手作業支援にのみ適用できます。逆にexcludeAgent: "code-review"で自動コードレビューには除外することもできます（[5]）。

# Spec Generation Prompt

`.specify/memory/constitution.md` を読み、プロジェクトの原則を理解してください。

ユーザーが説明した機能について、以下の構造で仕様書を生成してください：

1. **明確化が必要な点を質問する**（過小仕様を防ぐ）

# Technical Planning Prompt

# Task Breakdown Prompt

`features/{feature-name}/spec.md` と `plan.md` を読んでください。

1. `.specify/templates/tasks-template.md` に従ってタスクリストを作成
2. 各タスクは単独でテスト可能な粒度にする
3. 依存関係を明示する（並列可能なタスクを識別）
4. constitution.md・spec.md・plan.mdとの一貫性チェックを実施

08.SKILL.md — Progressive Disclosure ナレッジ

.github/skills/<name>/SKILL.mdはClaude Code・Antigravity・OpenAI Codexと完全に同一のオープン規格を採用しています（元はAnthropicが考案）。Copilot Coding Agent・VS Code Agent Mode・Copilot CLI（2025年12月18日〜、[15]）で動作し、エージェントが意図を検知したときのみオンデマンドでロードされます（[9]）。個人スキルは~/.copilot/skills/<name>/SKILL.mdに配置するとプロジェクトをまたいで利用できます。

.github/skills/db-migration/SKILL.md — 完全テンプレート 
---
name: db-migration
description: >
  Executes PostgreSQL schema migrations using the project's
  standard migration protocol. Use when the user asks to add
  tables, columns, indexes, or modify the DB schema.
  Do NOT use for seed data or application-level transforms.
# ↑ 意味的トリガー。"Use when / Do NOT use when" を明記
---

# Database Migration Skill

## Instructions
1. `migrations/` に `YYYYMMDD_HHMMSS_description.up.sql` を作成
2. ロールバック用 `.down.sql` を必ず同時作成
3. 整合性チェック: `python scripts/run_migration.py --check`
4. 人間レビュー後に適用: `--apply`
5. `features/.../tasks.md` の対象タスクをチェック済みに

## Constraints
- 既存マイグレーションファイルを絶対に編集しない
- `DROP TABLE` は人間確認なしに禁止
- NULL制約後付けはデータ移行計画なしに行わない
💡
SKILL.mdはClaude Codeとパスを共有できる（コピー不要）
Copilotは.claude/skills/を自動ピックアップするため、Claude Codeで作成したSKILL.mdをコピーなしにそのまま利用できます（[15]）。.github/skills/への配置も引き続き有効。4プラットフォーム間で完全に同一規格なので、ツール切り替え時の書き直しが不要です（[9]）。

09.AGENTS.md — オープン標準統合レイヤー

GitHub CopilotはLinux Foundation傘下のAAIF（Agentic AI Foundation）が管理するAGENTS.mdオープン標準をネイティブサポートしています。プロジェクトルートのAGENTS.mdをcopilot-instructions.mdと並行して自動読み込みします（[3], [16]）。CLAUDE.md・GEMINI.mdもCopilot Coding Agentがネイティブかつ自動で読み込むことが公式確認されています（[16]）。追加設定不要でマルチツール環境に対応します。

Section 13

13.横断ベストプラクティス 10則

01

copilot-instructions.mdは「2ページ以内」

詳細は.instructions.md・SKILL.mdに分離。長い指示はCopilotが無視する原因になる。Referencesパネルで参照確認必須。

02

AGENTS.mdを「唯一の真実のソース」に

Copilot Coding AgentはAGENTS.mdを自動読み込みするため、copilot-instructions.mdへの明示的インポートは不要。4ツール間でルールを二重管理しない。ツール乗り換え時の書き直しゼロ。

03

applyToで「コンテキストの汚染」を防ぐ

Goファイル編集時にはGoルールのみ、テストファイル編集時にはテストルールのみを注入。全ルールを常時注入しない。

04

Spec Kitで「バイブコーディング」と決別

constitution.md → specify → plan → tasks の順序を厳守。Spec Kitの一貫性チェックでspec/plan/tasksの食い違いを自動検出。

05

.prompt.mdのモデルをタスク特化で選ぶ

設計フェーズにclaude-opus-4-6、実装フェーズにgpt-4.1。プロンプトごとの最適モデル選択はCopilot独自の強み。

06

Coding Agentには「完了条件」を明記

GitHubのIssueテキストは「AIプロンプト」として機能する。受け入れ基準・変更ファイル・テスト要件を明示することがベストプラクティス。

07

.chatmode.mdと.agent.mdを使い分ける

「アーキテクト目線で会話したい」→ chatmode。「ファイルを実際に操作させたい」→ agent。ペルソナか実行委任かで判断。

08

SKILL.mdのdescriptionに境界を書く

「Use when...」「Do NOT use when...」の両方を記述。全4ツール共通規格なのでClaude CodeのSKILL.mdをそのまま移植できる。

09

Referencesパネルで動作確認を習慣化

Copilotのレスポンス下部のReferencesパネルに使用ファイルが表示される。リストにないファイルは無効か構文エラーがある。

10

constitution.mdを「生きた文書」として保守

技術スタック変更・アーキテクチャ決定変更があれば即constitution.mdを更新。コードとconstitutionの乖離がプロジェクト腐敗の根本原因。
