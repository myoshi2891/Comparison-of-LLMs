Section 01 — SKILL.md 完全ガイド

01.SKILL.mdとは — 概念・位置づけ・他ファイルとの違い

Agent Skillsは、AIエージェントに専門的な手順・スクリプト・リソースをオンデマンドで提供する再利用可能な知識パッケージです。 SKILL.mdはそのコアファイルで、YAMLフロントマターに「いつ・なぜ使うか」を定義し、本文に「どうやるか」の手順を記述します。 GitHub Copilot Coding Agent・VS Code Agent Mode・Copilot CLI・Claude Code・OpenAI Codexの4ツール以上で共通動作するオープン標準（agentskills.io）です（[A]）。

🎓
SKILL.mdを一言で表すと「再利用可能な手順書パッケージ」
通常のカスタム指示が「常に適用されるルール」であるのに対し、SKILL.mdは「特定のタスクが発生したときだけ読み込まれる専門手順書」です。 コンテキストウィンドウを消費しながら常時注入されるのではなく、エージェントが意図を検知したときに初めてロードされます。 これにより、数十〜数百のスキルをインストールしてもコンテキスト効率を保てます（[1], [A]）。

SKILL.md vs カスタム指示ファイル — 根本的な違い

観点 copilot-instructions.md / .instructions.md SKILL.md
ロードのタイミング 全リクエストに常時注入 関連するときのみオンデマンド読み込み
適した内容 コーディング規約・命名規則・言語設定など常時必要なルール DBマイグレーション・テスト生成・Terraformレビューなど特定タスク手順

02.3段階ローディング（Progressive Disclosure）の仕組み

SKILL.mdの最大の特徴はProgressive Disclosure（段階的開示）という3レベルの読み込み機構です。 エージェントはスキルのすべてを最初から読み込むのではなく、必要に応じて段階的にコンテキストに追加していきます。 これにより数十・数百のスキルを登録しても、コンテキストウィンドウを圧迫しません（[1], [B]）。

Level 1 — Discovery
🔍
発見フェーズ
エージェントはYAMLフロントマターのnameとdescriptionだけを読む。 ユーザーの質問とdescriptionを照合し、このスキルが関連するかを判断する。
約100トークン消費
Level 2 — Instructions
📋
指示ロードフェーズ
スキルが関連すると判断されたとき、SKILL.md本文全体をコンテキストに注入。 手順・ガイドライン・制約が全部読み込まれ、エージェントが従う。
本文全体（〜5,000トークン推奨）
Level 3 — Resources
📦
リソースアクセスフェーズ
指示の中でスクリプト・テンプレート・参照ドキュメントを参照した場合に限り、 そのファイルをスキルディレクトリから読み込む。
参照時のみ追加取得

05.ステップバイステップ — SKILL.md作成ガイド

01
スキルのスコープを決める — 「1スキル = 1タスク」の原則

Section 10

10.SKILL.md専用ベストプラクティス 10則

01

descriptionは「Use for: キーワード列挙」を必ず含める

自動発動の唯一の判断材料。日本語・英語の両方を含めると精度が上がる。「Use for: migration, マイグレーション, ALTER TABLE」のように具体的に。曖昧な表現はLevel 1で止まる（[C]）。

02

nameとディレクトリ名を完全一致させる

これが最も多い初心者ミス。.github/skills/db-migration/ならname: db-migration。大文字・アンダースコア・スペースは使用不可。一致しないとスキルが一切読み込まれない（[B]）。

03

本文は500行・5,000トークン以内に収める

超えると発見・ロードが不安定になる。詳細な資料はreferences/に移してリンク参照にする。Progressive Disclosureを活かし、Level 2の手順はシンプルに、詳細はLevel 3リソースへ（[F]）。

04

1スキル = 1タスクの単一責任原則

「DBマイグレーション + APIエンドポイント実装 + テスト」を1スキルにしない。スコープが広すぎると自動発動が不正確になる。タスクが複数なら複数スキルに分割（[3]）。

05

metadataが最後フィールドにならないよう注意

Copilot CLIのバグ（2026年1月報告）。metadataが最終フィールドだとCLIでスキルが発見されない。metadataの後にlicense: MITなど1フィールド追加して回避（[E]）。

06

スクリプトは scripts/ に、テンプレートは assets/ に

公式仕様（agentskills.io）で定義された3つのサブディレクトリを活用。scripts/（実行可能）・references/（ドキュメント）・assets/（テンプレート）で役割を分ける（[F]）。

07

デリケートな操作は disable-model-invocation: true に

本番DBの直接操作・本番デプロイ・Secrets変更など、誤トリガーが危険な操作は自動発動を無効化。手動で/スキル名として呼び出した場合のみ動作させる（[B]）。

08

スキルはgit管理してチームで共有・バージョン管理する

SKILL.mdはコードと同様にgit管理。metadataにversionを記録し、変更履歴を追う。チームで改善サイクルを回す。コミュニティスキル（awesome-copilot）を活用する前に必ずレビューする（[9]）。

09

skills-ref バリデーターでCI/CDに品質チェックを組み込む

公式バリデーターuvx skills-ref validateをCIパイプラインに追加。PRでSKILL.mdが更新されたら自動チェック。仕様違反・トークン超過・name不一致を事前に検出（[F]）。

10

Claude CodeのSKILL.mdをCopilotで再利用する

Copilotは.claude/skills/を自動ピックアップ。Claude Codeで作成したSKILL.mdをコピー不要でそのまま利用できる。4ツール共通のオープン標準なのでツール切り替え時の書き直しが不要（[A]）。

13.横断ベストプラクティス 10則

01

copilot-instructions.mdは「2ページ以内」

詳細は.instructions.md・SKILL.mdに分離。長い指示はCopilotが無視する原因になる。Referencesパネルで参照確認必須。

02

AGENTS.mdを「唯一の真実のソース」に

Copilot Coding AgentはAGENTS.mdを自動読み込みするため、copilot-instructions.mdへの明示的インポートは不要。4ツール間でルールを二重管理しない。ツール乗り換え時の書き直しゼロ。

03

applyToで「コンテキストの汚染」を防ぐ

Goファイル編集時にはGoルールのみ、テストファイル編集時にはテストルールのみを注入。全ルールを常時注入しない。

04

Spec Kitで「バイブコーディング」と決別

constitution.md → specify → plan → tasks の順序を厳守。Spec Kitの一貫性チェックでspec/plan/tasksの食い違いを自動検出。

05

.prompt.mdのモデルをタスク特化で選ぶ

設計フェーズにclaude-opus-4-6、実装フェーズにgpt-4.1。プロンプトごとの最適モデル選択はCopilot独自の強み。

06

Coding Agentには「完了条件」を明記

GitHubのIssueテキストは「AIプロンプト」として機能する。受け入れ基準・変更ファイル・テスト要件を明示することがベストプラクティス。

07

.chatmode.mdと.agent.mdを使い分ける

「アーキテクト目線で会話したい」→ chatmode。「ファイルを実際に操作させたい」→ agent。ペルソナか実行委任かで判断。

08

SKILL.mdのdescriptionに境界を書く

「Use when...」「Do NOT use when...」の両方を記述。全4ツール共通規格なのでClaude CodeのSKILL.mdをそのまま移植できる。

09

Referencesパネルで動作確認を習慣化

Copilotのレスポンス下部のReferencesパネルに使用ファイルが表示される。リストにないファイルは無効か構文エラーがある。

10

constitution.mdを「生きた文書」として保守

技術スタック変更・アーキテクチャ決定変更があれば即constitution.mdを更新。コードとconstitutionの乖離がプロジェクト腐敗の根本原因。
