Section 01

01.AIエージェントのテストとは何か

通常のソフトウェアは「同じ入力 → 必ず同じ出力」という決定論的な性質を持ちます。一方、LLMベースのAIエージェントは確率的であり、同じ質問でも毎回微妙に異なる回答を生成します。このため、従来の assert output == expected アプローチが使えません。

通常ソフトウェア vs AIエージェントの違い

比較軸 通常のソフトウェア AIエージェント
出力の性質 決定論的（毎回同じ） 確率的（毎回変わりうる）
テスト手法 assert output == expected スコアリング・LLM-as-Judge
失敗基準 1件でも FAIL スコアが閾値を下回る
テストデータ 単体テストケース 評価セット（Eval Set）
検証対象 戻り値・副作用 ツール呼び出し・応答品質・安全性
実行コスト ミリ秒〜秒単位・無料 秒〜分単位・APIコスト発生

AIエージェントのテスト3観点

テスト観点の全体像
💡
例え話：自動車のクラッシュテストでは、ダミー人形・センサー・高速カメラ・制御された環境がすべて揃って初めて「安全性スコア」が出ます。AIエージェントも同様に、テスト環境・評価データ・採点ロジックがセットになったテストハーネスがなければ品質を数値化できません。

02.ハーネスアーキテクチャ全体像

Google推奨エージェント評価ハーネスの構成

3種類のテストレベル

Googleはエージェントテストを Small / Medium / Large で分類しています（Software Engineering at Google の分類をAIに適用）。

レベル 名称 LLM 外部API 目的 実行時間
Small ユニット評価 フェイクLLM モック ツール呼び出しロジックのテスト 〜5秒
Medium 統合評価 本物LLM（安価なモデル） スタブ 1〜3ターンの会話品質評価 〜60秒
Large E2E評価 本物LLM（最高精度モデル） 本物API 複数ステップの自律タスク評価 〜300秒
🔬 Small テストを最大化

フェイクLLMを使い APIコスト 0円・高速で設計ミスを早期発見。全テストの70%以上をSmallに。

🔗 Medium テストでバランス

安価な gemini-3-flash-preview を使い実際の会話品質を確認。コスト90%削減。

🚀 Large テストは最小限に

最高精度モデルのE2E評価は時間・コストが高い。リリース前の最終確認に限定。

04.テストダブル — フェイクLLM・フェイクツールの実装

なぜテストダブルが必要か

本物APIを毎回呼ぶ問題

フェイクLLM（Fake LLM）の実装 — Small テスト用

tests/doubles/fake_llm.py 
from google.adk.models import BaseLlm, LlmResponse

class FakeLlm(BaseLlm):
    """
    テスト用フェイクLLM。
    事前に定義したシナリオ（query → response）に従って
    決定論的な応答を返す。APIコスト・ネットワーク接続なし。
    """

    def __init__(self, scenarios: dict[str, LlmResponse]):
        self._scenarios = scenarios
        self._call_history: list[str] = []  # 呼び出し記録（検証用）

    async def generate_content_async(self, llm_request, **kwargs) -> LlmResponse:
        query = llm_request.contents[-1].parts[0].text
        self._call_history.append(query)

        for pattern, response in self._scenarios.items():
            if pattern in query:
                return response

        return LlmResponse(text="デフォルト応答です。")

    @property
    def call_count(self) -> int:
        """何回呼ばれたかを返す（アサーション用）"""
        return len(self._call_history)

フェイクツール（Fake Tool）の実装

tests/doubles/fake_tools.py 
from google.adk.tools import FunctionTool

def make_fake_deploy_tool(should_succeed: bool = True):
    """デプロイツールの替え玉。本番環境には触れず成功/失敗を制御できる。"""
    call_log = []  # 呼び出し記録

    def fake_deploy(environment: str) -> dict:
        call_log.append({"environment": environment})
        if should_succeed:
            return {"status": "success", "url": f"https://{environment}.example.com"}
        return {"status": "error", "message": "Deployment failed"}

    fake_deploy.call_log = call_log  # テストから参照できるように付与
    return FunctionTool(fake_deploy)

テストダブルの種類と使い分け

種類 特徴 AIエージェントでの使用場面
Fake LLM 決定論的な応答を返す軽量実装 ツール呼び出しロジックのSmallテスト
Stub Tool 固定値を返すだけのツール 外部APIが不安定な統合テスト
Mock Tool 呼び出しを記録して検証するツール 「このツールが正確に呼ばれたか」の検証
Spy 本物のツールを使いつつ呼び出しを記録 本番に近い環境での動作監視
Recording/Replay 実際のAPI応答を録画・再生 高コストAPIのコスト削減

05.ADK Eval — Google公式評価フレームワーク

ADK Eval（Agent Development Kit Evaluation）はGoogleが提供するAIエージェント専用の評価フレームワークです。ツール呼び出しの正確性・応答品質・会話の流れを自動で評価します。

ADK Eval の採点器（Scorers）一覧

採点器 測定内容 スコア範囲 推奨閾値
tool_call_quality ツール名・引数の正確性 0.0〜1.0 ≥ 0.8
response_quality 応答の正確性・有用性 0.0〜1.0 ≥ 0.7
trajectory_accuracy 会話の経路が期待通りか 0.0〜1.0 ≥ 0.75
safety_score 有害コンテンツがないか 0.0〜1.0 ≥ 0.95

セットアップと実行

shell — ADK Eval のインストールと実行 
# インストール
pip install google-adk[eval]

# コマンドラインから評価を実行
adk eval \
  --agent_module agents.my_agent \
  --eval_set_file evals/eval_set.json \
  --output_dir results/

Python API を使った評価の実装

evals/run_eval.py 
import asyncio
from google.adk.evaluation import AgentEvaluator
from agents.my_agent import root_agent

async def run_evaluation():
    evaluator = AgentEvaluator(
        agent=root_agent,
        eval_set_file="evals/eval_set.json",
        scorers=["tool_call_quality", "response_quality"],
        model="gemini-3-flash-preview",  # コスト最適化モデル
    )

    results = await evaluator.evaluate()

    print(f"全体スコア: {results.overall_score:.2f}")
    print(f"ツール呼び出し精度: {results.scores['tool_call_quality']:.2f}")

    # CI での合否判定
    assert results.scores["tool_call_quality"] >= 0.8, \
        f"ツール呼び出し精度が基準値を下回りました: {results.scores['tool_call_quality']:.2f}"

if __name__ == "__main__":
    asyncio.run(run_evaluation())

06.LLM-as-Judge パターン

LLM-as-Judgeとは、AIエージェントの出力品質を別のLLMが採点するパターンです。AIの応答は「意味的に正しいが文字列が一致しない」ことが多いため、通常の assert では検証できません。

💡
例え話：小学生の作文を採点するとき「正解の文字列と完全一致するか」ではなく「内容の正確性・文章の流れ・誤字脱字」を先生が総合評価しますよね。LLM-as-Judgeも同じ発想で、別のLLMが採点者（Judge）として機能します。

Judge の実装例

evals/judges/response_judge.py 
import json
import anthropic

def judge_response(
    question: str,
    actual_response: str,
    reference_answer: str,
    model: str = "claude-sonnet-4-6"  # 被評価モデルとは別のモデルを使う
) -> dict:
    client = anthropic.Anthropic()

    judge_prompt = f"""あなたはAIエージェントの応答品質を評価する専門家です。
以下の質問に対するエージェントの応答を0.0〜1.0で採点してください。

## 質問
{question}

## 参考となる期待応答
{reference_answer}

## エージェントの実際の応答
{actual_response}

## 採点基準
- 1.0: 完全に正確で有用
- 0.8: 概ね正確。軽微な情報の欠落あり
- 0.6: 部分的に正確。重要な情報の欠落あり
- 0.0: 完全に誤り、または有害なコンテンツを含む

必ずJSON形式のみで回答: {{"score": X.X, "reason": "理由", "improvement": "改善点"}}
"""

    message = client.messages.create(
        model=model,
        max_tokens=300,
        messages=[{"role": "user", "content": judge_prompt}]
    )
    return json.loads(message.content[0].text)

LLM-as-Judge の3つのパターン

LLM-as-Judge のパターン分類
⚠️
重要：JudgeモデルはテストされるAIエージェントと異なるモデルを使うこと。同じモデルが自分を採点すると採点が甘くなります。GeminiエージェントにはClaude、ClaudeエージェントにはGeminiをJudgeとして使うのがベストプラクティスです。

07.マルチエージェント評価ハーネス

複数のエージェントが連携するシステム（ADKの SequentialAgent / ParallelAgent）では、単一エージェントの評価に加えてエージェント間の連携品質も評価する必要があります。

単一 vs マルチエージェントの評価観点の違い

評価観点 単一エージェント マルチエージェント
ツール呼び出し 単体の正確性 適切なエージェントへの委譲
応答品質 最終応答 各エージェントの中間出力
実行経路 1つのトレース 複数エージェントのトレース連鎖
並行性 非該当 ParallelAgentの出力競合なし
コスト 1エージェント分 エージェント数×コスト

マルチエージェント向けハーネス設計

マルチエージェント評価ハーネスのアーキテクチャ
🚨
ParallelAgent の評価における落とし穴：ParallelAgentの各サブエージェントが同じ output_key に書き込むとデータが上書きされます（レースコンディション）。評価ケースでは "expected_output_keys": ["frontend_result", "backend_result"] のように各エージェントが異なるキーに書き込むことを検証してください。

08.フレイキー（不安定）評価への対処

「今日は通ったが明日は落ちる」という不安定な評価はAIエージェント評価特有の問題です。

原因 具体例 対策
LLMの確率的出力 同じ質問でも違うツールを選ぶ スコア閾値を使い完全一致を避ける
モデルのバージョン更新 Gemini 3のアップデートで挙動変化 モデルバージョンをピン留めする
APIレート制限 高頻度実行でレート制限エラー リトライ＋指数バックオフを実装
評価LLMの変動 Judgeとして使うLLMも確率的 複数回評価して平均スコアを使う
外部APIの状態変化 テスト環境のDBデータが変わる ハーミティック環境でDBをリセット

スコア安定化のための多数決評価

evals/stable_evaluator.py 
from statistics import mean, stdev

async def stable_evaluate(
    evaluator,
    query: str,
    n_trials: int = 5,  # 5回評価して平均を使う
    threshold: float = 0.75
) -> dict:
    """
    同じクエリをn_trials回評価し、平均・標準偏差でスコアを安定化する。
    LLMの確率的な出力によるフレイキーを緩和するGoogle推奨の手法。
    """
    scores = []
    for i in range(n_trials):
        result = await evaluator.evaluate_single(query)
        scores.append(result.score)

    avg_score = mean(scores)
    score_std = stdev(scores) if len(scores) > 1 else 0.0

    return {
        "average_score": avg_score,
        "std_deviation": score_std,
        "is_stable": score_std < 0.1,  # 標準偏差0.1未満を「安定」と判定
        "passes_threshold": avg_score >= threshold,
        "all_scores": scores
    }

name: AI Agent Evaluation Pipeline
on:
  push:
    branches: [main, develop]
  pull_request:

  GOOGLE_API_KEY: ${{ secrets.GOOGLE_API_KEY }}

10.ベストプラクティス 10則 & チェックリスト

01

Small テストを最大化する

フェイクLLMを使いユニット評価を全体の70%以上に。高速・低コストで設計ミスを早期発見。

02

評価データはコードと分離する

JSON/YAMLの評価セットをGit管理。評価観点の変更にコード変更不要。

03

完全一致より閾値スコアを使う

LLMの確率的な出力に対応。score >= 0.8 のような基準を設定。

04

ハーミティック環境を維持する

各テストでDBリセット・フェイクAPI使用。テスト間の干渉・環境依存を排除。

05

安全性テストは必ず別途実施

有害コンテンツ・プロンプトインジェクションを専用ケースで検証。閾値 ≥ 0.95 を守る。

06

Judgeモデルは被評価モデルと変える

Geminiエージェントの評価にClaudeを使うなど。同じモデルが自分を採点すると甘くなる。

07

フレイキー評価をその日に直す

検出したら隔離（Quarantine）→1営業日以内に対処。放置すると「赤を無視する文化」が生まれる。

08

モデルバージョンをピン留めする

gemini-3-flash-preview-20261001 のようにバージョンを固定。モデル更新で評価結果が変わるのを防ぐ。

09

評価ケースに「なぜ」を書く

"name": "malicious_sql_injection_resistance" のように命名。将来のメンテナーが意図を理解できる。

10

評価スコアの推移をグラフ化する

Allure Report / Grafana でスコア時系列を可視化。リグレッションをリリース前に検出。
