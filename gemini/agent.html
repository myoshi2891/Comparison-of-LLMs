GEMINI.md / AGENT.md — コンテキストファイルの設計

ℹ️
GEMINI.md vs AGENT.md — ツール別対応表 Gemini CLI は GEMINI.md がデフォルト。Gemini Code Assist (VS Code / Cloud) は GEMINI.md か AGENT.md を使用。Android Studio は AGENTS.md を採用。すべて settings.json の context.fileName でカスタマイズ可能です。
📋 GEMINI.md に書くべきこと / 書かないべきこと
✅ 書くべき内容
  • プロジェクト概要・目的（2〜4文）
  • 技術スタック・主要ライブラリのバージョン
  • サブエージェント委譲ルール
  • ビルド・テスト・デプロイコマンド
  • 禁止操作の明示（DB reset 等）
  • @./subdir/guide.md でのモジュール分割参照
  • コーディング規約（コンパクトに）
  • 重要ドキュメントへのパス参照
✗ 書かないべき内容
  • 長大なコードスニペット（トークン浪費）
  • 過去の変更履歴・修正ログ
  • ツール固有の設定（settings.json に分離）
  • 機密情報・API キー（.geminiignore で除外）
  • すべての規約を1ファイルに詰め込む（@import で分割）
  • 否定形のみの指示（代替案をセットで示す）
📄 実践的な GEMINI.md テンプレート（サブエージェント対応版）
GEMINI.md (project root)Markdown
# PROJECT: my-saas-app ## Overview Next.js 15 + Supabase + Stripe のマルチテナント SaaS。 本番: Vercel Edge Runtime / DB: Supabase (PostgreSQL + pgvector)。 ## Tech Stack - Frontend: Next.js 15 App Router, TypeScript 5.x, Tailwind CSS - Backend: Supabase Edge Functions (Deno 2.x) - Auth: Supabase Auth + Row Level Security - Payment: Stripe Checkout / Billing Portal / Webhooks - AI: Gemini 2.5 Flash (via ADK), pgvector for embeddings ## Build & Test - Build: `pnpm build` - Test: `pnpm test` (Vitest + Testing Library) - ADK dev: `adk web` (Agent Dev UI on :8000) - Lint: `pnpm lint` — ESLint/Prettier に全面委任すること - DB types: `pnpm supabase gen types` ## Sub-Agent Routing Parallel dispatch 条件（すべて満たす場合のみ）: - タスクが 3件以上かつ互いに独立 - 共有ファイル・共有状態なし - ドメイン境界が明確 (frontend / backend / db / ai) - ParallelAgent で書く場合: 各エージェントが異なるキーに書き込むこと Sequential dispatch（いずれかの条件で）: - タスクに依存関係がある（B に A の出力が必要） - 共有ファイルに触る処理がある ## Domain Agents (ADK) - orchestrator → ルーティングのみ、実装しない - frontend-agent → app/, components/, styles/ のみ - backend-agent → supabase/functions/, lib/server/ のみ - ai-agent → lib/ai/, embeddings/, vector queries のみ - db-agent → supabase/migrations/, schema のみ ## Forbidden Operations - `supabase db reset` は絶対に実行しない（本番データ消去） - `.env.production` の読み書き禁止 - `--force` フラグは使わない、代わりに `--dry-run` で確認 ## @-imports (modular context) @./docs/architecture.md @./src/frontend/GEMINI.md @./src/backend/GEMINI.md
💡
@-import 構文でファイルを分割する @./path/to/file.md 構文で他のMarkdownファイルをインポートできます（.md ファイルのみ対応）。大きなプロジェクトでは「ルートは薄く、ドメイン別に分割」が鉄則です。CLIが正規表現でインライン展開するため、追加トークンコストなしで利用できます。

ADK agent.py — サブエージェント定義のベストプラクティス

Google ADK (Agent Development Kit) では、サブエージェントの「system prompt」は instruction フィールド に記述します。このフィールドが Claude の .claude/agents/*.md に相当します。description フィールドが AutoFlow ルーティングの判断基準になるため、特に重要です。

⚠️
description フィールドが「APIドキュメント」 LLM ルーティング（AutoFlow）では、description の内容だけを見て「どのサブエージェントに委譲するか」を判断します。曖昧な description は誤ルーティングの原因になります。「いつ・何のために呼ぶか」を具体的に記述してください。
agents/orchestrator/agent.pyPython / ADK
# agents/orchestrator/agent.py — ルートエージェント（コーディネーター） from google.adk.agents import LlmAgent # サブエージェントをインポート（先に定義が必要） from agents.researcher.agent import researcher_agent from agents.implementer.agent import implementer_agent from agents.reviewer.agent import reviewer_agent root_agent = LlmAgent( name="orchestrator", model="gemini-2.5-flash", description="ユーザーの要求を分析し、適切なサブエージェントに委譲するコーディネーター。", instruction=""" あなたは開発チームのオーケストレーターです。 ユーザーの要求を分析し、以下のルールで適切なエージェントに委譲してください。 ## 委譲ルール - 調査・情報収集 → researcher_agent - コード実装 → implementer_agent - コードレビュー → reviewer_agent - 複合タスク → SequentialAgent で順番に処理 ## 禁止事項 - 自身でコードを書かない（必ずサブエージェントに委譲） - 曖昧なタスクはそのまま渡さず、明確化してから委譲 """, sub_agents=[ researcher_agent, # 調査・情報収集 implementer_agent, # コード実装 reviewer_agent, # コードレビュー (Read-only) ], )
agents/reviewer/agent.pyPython / ADK
# agents/reviewer/agent.py — 特化型サブエージェント（コードレビュー専用） from google.adk.agents import LlmAgent from google.adk.tools import google_search reviewer_agent = LlmAgent( name="reviewer", model="gemini-2.5-flash", # コスト最適化 description=""" コードレビューを実施するエージェント。 以下の場合に呼び出す: - Pull Request 作成後 - コード変更が完了した後 - セキュリティ・パフォーマンス確認が必要な場合 Read-only: コードの変更は行わず、レポートのみ出力する。 """, instruction=""" あなたはシニアエンジニアのコードレビュアーです。 提供されたコードを以下の観点でレビューしてください。 ## Review Checklist ### 🔴 Security - SQLインジェクション / XSS リスク - 認証・認可の適切な実装 - 秘密情報のハードコードがないか ### 🟡 Performance - N+1 クエリの有無 - 不要な再レンダリング (React) - 重い処理の非同期化 ### 🟢 Code Quality - TypeScript 型安全性 - エラーハンドリングの網羅 - テストカバレッジ ## 出力フォーマット ``` ## Code Review Report ### 🔴 Critical (要対応) ### 🟡 Warning (推奨対応) ### 🟢 Good (良い点) ``` """, tools=[], # ツールなし: Read-only のレビューのみ output_key="review_result", # 共有状態への書き込みキー )
🔀 ADK ワークフローエージェントの選択基準
⛓️
SequentialAgent
サブエージェントを順番に実行。前のエージェントの出力を次が受け取る。依存関係がある場合に使用。
例: spec → design → implement → review
⚡
ParallelAgent
すべてのサブエージェントを並列実行。共有状態に書くため必ず異なるキーを使うこと（レースコンディション防止）。
例: frontend || backend || db（独立タスク）
🔁
LoopAgent
条件を満たすまでサブエージェントを繰り返し実行。反復精緻化・ポーリング・リトライに最適。
例: draft → critique → refine（品質基準達成まで）
🚨
ParallelAgent の罠：共有状態のレースコンディション ParallelAgent の子エージェントは 同じ session state を共有します。複数のエージェントが同じキーに書き込むと値が上書きされます。各エージェントには必ず output_key="unique_key_name" のように一意なキーを設定してください。

絶対に避けるべき Anti-Patterns

✗ GEMINI.md の Anti-Patterns
  • コードスタイル規約をすべて1ファイルに詰める（@import で分割）
  • 長大なコードスニペットを埋め込む（トークン浪費）
  • 機密情報・APIキーを記述（.geminiignore で除外）
  • 否定形のみの指示（代替案をセットで示す）
  • 修正履歴・変更ログを蓄積
  • ツール固有の設定を混在させる
✗ ADK agent.py の Anti-Patterns
  • description が曖昧（「なんでもやる」エージェント）
  • ParallelAgent で同じ output_key を使う（上書き発生）
  • LoopAgent の終了条件を設定しない（無限ループ）
  • 全エージェントに gemini-2.5-pro を使う（コスト爆発）
  • ルートエージェントが直接コードを書く（委譲しない）
  • 10件以上のサブエージェントを ParallelAgent で同時起動

サブエージェント vs マルチエージェント — ADK × A2A × MCP の3層構造

🌐 ADK Python v1.0 GA (2025年5月 Google I/O) · A2A Protocol (Linux Foundation 移管済み)

Google が推奨する 3層アーキテクチャ

Google のマルチエージェント設計は ADK（エージェント内部ロジック）・A2A プロトコル（エージェント間通信）・MCP（外部ツール・データ接続） の3層で構成されます。 A2A は Atlassian・SAP・Salesforce・ServiceNow など 50 以上のパートナーが対応するオープンスタンダードです。 ADK の RemoteA2aAgent を使えば、リモートエージェントへの接続がローカルのツール呼び出しと同じ感覚で実装できます。

ADK
🧠 エージェント内部ロジック
agent.py の instruction / tools / sub_agents / output_key を定義。SequentialAgent・ParallelAgent・LoopAgent でワークフローを制御。
agents/*/agent.py agents/*/GEMINI.md
A2A
🔗 エージェント間通信プロトコル
Agent Card（agent.json）で能力を公開。RemoteA2aAgent でリモートエージェントをローカルツールとして利用。HTTPS + JSON-RPC 2.0 で通信。フレームワーク・ベンダーを問わず相互接続可能。
agents/{name}/agent.json /.well-known/agent.json（自動生成）
MCP
🔧 外部ツール・データ接続
エージェントが外部 API・DB・ファイルシステムに接続するための標準プロトコル。ADK は MCPToolset で透過的に統合。A2A と補完関係にある（MCP = ツール接続、A2A = エージェント接続）。
settings.json → mcpServers
🔵 ローカル サブエージェント（ADK 内部）
  • 同一プロセス内で動作（低レイテンシ）
  • 親エージェントのセッション状態を共有
  • sub_agents パラメータで静的に定義
  • SequentialAgent / ParallelAgent / LoopAgent で制御
  • output_key で結果を共有 state に書き込む
  • チームが同じコードベースを管理する場合に適合
  • 適用: 同一サービス内の処理分業・低レイテンシ重視
🟢 リモート マルチエージェント（A2A 経由）
  • 異なるサービス・マシン上で動作（高スケーラビリティ）
  • 独立したセッション・コンテキストを保持
  • RemoteA2aAgent で動的ディスカバリーも可能
  • Agent Card で能力を公開・JSON-RPC で通信
  • フレームワーク（ADK / LangGraph / CrewAI）を問わず相互接続
  • to_a2a() で既存 ADK エージェントを即座に A2A 公開
  • 適用: クロスチーム・エンタープライズ・ベンダー横断ワークフロー

A2A の核心：agent.json（Agent Card）— リモートエージェントの「能力書」

Agent Card は A2A マルチエージェント設計で最も重要なファイルです。 他のエージェントがこのエージェントを発見（Discovery）→ 理解（Capability）→ 接続（Auth）するための唯一の情報源です。 ADK の to_a2a(agent) を使えば agent.py の内容から自動生成されますが、 adk api_server --a2a で複数エージェントを管理する場合は手動で作成して品質を担保します。

Agent Card の description と skills フィールドが Orchestrator のルーティング判断基準となるため、 ADK の agent.py における description と同様に「いつ使う・いつ使わない」を明記することが最重要です。

ℹ️
自動公開エンドポイント ADK の A2AServer または adk api_server --a2a を起動すると、/.well-known/agent.json エンドポイントで Agent Card が自動公開されます（Swagger の AI 版）。クライアントエージェントはこの URL にアクセスして能力をディスカバリーします。
agents/code-review/agent.json — Agent Card ベストプラクティスJSON
{ "name": "code-review-agent", "version": "2.1.0", // ── 【最重要】Orchestrator がルーティング判断に使うフィールド ────────── "description": "コードレビューを実施する専門エージェント。 【呼び出す場合】: - PR 作成前のセキュリティ・品質・パフォーマンスチェック - TypeScript/Python/Go コードの静的解析レポートが必要な場合 - コード品質スコア（0-100）と改善提案が必要な場合 【呼び出さない場合】: - コードの実装・修正 → code-implementer-agent を使う - テスト生成 → test-generator-agent を使う - ドキュメント作成 → doc-writer-agent を使う", "url": "https://code-review.internal.example.com", // ── スキル定義（Orchestrator が能力を詳細に理解するために使用）──────── "skills": [ { "id": "security-review", "name": "Security Review", "description": "SQLインジェクション・XSS・SSRF・認証バイパスを自動検出する", "tags": ["security", "vulnerability", "owasp"], "examples": ["このPRにSQLインジェクションのリスクはありますか？"] }, { "id": "performance-review", "name": "Performance Review", "description": "N+1クエリ・不要な再レンダリング・O(n²)アルゴリズムを検出する", "tags": ["performance", "optimization", "complexity"] } ], // ── 認証（A2A は OpenAPI 互換の securitySchemes を採用）─────────────── "securitySchemes": { "bearerAuth": { "type": "http", "scheme": "bearer", "bearerFormat": "JWT" } }, "security": [{ "bearerAuth": [] }], // ── 入出力形式 ───────────────────────────────────────────────────────── "defaultInputModes": ["text/plain", "application/json"], "defaultOutputModes": ["text/plain", "application/json"], "provider": { "organization": "Platform Engineering Team", "url": "https://internal.example.com/agents" } }
✅ agent.json ベストプラクティス
  • description に「いつ使う・いつ使わない」を両方記述
  • skills は機能単位で細かく分割（タグ・examples 付き）
  • securitySchemes を必ず設定（認証なし公開は危険）
  • version を semantic versioning で管理（破壊的変更 = メジャーバージョンアップ）
  • examples フィールドで Orchestrator の理解を補助
  • provider に連絡先・組織情報を明記（オーナー不明問題を防止）
✗ agent.json Anti-Patterns
  • description が「なんでもやります」（誤ルーティングの最大要因）
  • securitySchemes なしで公開（セキュリティリスク）
  • skills を定義しない（能力発見できず Orchestrator から使われない）
  • url が内部 IP / localhost のまま本番公開
  • version を更新せず API 変更（クライアントが無警告で壊れる）
  • description が agent.py の instruction と乖離している

マルチエージェント向け GEMINI.md — Orchestrator の「作戦指令書」

✅
GEMINI.md はマルチエージェント全体の品質ゲートウェイ Orchestrator は GEMINI.md を読み込んで「どのエージェントをいつ呼ぶか」を判断します。 サブエージェントには独立した agent.py instruction がありますが、Orchestrator のルーティング精度は GEMINI.md の記述品質で決まります。 A2A リモートエージェントのエンドポイント URL・ファイル所有権・フォールバック戦略を必ず記載します。
GEMINI.md — マルチエージェント向け追記テンプレート（rootに追加）Markdown
# PROJECT: enterprise-dev-platform ## Stack / Overview ... （既存セクションはそのまま） ## ──────────────────────────────────────────────────────────── ## Multi-Agent Configuration（マルチエージェント設定） ## ──────────────────────────────────────────────────────────── ### ローカル サブエージェント（同一プロセス・ADK sub_agents） 同一サービス内・低レイテンシが必要な場合に使用: | Agent name | 役割 | 書き込み可能パス | |---|---|---| | spec-writer | 新機能の仕様書作成 | docs/specs/ | | code-implementer | コード実装 | src/, tests/ | | test-generator | テスト生成 | tests/ | ファイル所有権ルール: 担当外のパスへの書き込みは禁止。 同一ファイルへの複数エージェント同時書き込みは禁止。 ### リモート エージェント（A2A Protocol 経由） 別チーム・別フレームワーク・専門性が必要な場合に使用: | Agent name | Agent Card URL | 備考 | |---|---|---| | code-review-agent | https://review.internal.example.com/.well-known/agent.json | セキュリティ・品質レビュー | | security-scanner | https://security.internal.example.com/.well-known/agent.json | 本番デプロイ前必須 | | doc-generator | https://docs.internal.example.com/.well-known/agent.json | API ドキュメント生成 | ### A2A 通信ルール - リモートエージェント呼び出し前に Agent Card の skills を確認して適切なエージェントを選択 - タイムアウト（> 30秒）→ ローカルフォールバックを試みる - 認証エラー → リトライせず即座にユーザーへ報告 - リモートエージェントが「Read-only」の場合、ファイル変更は要求しない ### Quality Gates（全エージェント共通） - コード変更後: `pytest tests/` と `ruff check .` を必ず実行 - 本番デプロイ前: security-scanner エージェントを**必ず**呼ぶ - DB マイグレーション: db-agent 経由でのみ実行（直接 SQL は禁止） ### @import で分割管理（GEMINI.md が肥大化した場合） # 各エージェントのルールが多い場合は以下のように分割 @import ./agents/orchestrator/GEMINI.md # Orchestratorルーティングルール @import ./agents/implementer/GEMINI.md # 実装エージェント固有ルール @import ./docs/agent-endpoints.md # リモートエンドポイント一覧
⚠️
サービス固有 GEMINI.md の配置戦略（Auto-scan 活用） マルチエージェントシステムではサービス数が増えるため GEMINI.md が肥大化しやすい。 agents/code-review/GEMINI.md（レビュー固有ルール）のようにエージェントディレクトリ内に配置し、 そのエージェントが作業するときのみ Auto-scan でロードされる構成を推奨。 ルート GEMINI.md からは @import で共通部分のみを参照させる。

マルチエージェント agent.py — Orchestrator + RemoteA2aAgent 実装パターン

A2A マルチエージェントでは Orchestrator が RemoteA2aAgent を「ローカルのツール」として利用します。 RemoteA2aAgent は Agent Card URL を受け取り、ネットワーク通信・認証・データ形式変換をすべて隠蔽するため、 開発者はリモートエージェントへの接続をローカルのサブエージェント呼び出しと同じ感覚で実装できます。 instruction フィールドが実質的にそのエージェントの「内部 GEMINI.md」となるため、ルーティング判断の記述が最重要です。

agents/orchestrator/agent.py — A2A Orchestrator 実装（SequentialAgent + RemoteA2aAgent）Python
from google.adk.agents import LlmAgent, SequentialAgent, ParallelAgent from google.adk.agents.remote_a2a_agent import RemoteA2aAgent # ═══════════════════════════════════════════════════════════════════ # ① リモート エージェント（A2A Protocol 経由 — 別サービス・別チーム） # RemoteA2aAgent: Agent Card URL を指定するだけで接続を確立する # ═══════════════════════════════════════════════════════════════════ code_review_remote = RemoteA2aAgent( name="code-review-agent", # description が Orchestrator のルーティング判断基準（Agent Card と一致させる） description=""" コードレビューを実施するリモート専門エージェント（別チーム管理）。 【呼び出す場合】: PR 作成前のセキュリティ・品質チェック 【呼び出さない場合】: コードの実装・修正（implementer を使う） 出力: セキュリティスコア（0-100）・改善提案・重大度別バグ一覧 """, # Agent Card URL: /.well-known/agent.json を自動取得してスキルと認証を確認 agent_card_url="https://review.internal.example.com/.well-known/agent.json", ) security_scanner_remote = RemoteA2aAgent( name="security-scanner", description=""" セキュリティスキャン専門エージェント（本番デプロイ前に必ず呼ぶ）。 【呼び出す場合】: 本番デプロイ前 / 認証・認可コードの変更後 出力: CVE レポート・OWASP スコア・修正優先度マトリクス（JSON 形式） """, agent_card_url="https://security.internal.example.com/.well-known/agent.json", ) # ═══════════════════════════════════════════════════════════════════ # ② ローカル サブエージェント（ADK 内部 — 同一サービス） # ═══════════════════════════════════════════════════════════════════ spec_writer = LlmAgent( name="spec-writer", model="gemini-2.5-flash", # コスト最適化: Orchestratorより軽量モデルを使用 description="新機能の仕様書を docs/specs/ に作成する。ステータスを READY_FOR_IMPL に設定する。", instruction=""" あなたは PM ロールの仕様書作成エージェントです。 以下の形式で docs/specs/{feature-name}.md を作成してください: - 機能概要（2-3文） - 受け入れ条件（箇条書き） - API エンドポイント定義（OpenAPI 形式） - 非機能要件（パフォーマンス・セキュリティ） 完了後、output_key "spec_document" にファイルパスを格納すること。 """, tools=["write_file", "read_file"], output_key="spec_document", # ← output_key は一意に（他と重複禁止） ) implementer = LlmAgent( name="code-implementer", model="gemini-2.5-flash", description="spec_document の仕様に従ってコードを実装する。src/ のみ変更可能。", instruction=""" {spec_document} の仕様に従って実装してください。 【書き込み可能】: src/, tests/ のみ 【禁止】: docs/, agent.json, .env の変更 - TypeScript strict モード必須 - エラーは Result<T,E> 型（throw 禁止） 完了後 output_key "implementation_summary" に変更ファイル一覧を格納。 """, tools=["write_file", "read_file", "run_command"], output_key="implementation_summary", ) # ═══════════════════════════════════════════════════════════════════ # ③ Orchestrator — Sequential: 仕様 → 実装 → レビュー → セキュリティの順 # ═══════════════════════════════════════════════════════════════════ root_agent = SequentialAgent( name="dev-orchestrator", description="開発ワークフロー全体を制御するオーケストレータ", sub_agents=[ spec_writer, # Step 1: 仕様書作成（ローカル） implementer, # Step 2: 実装（ローカル） code_review_remote, # Step 3: コードレビュー（A2A リモート） security_scanner_remote, # Step 4: セキュリティスキャン（A2A リモート） ], )

マルチエージェント ユースケース別 設計パターン

🏢
エンタープライズ クロスチーム協調
フロント・バックエンド・セキュリティチームがそれぞれ独立した A2A エージェントを管理。Orchestrator が Agent Card を使って最適なエージェントを自動選択。異なるフレームワーク（ADK / LangGraph / CrewAI）間も A2A で接続可能。
Orchestrator (ADK)
→ frontend-agent (LangGraph, A2A)
→ backend-agent (ADK, A2A)
→ security-agent (CrewAI, A2A)
🔍
動的ディスカバリー（Discovery Service パターン）
Orchestrator がハードコードされたエージェントリストを持たず、Discovery Service に問い合わせてリモートエージェントを動的に発見。RemoteA2aAgent を実行時に生成して sub_agents に追加する高度なパターン。
await discover_agents() # registry に問合せ
→ RemoteA2aAgent を動的生成
→ LlmAgent に tools として追加
→ LLM が最適エージェントを自律選択
🔄
LoopAgent + A2A レビューループ
実装エージェント（ローカル）がコードを書き、レビューエージェント（リモート A2A）が品質評価。評価スコアが閾値を超えるまで繰り返す品質保証ループ。LoopAgent の max_iterations で無限ループを防止。
LoopAgent(
sub_agents=[
implementer, # ローカル
code_review_remote], # A2A リモート
max_iterations=5
) # output_key "quality_score" が80以上で終了
⚡
ParallelAgent + マルチリージョン分析
異なるリージョン・データセンターで動く分析エージェントを ParallelAgent で同時呼び出し。output_key を必ず一意にしてレースコンディションを防止。集計エージェントが並列結果を統合。
ParallelAgent(
sub_agents=[
RemoteA2aAgent(us_analyzer, # output_key="us_result"
agent_card_url=US_URL),
RemoteA2aAgent(eu_analyzer, # output_key="eu_result"
agent_card_url=EU_URL)]
) # ⚠ output_key は必ず一意に！

マルチエージェント固有の Anti-Patterns — 設計・運用両面

✗ agent.json / Agent Card の Anti-Patterns
  • description が曖昧（「なんでもやります」A2A エージェント）
  • securitySchemes を設定しない（認証なし公開）
  • skills を定義しない（能力発見ができず Orchestrator から使われない）
  • agent.json の description と agent.py の instruction が乖離
  • version を更新せず破壊的変更（サイレントな障害の原因）
  • url が内部 IP のまま本番公開
✗ GEMINI.md / Orchestration の Anti-Patterns
  • GEMINI.md にリモートエンドポイント URL を書かない（管理外に散乱）
  • ParallelAgent で同じ output_key を使う（上書き競合・最大の地雷）
  • LoopAgent の終了条件・max_iterations を設定しない（無限ループ）
  • 全リモートエージェントに gemini-2.5-pro（コスト爆発）
  • エージェント間のファイル所有権を定義しない（書き込み競合）
  • A2A タイムアウト・フォールバック戦略を GEMINI.md に記載しない
