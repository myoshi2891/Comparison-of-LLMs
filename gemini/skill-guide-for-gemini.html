❓ なぜ SKILL.md が必要なのか

🔥 コンテキスト飽和（Context Saturation）問題

AIエージェントに「すべてのドキュメントを読ませる」アプローチでは、モデルの注意機構（Attention Mechanism）が分散し、重要な情報を見落としたり推論精度が著しく低下します。 また大量のトークン消費によって速度低下・コスト増大が起きます。

📊 プログレッシブ・ディスクロージャー — 3段階読み込みの仕組み

SKILL.md は「必要なときだけ、必要な分だけ」読み込まれます。この3段階設計によってトークン消費を極限まで抑えています。

graph LR L1["🏷️ レベル1: メタデータ\nname + description\n常にコンテキストにある\n～100 tokens\nセッション開始時に読込"] L2["📄 レベル2: SKILL.md 本文\nスキルが呼ばれたときのみ\n5,000 tokens以内推奨\n具体的な手順・制約"] L3["📚 レベル3: 参照ファイル\n本文から参照された時のみ\n容量無制限\nAPIドキュメント等"] L1 -->|"スキルが選ばれたとき"| L2 L2 -->|"本文から参照されたとき"| L3 style L1 fill:#fef3c7,stroke:#f59e0b,color:#78350f style L2 fill:#bfdbfe,stroke:#0284c7,color:#0c4a6e style L3 fill:#d1fae5,stroke:#059669,color:#065f46
レベル1 / 常駐
セッション開始時に全スキルのメタデータをロード。1スキル約100トークンなので数十〜数百スキルを待機させてもOK
レベル2 / オンデマンド
ユーザーの入力とdescriptionを意味論的に照合し、最適なスキルを特定。activate_skillツールで本文を展開
レベル3 / 動的参照
本文中で参照された外部ファイルのみを動的に読み込む。スクリプトは実行結果のみがトークン消費

references/ — 参照ドキュメント（必要時のみ読込）

SKILL.md の構成

🔑 YAMLフロントマター（必須）
--- で囲まれたYAML形式のメタデータ。
name（識別子）と description（トリガー条件）が最重要
↕
📝 マークダウン本文（必須）
エージェントへの具体的な指示。以下のセクションで構成：
Overview → Before Starting → Step-by-Step → Examples → Rules
↕
🔗 参照ファイルリンク（任意）
本文中で references/api-docs.md のように外部ファイルを参照。必要時だけ読込

YAMLフロントマター 詳細解説

---
# ✅ 必須: スキルの一意な識別子（ケバブケース・英語小文字）
name: git-commit-formatter

# ✅ 必須: エージェントへのトリガー条件（最重要！）
#    「いつ」「どんな言葉が出たら」このスキルを使うかを明記する
description: |
  Gitコミットメッセージを Conventional Commits 仕様に整形する。
  「コミット」「commit」「コミットメッセージ」などが出たら必ず使用。
  コード変更の記録・バージョン管理作業で呼び出すこと。

# 任意: 依存関係（一部ツールでサポート）
compatibility:
  - git
  - node >= 18

# 任意: 手動呼び出し専用にする（危険な操作に有効）
# disable-model-invocation: true
---
⚠️ description はエージェントの「発動スイッチ」
❌ 悪い例（曖昧）
description: コードをレビューするスキル。
→ AIがいつ使うか判断できない
✅ 良い例（具体的）
description: |
コードレビューを実行する。
"review","レビュー","PR確認"が出たら必ず使う。
→ トリガーが明確で確実に呼ばれる

# Frontend Code Reviewer

## Overview
Reactコンポーネントに対して一貫性のある厳格なレビューを行い、
パフォーマンス・型安全性・アクセシビリティを向上させる。

## Before Starting
レビューを開始する前に以下を確認すること：
- レビュー対象ファイルのパスをユーザーから取得
- 依存するフックやユーティリティファイルも一緒に読み込む

## Step-by-Step Guide
1. **文脈と依存関係の解析**: 対象ファイルと依存関係を追跡して読み込む
2. **静的解析のシミュレーション**: 型の安全性とReactの依存配列を検証
3. **パフォーマンス評価**: useMemo/useCallbackの欠落を特定
4. **a11y確認**: WAI-ARIA属性・キーボードナビゲーションを確認
5. **アーティファクト生成**: 修正後のコードスニペットと共に提示

## Examples
**Input**: `UserProfile.tsx`のパフォーマンスをレビューして

**Output Structure**:
1. 概要: 現在の実装におけるボトルネック
2. 修正案: `React.memo`の適用箇所と修正コード
3. 副作用の検証: 修正による影響範囲

## Rules
❌ 絶対に行わないこと:
- ビジネスロジックの意図を推測して勝手に変更すること
- 不明な点を質問せずに仮定で進めること

✅ 必ず行うこと:
- 変更提案の根拠として公式ドキュメントを引用
- `references/css-guidelines.md` を参照してスタイルをレビュー

gemini skills install https://github.com/example/my-skills.git

skills CLI（統合管理ツール）

npx skills コマンドを使うと、Gemini CLI と Antigravity の両方に同時にスキルを追加できます。 

# Gemini CLI と Antigravity 両方に追加
npx skills add firebase/agent-skills -a gemini-cli -a antigravity

# スキルを検索
npx skills find flutter

# スキルを削除
npx skills remove firebase/agent-skills

# インストール済みスキルを一覧表示
npx skills list

Antigravity への手動コピー

# グローバルインストール（全プロジェクトで使えるようにする）
cp -r my-skill/ ~/.gemini/antigravity/skills/

# ワークスペースインストール（このプロジェクトだけ）
cp -r my-skill/ .agent/skills/
