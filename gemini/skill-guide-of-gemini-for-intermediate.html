01 / BACKGROUND

なぜ SKILL.md が必要か

コンテキストウィンドウが大容量になっても、なぜ「すべてを読ませる」アプローチは破綻するのか。コンテキスト飽和問題と解決策を理解する。

🔥

コンテキスト飽和（Context Saturation）

ドキュメント・コーディング規約・設計書をすべてエージェントに与えると、Attention Mechanismが分散し重要な情報を見落とす。トークン消費が爆発し推論精度が低下する。

⚡

オンデマンド専門知識

SKILL.md は必要なタイミングでのみコンテキストに展開される。セッション中は名前と説明（約100トークン）のみが常駐し、発動時に本文を読み込む設計。

🔄

再利用可能なナレッジ

一度書けば複数プロジェクト・複数ツール（Gemini CLI / Antigravity / Claude Code / GitHub Copilot）で共通利用できるオープンスタンダード。

核心的な洞察: モデルのコンテキストウィンドウが100万トークンになっても、「何もかもを与える」ことは解決策にならない。注意機構の分散という本質的な問題は変わらないからだ。SKILL.md は「何をいつ与えるか」を制御するアーキテクチャである。

プログレッシブ・ディスクロージャー

LEVEL 1
常時コンテキスト常駐

メタデータのロード

セッション開始時に全スキルの name と description のみがシステムプロンプトに注入される。スキル100個あっても約10,000トークン程度の軽量な状態。エージェントはこの情報でスキル一覧を「知っている」。

LEVEL 2
アクティベーション時のみ

SKILL.md 本文の展開

ユーザーのプロンプトとスキルの description を意味論的に照合し最も関連するスキルを特定。activate_skill ツールが呼ばれて初めて SKILL.md の本文（5,000トークン未満推奨）がコンテキストに展開される。

LEVEL 3
動的アクセス

リソースの動的実行

references/ 内のドキュメントはBashでの読み取り時のみ消費。scripts/ 内のスクリプトはソースコードではなく実行結果のみがコンテキストに入る。実質的に無制限の処理能力を「軽量に」実現。

決定論的ツールの統合: LLMは本質的に確率論的（非決定論的）。scripts/ にPython/Bashスクリプトを同梱し SKILL.md から呼び出すことで、検証処理を決定論的にし、出力の一貫性と精度を飛躍的に高めることができる。

05 / SYNTAX

SKILL.md の書き方完全ガイド

YAMLフロントマターの設計から、エージェントを確実に動かすマークダウン本文の構造まで。中級者が陥りやすいミスも解説する。

1. YAMLフロントマター（メタデータ定義）

ファイル最上部に配置し、トリプルダッシュ（---）で囲む。エージェントがスキルを「見つける」ためのルーティング情報。

yaml — SKILL.md Frontmatter
--- name: frontend-reviewer # 必須: ケバブケース・英語小文字・64文字以内 description: | # 必須: エージェントのトリガー条件（最重要） ReactおよびTypeScriptプロジェクトの フロントエンドコードをレビューし、 コードレビュー、PR確認、品質チェック、 という言葉が出たら必ずこのスキルを使うこと。 disable-model-invocation: false # trueにすると自動発動を無効化（危険操作スキルに） category: code-quality ---

2. description フィールドの設計が成否を決める

❌ アンダートリガー（呼ばれない）
name: security-auditor
description: |
  セキュリティ監査を行うスキル。
✅ 正しいトリガー設計
name: security-auditor
description: |
  コードのセキュリティ監査を実施する。
  脆弱性チェック、OWASP Top 10の検証、
  セキュリティ、認証、XSS、SQLi、
  という言葉が出たら必ず使うこと。

3. マークダウン本文の推奨スキーマ

markdown — SKILL.md Body Template
# スキル名 ## Overview このスキルが達成する目的を1〜2文で。 ## Before Starting（前提条件） - ユーザーから提供されるべき入力情報 ## Step-by-Step Guide（実行手順） # 抽象的な思考でなく具体的なアクションを定義 1. **ファイルの解析**: 対象ファイルと依存モジュールを読み込む 2. **静的解析**: 型安全性とReact依存配列の完全性を検証 3. **アーティファクト生成**: 修正後のコードスニペットと共に提示 ## Examples（Few-shot 使用例） # LLMはここを見て期待される振る舞いを模倣する **Input**: `UserProfile.tsx`をレビューして **Output Structure**: 1.ボトルネック 2.修正案 3.影響範囲 ## Rules（制約事項） - ❌ ビジネスロジックを推測で変更しないこと - ✅ 修正根拠として公式ドキュメントを引用すること - ✅ 破壊的変更の前には必ず確認を求めること
06 / PATTERNS

実装パターン4選

シンプルなプロンプト型から、スクリプト統合・ドキュメント参照を組み合わせた高度なパターンまで。

Pattern 1 — Basic（SKILL.md のみ）
最もシンプルな構成。プロンプトエンジニアリングだけで完結するスキル。
git-commit-formatter/SKILL.md
--- name: git-commit-formatter description: | Gitコミットメッセージを Conventional Commits 仕様に従って整形する。 コミット、git commit、コミットメッセージという言葉が出たら必ず使うこと。 --- ## ルール 形式: `<type>(<scope>): <subject>` ## Examples feat(auth): ログイン機能を追加 fix(api): レート制限のバグを修正
軽量即効性あり初心者にも適用可
Pattern 2 — Reference（SKILL.md + /references）
仕様書やAPIドキュメントを参照しながら動作するスキル。コンテキストを汚染せずに豊富な知識を提供できる。
api-integrator/
├── SKILL.md
└── references/
    └── api-spec.md ← オンデマンド参照
仕様書との整合性コンテキスト節約大規模ドキュメントに有効
Pattern 3 — Tool Use（SKILL.md + /scripts）
スクリプトを実行するスキル。決定論的な検証処理を実現。危険な操作には disable-model-invocation: true 必須。
db-migrator/
├── SKILL.md
└── scripts/
    └── migrate.py ← 実際に実行されるスクリプト
決定論的自動化に最強危険操作はdisable-model-invocation必須
Pattern 4 — All-in-One（全要素の組み合わせ）
SKILL.md + scripts/ + references/ + assets/ を全て活用する最高度のパターン。
full-stack-generator/
├── SKILL.md ← 統合指示書
├── scripts/
│   └── scaffold.py
├── references/
│   └── best-practices.md
└── assets/
    └── template.tsx
最大の能力エンタープライズ向け設計コストが高い

09 / LIFECYCLE

実行ライフサイクルの全貌

Discovery → Activation → Consent → Injection → Execution の5フェーズを理解する。

EXECUTION LIFECYCLE — SEQUENCE DIAGRAM
sequenceDiagram participant User as Developer participant IDE as CLI or IDE Interface participant Agent as Agent Core participant FS as Local File System Note over Agent,FS: Phase 1 — Discovery Agent->>FS: Scan .agents/skills/ hierarchy FS-->>Agent: Extract name + description from each SKILL.md Agent->>Agent: Inject metadata into system prompt static User->>IDE: "Review this code for performance issues" IDE->>Agent: Forward prompt Note over Agent: Phase 2 — Activation Logic Agent->>Agent: Semantically match prompt vs all skill descriptions Agent->>Agent: Identify best skill and call activate_skill tool Note over Agent,User: Phase 3 — Consent and Security Agent->>IDE: Request tool execution with skill details IDE->>User: Show approval prompt with skill name and directory path User->>IDE: Approve execution IDE-->>Agent: Approval signal received Note over Agent,FS: Phase 4 — Context Injection Agent->>FS: Obtain read permission for skill directory FS-->>Agent: Return SKILL.md body and referenced assets Agent->>Agent: Expand detailed instructions into context window Note over Agent,User: Phase 5 — Execution Agent->>Agent: Follow Step-by-Step from SKILL.md Agent->>FS: Execute scripts if needed via bash FS-->>Agent: Return execution results only not source code Agent->>IDE: Generate artifacts and analysis results IDE->>User: Report task completion

Phase 3: Consent

スキルがアクティブになる際、UIにスキル名・目的・アクセスするディレクトリパスが明示され承認を求める。悪意あるスクリプトの意図しない実行を防ぐセキュリティモデル。

Phase 4: Context Injection

スクリプトはソースコードではなく実行結果のみがコンテキストに入る。これによりレベル3（scripts/）は実質的にコンテキスト無制限で複雑な処理を実現できる。

11 / BEST PRACTICES

ベストプラクティス とチェックリスト

アンチパターン vs ベストプラクティス

❌ よくある失敗パターン
• description が曖昧すぎる
  → スキルが発動しない

• SKILL.md が 500 行を超える
  → コンテキストを圧迫

• スクリプトにハードコードされた値
  → 再利用できない

• スキルを 20 個以上有効にしている
  → AIが混乱し精度低下

• 危険操作スキルに自動発動を許可
  → disable-model-invocation 必須

• /plan を使わず本番コードを変更
  → 意図しない変更が発生するリスク
✅ ベストプラクティス
• description にトリガーワードを明記
  → 「この言葉が出たら使うこと」

• 大きな参照資料は references/ に分離
  → 本文は指示のみ

• アクティブスキルは 10〜15 個以内
  → 使わないスキルは disable

• Few-shot 例を Examples セクションに
  → モデルが期待動作を模倣する

• /plan で変更前に安全に計画立案
  → read-only で実装計画を確認

• skill-creator でスキルを自動生成
  → v0.26.0〜標準搭載
