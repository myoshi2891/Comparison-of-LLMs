01 / BACKGROUND

なぜ SKILL.md が必要か

コンテキストウィンドウが大容量になっても、なぜ「すべてを読ませる」アプローチは破綻するのか。コンテキスト飽和問題と解決策を理解する。

🔥

コンテキスト飽和（Context Saturation）

ドキュメント・コーディング規約・設計書をすべてエージェントに与えると、Attention Mechanismが分散し、重要な情報を見落とす。トークン消費が爆発し、推論精度が低下する。

⚡

オンデマンド専門知識

SKILL.md は必要なタイミングでのみコンテキストに展開される。セッション中は名前と説明（約100トークン）のみが常駐し、発動時に本文を読み込む設計。

🔄

再利用可能なナレッジ

一度書けば複数プロジェクト・複数ツール（Gemini CLI / Antigravity / Claude Code / GitHub Copilot）で共通利用できるオープンスタンダード。

核心的な洞察: モデルのコンテキストウィンドウが100万トークンになっても、「何もかもを与える」ことは解決策にならない。注意機構の分散という本質的な問題は変わらないからだ。SKILL.md は「何をいつ与えるか」を制御するアーキテクチャである。

LEVEL 1
常時コンテキスト常駐

メタデータのロード

セッション開始時に、全スキルの name と description のみがシステムプロンプトに注入される。スキル100個あっても約10,000トークン程度の軽量な状態。エージェントはこの情報でスキル一覧を「知っている」。

LEVEL 2
アクティベーション時のみ

SKILL.md 本文の展開

ユーザーのプロンプトとスキルの description を意味論的に照合し、最も関連するスキルを特定。activate_skill ツールが呼ばれて初めて SKILL.md の本文（5,000トークン未満推奨）がコンテキストに展開される。

LEVEL 3
動的アクセス

リソースの動的実行

references/ 内のドキュメントはBashでの読み取り時のみ消費。scripts/ 内のスクリプトはソースコードではなく実行結果のみがコンテキストに入る。実質的に無制限の処理能力を「軽量に」実現。

決定論的ツールの統合: LLMは本質的に確率論的（非決定論的）。同じ入力でも異なる出力を返す可能性がある。scripts/ にPython/Bashスクリプトを同梱し SKILL.md から呼び出すことで、検証処理を決定論的にし、出力の一貫性と精度を飛躍的に高めることができる。

05 / SYNTAX

SKILL.md の書き方完全ガイド

YAMLフロントマターの設計から、エージェントを確実に動かすマークダウン本文の構造まで。中級者が陥りやすいミスも解説する。

1. YAMLフロントマター（メタデータ定義）

ファイル最上部に配置し、トリプルダッシュ（---）で囲む。エージェントがスキルを「見つける」ためのルーティング情報。

yaml — SKILL.md Frontmatter
--- name: frontend-reviewer # 必須: ケバブケース・英語小文字・64文字以内 description: | # 必須: エージェントのトリガー条件（最重要） ReactおよびTypeScriptプロジェクトの フロントエンドコードをレビューし、 コンポーネント設計・パフォーマンス・ アクセシビリティの観点から品質を向上させる。 コードレビュー、PR確認、品質チェック、 という言葉が出たら必ずこのスキルを使うこと。 disable-model-invocation: false # trueにすると自動発動を無効化（危険操作スキルに） category: code-quality # 任意: 整理用カテゴリ ---

2. description フィールドの設計が成否を決める

description は人間向けの説明文ではなく、エージェントへのトリガー条件の定義。曖昧すぎるとスキルが呼ばれない（アンダートリガー）。具体的なキーワードを含めることが必須。

❌ アンダートリガー（呼ばれない）
name: security-auditor
description: |
  セキュリティ監査を行うスキル。
✅ 正しいトリガー設計
name: security-auditor
description: |
  コードのセキュリティ監査を実施する。
  脆弱性チェック、OWASP Top 10の検証、
  セキュリティ、認証、XSS、SQLi、
  という言葉が出たら必ず使うこと。

3. マークダウン本文の推奨スキーマ

本文はエージェントへの「行動仕様書」。以下の標準構造に従うことで、エージェントの動作が安定する。

markdown — SKILL.md Body Template
# スキル名 ## Overview このスキルが達成する目的を1〜2文で。 ## Before Starting（前提条件） - ユーザーから提供されるべき入力情報 - 事前に必要な準備・確認事項 ## Step-by-Step Guide（実行手順） # ← 核心部分。抽象的な思考でなく具体的なアクションを定義 1. **ファイルの解析**: 対象ファイルと依存モジュールを読み込む 2. **静的解析**: 型安全性とReact依存配列の完全性を検証 3. **パフォーマンス評価**: 不要な再レンダリングを特定 4. **アーティファクト生成**: 修正後のコードスニペットと共に提示 ## Examples（Few-shot 使用例） # ← LLMはここを見て期待される振る舞いを模倣する **Input**: `UserProfile.tsx`をレビューして **Output Structure**: 1. ボトルネック概要 2. 修正案とコード 3. 影響範囲の考察 ## Rules（制約事項） - ❌ ビジネスロジックを推測で変更しないこと → 不明点はask_userで確認 - ✅ 修正根拠として公式ドキュメントを引用すること - ✅ 破壊的変更の前には必ず確認を求めること
06 / PATTERNS

実装パターン4選

シンプルなプロンプト型から、スクリプト統合・ドキュメント参照を組み合わせた高度なパターンまで。用途に応じた選択基準も解説する。

Pattern 1 — Basic（SKILL.md のみ）
最もシンプルな構成。プロンプトエンジニアリングだけで完結するスキル。コーディング規約の強制、フォーマット統一、定型レビューに最適。
git-commit-formatter/SKILL.md
--- name: git-commit-formatter description: | Gitコミットメッセージを Conventional Commits 仕様に従って整形する。 コミット、git commit、コミットメッセージという言葉が出たら必ず使うこと。 --- # Git コミットフォーマッター ## ルール 形式: `<type>(<scope>): <subject>` ## type の種類 - `feat`: 新機能 - `fix`: バグ修正 - `docs`: ドキュメント変更 - `refactor`: リファクタリング - `test`: テスト追加・修正 ## Examples feat(auth): ログイン機能を追加 fix(api): レート制限のバグを修正
軽量 即効性あり 初心者にも適用可
Pattern 2 — Reference（SKILL.md + /references）
仕様書やAPIドキュメントを参照しながら動作するスキル。SKILL.md 本文には「何を参照すべきか」だけを記述し、詳細は references/ に分離。コンテキストを汚染せずに豊富な知識を提供できる。
api-integrator/
├── SKILL.md
└── references/
    └── api-spec.md ← API仕様書（オンデマンド参照）
api-integrator/SKILL.md
--- name: api-integrator description: | 外部APIとの統合コードを生成する。APIの仕様に基づき正確なコードを作成する。 API連携、HTTPリクエスト、fetch、axios という言葉が出たら使用。 --- # API インテグレーター ## 手順 1. `references/api-spec.md` を読み込んでAPIの仕様を確認する 2. ユーザーが求めるエンドポイントを特定する 3. 仕様に基づき型安全なコードを生成する ## Rules - 必ず参照ファイルを読んでから実装すること - 推測でAPIを呼び出さないこと
仕様書との整合性 コンテキスト節約 大規模ドキュメントに有効
Pattern 3 — Tool Use（SKILL.md + /scripts）
スクリプトを実行するスキル。LLMの確率論的な出力ではなく、決定論的な検証処理を実現。CI/CDパイプラインへの組み込み、繰り返し作業の自動化に最適。
db-migrator/
├── SKILL.md
└── scripts/
    └── migrate.py ← 実際に実行されるスクリプト
db-migrator/SKILL.md
--- name: db-migrator description: | データベースマイグレーションを実行する。 マイグレーション、DB更新、スキーマ変更という言葉が出たら使用。 disable-model-invocation: true # 自動発動を無効化（危険操作） --- # DB マイグレーター ## 実行コマンド ```bash python scripts/migrate.py --env <environment> --version <version> ``` ## 引数 - `--env`: 環境（dev / staging / prod） - `--version`: マイグレーションバージョン番号 ## Rules - スクリプト実行前に必ずユーザーに確認を求めること - prod環境は二重確認必須 - エラー時はロールバックコマンドを提示すること
決定論的 自動化に最強 危険操作はdisable-model-invocation必須
Pattern 4 — All-in-One（全要素の組み合わせ）
SKILL.md + scripts/ + references/ + assets/ を全て活用する最高度のパターン。複雑なドメイン知識・自動検証・出力テンプレートを統合したエンタープライズレベルのスキル。

07 / INSTALLATION

スキルのインストール方法

Git リポジトリ、ローカルディレクトリ、統合CLIツール。3つのインストール手段と、それぞれの使いどころ。

01
Gemini CLI — コマンドでインストール

最もスタンダードな方法。チームで共有するスキルはGitリポジトリに置き、URLを指定してインストールする。

Antigravity — ファイルコピーで配置

Antigravity IDE ではGUIまたはファイルコピーでスキルを配置する。グローバルかワークスペースかを意識して配置先を決める。

09 / LIFECYCLE

実行ライフサイクルの全貌

Discovery → Activation → Consent → Injection → Execution の5フェーズを理解することで、スキルのデバッグと設計が格段に改善される。

EXECUTION LIFECYCLE — SEQUENCE DIAGRAM
sequenceDiagram participant User as Developer participant IDE as CLI or IDE Interface participant Agent as Agent Core participant FS as Local File System Note over Agent,FS: Phase 1 — Discovery Agent->>FS: Scan .agents/skills/ hierarchy FS-->>Agent: Extract name + description from each SKILL.md Agent->>Agent: Inject metadata into system prompt static User->>IDE: "Review this code for performance issues" IDE->>Agent: Forward prompt Note over Agent: Phase 2 — Activation Logic Agent->>Agent: Semantically match prompt vs all skill descriptions Agent->>Agent: Identify best skill and call activate_skill tool Note over Agent,User: Phase 3 — Consent and Security Agent->>IDE: Request tool execution with skill details IDE->>User: Show approval prompt with skill name and directory path User->>IDE: Approve execution IDE-->>Agent: Approval signal received Note over Agent,FS: Phase 4 — Context Injection Agent->>FS: Obtain read permission for skill directory FS-->>Agent: Return SKILL.md body and referenced assets Agent->>Agent: Expand detailed instructions into context window Note over Agent,User: Phase 5 — Execution Agent->>Agent: Follow Step-by-Step from SKILL.md Agent->>FS: Execute scripts if needed via bash FS-->>Agent: Return execution results only not source code Agent->>IDE: Generate artifacts and analysis results IDE->>User: Report task completion

Phase 3: Consent

スキルがアクティブになる際、UIにスキル名・目的・アクセスするディレクトリパスが明示され、ユーザーの承認を求める確認プロンプトが表示される。悪意あるスクリプトの意図しない実行を防ぐセキュリティモデル。

Phase 4: Context Injection

スクリプトはソースコードではなく実行結果のみがコンテキストに入る。これによりレベル3（scripts/）は実質的にコンテキスト無制限で複雑な処理を実現できる。

• リモートDBへのクエリ発行（Fetch）

🧠 SKILL.md の役割
エージェントへの「知識と判断力」

• 何を調べるべきか（What）
• いつ調べるべきか（When）
• 取得したデータをどう適用するか（How）

→ MCPで取得した生データに
　「文脈と制約」を与える
実装例 — クラウドプロビジョニングスキル:
"クラウドのプロビジョニングを行う際は、まずMCPツールを利用して最新のリファレンスを検索し、そのデータに基づいて、このプロジェクト独自の命名規則とセキュリティポリシーに沿ってTerraformコードを生成せよ"

MCPが最新データを取得し、SKILL.md がプロジェクト制約を適用する。この分業により、「常に最新・常にプロジェクト仕様に準拠」した出力が自動的に実現される。
11 / BEST PRACTICES

ベストプラクティス とチェックリスト

中級者が陥りやすいアンチパターンと、公開・運用前に確認すべきチェックリスト。スキル数の適切な管理方法も解説する。

アンチパターン vs ベストプラクティス

❌ よくある失敗パターン
• description が曖昧すぎる
  → スキルが発動しない

• SKILL.md が 500 行を超える
  → コンテキストを圧迫

• スクリプトにハードコードされた値
  → 再利用できない

• スキルを 20 個以上有効にしている
  → AIが混乱し精度低下

• 全ての情報を SKILL.md 本文に書く
  → references/ を活用すべき

• 危険操作スキルに自動発動を許可
  → disable-model-invocation 必須
✅ ベストプラクティス
• description にトリガーワードを明記
  → 「この言葉が出たら使うこと」

• 大きな参照資料は references/ に分離
  → 本文は指示のみ、詳細は動的参照

• スクリプトは引数で動作変更可能に
  → --env dev|staging|prod 等

• アクティブスキルは 10〜15 個以内
  → 使わないスキルは disable

• Few-shot例を Examples セクションに
  → モデルが期待動作を模倣する

• disable-model-invocation で安全管理
  → 破壊的操作は手動発動のみに

公開・運用前チェックリスト

項目をクリックしてチェックを記録できます。

name はハイフン区切りの英語小文字か（64文字以内・予約語なし）
description に具体的なトリガーキーワードが含まれているか（「この言葉が出たら使うこと」）
SKILL.md 本文は 500 行以内か（参照資料は references/ に分離済みか）
スクリプトはハードコードされた値ではなく引数で動作変更できるか
危険な操作（DB削除・インフラ変更等）のスキルに disable-model-invocation: true が設定されているか
Few-shot の Examples セクションが含まれているか（期待する入出力の例示）
Rules セクションで ❌禁止事項 と ✅推奨行動 が明確に定義されているか
実際にエージェントから呼び出されるかテストしたか（gemini skills list で確認）
アクティブなスキルの合計数が 15 個以内に収まっているか
ワークスペーススキルは .gitignore から除外され、チームで共有できるようにコミットされているか
