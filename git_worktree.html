git worktree とは何か — なぜ並列開発に有効か
通常の git clone との根本的な違いと、4プラットフォームのドキュメント並列更新における優位性を理解する。

git worktree は、1つのリポジトリ（.git）を共有しながら、複数のディレクトリで別々のブランチを同時チェックアウトできる Git 標準機能です。Git 2.5（2015年）から搭載されており、追加インストール不要で利用できます。

▸ 構造比較 — clone × 4（アンチパターン） vs worktree × 4（推奨）

clone × 4 との決定的な違い

観点 git clone × 4 git worktree × 4
.git の数 4 つ（独立・完全コピー） 1 つ（共有）
ディスク使用量 全体の 4× 相当 追加分のみ（〜10% 増）
共通変更の反映 4 箇所に手動コピー必須 merge/rebase で即時反映
ブランチ切替コスト 不要（独立） ゼロ（常時並列チェックアウト）
stash 共有 不可 可能（.git が共通）
AI エージェント活用 各ツールが別リポジトリ認識 全ツールが同一履歴を参照

scripts/setup-worktrees.sh — 一括セットアップbash 
#!/bin/bash
set -euo pipefail

PLATFORMS_KEYS=(claude gemini codex copilot)
PLATFORMS_VALUES=(feat/claude-docs feat/gemini-docs feat/codex-docs feat/copilot-docs)

mkdir -p worktrees

for i in "${!PLATFORMS_KEYS[@]}"; do
  PLATFORM="${PLATFORMS_KEYS[$i]}"
  BRANCH="${PLATFORMS_VALUES[$i]}"
  TARGET="worktrees/$PLATFORM"

  [ -d "$TARGET" ] && { echo "⏭  $PLATFORM: skip"; continue; }

  git show-ref --verify --quiet "refs/heads/$BRANCH" || \
    git branch "$BRANCH" main

  git worktree add "$TARGET" "$BRANCH"
  echo "✓  $PLATFORM → $TARGET ($BRANCH)"
done

git worktree list

scripts/sync-all.sh — 全 WT 一括同期bash 
#!/bin/bash
set -euo pipefail
ROOT="$(git rev-parse --show-toplevel)"
WORKTREES=(claude gemini codex copilot)

for WT in "${WORKTREES[@]}"; do
  echo "━━━ Syncing: $WT ━━━━━━━━━━━━━━━━━━"
  cd "$ROOT/worktrees/$WT"

  if [ -n "$(git status --porcelain)" ]; then
    git stash push -u -m "auto: $WT before sync"
    STASHED=true
  else
    STASHED=false
  fi

  if git merge main --no-edit; then
    echo "  ✓ $WT: synced"
    if [ "$STASHED" = true ]; then
      if git stash apply; then
        git stash drop
      else
        echo "  ✗ $WT: STASH APPLY FAILED — 手動解決が必要"
      fi
    fi
  else
    echo "  ✗ $WT: CONFLICT — 手動解決が必要"
  fi
done

cd "$ROOT" && git worktree list
04
メリット・デメリット — 導入判断の材料
4 プラットフォームのドキュメント並列更新における worktree の優位性と注意すべき落とし穴。
✅ メリット
🚀
真の並列開発（stash・切替ゼロ）4 ターミナルで同時に別ブランチを操作。AI エージェントを 4 つ同時起動できる。ブランチ切替コストが完全にゼロ。
💾
ディスク・メモリを大幅節約.git オブジェクトを共有するため clone×4 に比べてディスク使用量が大幅に少ない（差分のみ追加）。
🔗
共通変更が一元管理shared/ の変更を main で 1 回 commit → sync-all.sh で全 WT に反映。手動コピー完全不要。
📜
Git 履歴の一元化全プラットフォームの変更履歴が 1 リポジトリに集約。git log / bisect / blame が横断的に使える。
🤖
AI エージェントとの最良の相性各 WT に設定ファイルを配置するだけで、AI が同一の git 履歴を参照しながら独立した作業を実行できる。
🔀
stash が全 WT 間で共有全 WT が同じ .git を参照するため、WT をまたいで stash を参照・適用できる。
❌ デメリット・注意点
🔒
同一ブランチの重複チェックアウト不可同じブランチを 2 つの WT で同時チェックアウトできない。--force は破壊的なので絶対禁止。
⚡
hook が全 WT に影響する.git/hooks は全 WT 共有。pre-commit が全 WT で実行される。個別設定には追加ロジックが必要。
📁
IDE が混乱しやすいVS Code がメインルートを認識し、worktrees/ 配下を正しく扱えない場合がある。WT ごとに別ウィンドウで開くことを推奨。
🔀
merge コンフリクトのリスク4 ブランチが並行して shared/ を変更すると main merge 時に複数コンフリクトが発生。保護ブランチ設定が必要。
🧹
WT 削除手順が特殊rm -rf だけでは git の追跡が残る。git worktree remove または git worktree prune が必要。
📊
gc / fetch が重くなりうる多数の WT 存在時に git gc や git fetch が遅くなる。定期的な git worktree prune が必要。

🚫
ルール 1: 同一ブランチを複数 WT にチェックアウトしない--force フラグは .git の整合性を破壊するため絶対禁止。git worktree list で現状確認してから追加する。
⚠️
ルール 2: WT の削除は必ず git worktree remove を使うrm -rf worktrees/claude だけでは .git 内のメタデータが残存する。削除後は git worktree prune で必ずクリーンアップする。
⚠️
ルール 3: shared/ (common-header リソース) への変更は dev ブランチ経由のみ各 WT から直接 shared/ 内の共通リソースを変更すると merge コンフリクトが多発する。CODEOWNERS で特定のレビュアーを指定するだけでなく、保護ブランチや Ruleset の設定で 「PRマージ必須」「特定ユーザーのみ変更可」を強制することが必須。
🔍
ルール 4: git stash のメッセージには WT 名を含めるstash は全 WT 共通の .git に保存されるため混在する。例: git stash push -m "claude: WIP nav"
📋
ルール 5: pre-commit hook は全 WT で動作することを前提に設計するhooks/ は全 WT で共有。特定 WT だけ除外したい場合は $GIT_DIR 変数で判定するロジックを hook に追加する。
🤖
ルール 6: AI エージェントはワークツリーのルートから必ず起動するメインルート（ai-docs/）から起動すると全 WT 横断でファイルを変更するリスクがある。cd worktrees/claude && claude のように WT 内から起動する。
🔄
ルール 7: 定期的に git worktree prune を実行する削除済み WT のメタデータが蓄積するため、週次または CI で git worktree prune && git fetch --prune を実行してリポジトリを健全に保つ。

name: Parallel Docs Validation
on:
  push:
    branches:
      - 'feat/*-docs'
      - 'dev'  # dev へのマージ時も検証
  pull_request:
    branches:
      - 'main'  # main への PR 時も検証

jobs:
  validate-docs:
    runs-on: ubuntu-latest
    strategy:
      matrix:
        platform: [claude, gemini, codex, copilot]
      fail-fast: false

    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0

      - uses: oven-sh/setup-bun@v2

      - name: Setup worktree
        run: |
          git worktree add /tmp/${{ matrix.platform }} \
            feat/${{ matrix.platform }}-docs || echo "Branch may not exist yet"

      - name: Validate HTML
        working-directory: /tmp/${{ matrix.platform }}
        run: |
          bunx vnu-jar --skip-non-html ${{ matrix.platform }}/agent.html
          bunx vnu-jar --skip-non-html ${{ matrix.platform }}/skill.html

      - name: Check Links
        uses: lycheeverse/lychee-action@v2
        with:
          args: /tmp/${{ matrix.platform }}/*.html

git worktree は 「同一リポジトリを複数の作業ディレクトリで同時並行作業する」 最もクリーンな解決策です。 4 プラットフォームのドキュメント（agent.html / skill.html 計8ファイル）を AI ツールと WebSearch で並列更新する今回のシナリオでは、clone × 4 よりも worktree × 4 が明確に優位です。 ただし ① 同一ブランチの重複チェックアウト禁止 ② shared/ (common-header リソース) は dev 経由のみ変更 ③ 削除時は git worktree remove 必須 の 3 ルールを守ることが成功の条件です。 各 WT で Claude Code / Antigravity / Codex / Copilot を起動し、WebSearch で最新情報を調査しながら agent.html と skill.html を更新することで、 4 ツール × 並列 = 最大スループット の開発体験が実現します。
