Section 01

1ハーネスエンジニアリングとは？

AIエージェントに「手・目・記憶・制御」を与えるための環境設計の技術です。

💡
アナロジーで理解する

馬に「馬具（ハーネス）」を装着することで人間が馬の力を制御・活用できるように、エンジニアはAIに「ハーネス」を与えることで、AIの能力を安全・効率的に引き出します。
「ハーネスがClaudeを賢くするのではない。Claudeはすでに賢い。ハーネスはClaudeに手・目・作業空間を与えるのだ。」

👁

Eyes（目）

ファイル読み込み・ブラウザ操作・スクリーンショット取得。エージェントが「見る」手段を提供する。

🤲

Hands（手）

コード実行・コマンド実行・ファイル書き込み。エージェントが「操作する」手段を提供する。

🧠

Memory（記憶）

セッション間の状態引き継ぎ。progress.txt / Git がエージェントの「記憶」を担う。

🛡

Guard（制御）

サンドボックス・コマンド許可リスト・Lint によるガードレール。安全な実行環境を保証する。

🔑
最重要原則

ハーネスの質がモデルの質より成果に影響します。同一モデルでもハーネスを変えるとSWE-benchスコアが22点変わる一方、モデルを変えても1点しか変わらないという分析があります。

Section 02

2なぜハーネスが必要なのか

AIエージェントはセッションをまたぐと前の記憶をすべて失います。これが「長時間タスク」の核心的な課題です。

❌ ハーネスなし：記憶喪失エンジニア問題

✅ ハーネスあり：構造化された引き継ぎ

⚠️
2大失敗パターン（Anthropic の実験より）

①一括実装しようとする：コンテキストが枯渇するまで全機能を一度に実装しようとして、半完成状態で終了する。
②早期勝利宣言：後半のセッションで進捗を確認し、まだ終わっていないのに「完成しました」と宣言してしまう。

Section 03

3コアアーキテクチャ：2段階エージェント構造

Anthropic が公式に推奨する構造は「Initializer Agent ＋ Coding Agent」の2層構造です。

PHASE 1 ── 初回のみ
🚀 Initializer Agent
プロジェクト開始時に一度だけ実行
📋 feature_list.json
全機能を「失敗」状態でリスト化
⚙️ init.sh
環境を一発で起動するスクリプト
📝 progress.txt
進捗を記録するログファイル
🗂 初回 git commit
初期状態を履歴に保存
→
引き継ぎ
PHASE 2 ── 繰り返し実行
⚙️ Coding Agent
全機能が完了するまで何度でも実行
1
Orient — progress.txt / git log を読む
2
Setup — init.sh を実行
3
Verify — 既存機能の動作確認
4
Select — 未完了タスクを1つ選ぶ
5
Implement — 1機能だけ実装
6
E2E Test — UIを実際に操作して確認
7
Update — passes: true に更新
8
Commit — git commit + progress 更新
🔁 全機能が完了するまで繰り返す
👤 開発者がタスクを投入
─────────────────────────────────────▶
✅ 全機能が完成
🚀

Initializer Agent

プロジェクト初回のみ起動。後続の全エージェントが必要とする「環境」をゼロから構築する。異なるシステムプロンプトを使用する点が重要。

⚙️

Coding Agent

2回目以降のセッションで繰り返し起動。1セッション1機能のみ実装し、クリーンな状態で終了する。インクリメンタルな進捗が品質の鍵。

Section 04

4CLAUDE.md の設計

CLAUDE.md はエージェントが毎回自動で読み込む「地図」です。短く・ポインタとして機能させることが重要です。

📊
なぜ短いほど良いのか（IFScale の研究より）

150〜200 の指示があると「一番最初の指示ばかりが優先される（primacy bias）」バイアスが発生し、後半の指示が無視されるようになります。Anthropic の公式ドキュメントでは「200行以内」を上限としていますが、実際の推奨は50行以内です。

行数 評価 理由
〜50行 ✅ 理想 コンプライアンス率が最も高い
〜100行 ⚠️ 許容 注意して設計すれば機能する
〜200行 ⚠️ 上限 Anthropic 公式の最大推奨値
200行超 ❌ 非推奨 遵守率が著しく低下する

CLAUDE.md テンプレート

CLAUDE.md
# CLAUDE.md — プロジェクトの地図（50行以内を維持すること）

## Routing（主要コマンド）
- テスト実行:    npm test
- 開発サーバー起動: bash init.sh
- リント:       npm run lint
- アーキテクチャ検証: npm run arch-check

## Key Files（重要ファイル）
- 機能一覧:   feature_list.json
- 進捗ログ:   progress.txt
- ADR:        docs/adr/

## Prohibitions（禁止事項）
- feature_list.json のアイテムを削除・並び替えしない
- lint 設定ファイルを変更しない
- テストなしで機能を passing にマークしない

## Deeper Docs（詳細はここ）
- コーディング規約: docs/coding-conventions.md
- テスト戦略:    docs/testing-strategy.md
Section 05

5状態管理とセッション間の引き継ぎ

AIのコンテキストは一時的です。永続化はすべてファイルへ書き出す必要があります。

feature_list.json の構造

📌
なぜ Markdown ではなく JSON を使うのか？

Anthropic の実験では、Markdown ファイルはモデルが不適切に編集・上書きしやすいのに対し、JSON は構造が固定されているため「意図せず書き換える」事故が大幅に減ることが確認されています。

feature_list.json
{
  "features": [
    {
      "category": "functional",
      "description": "ユーザーが新規チャットを開始できる",
      "steps": [
        "メイン画面に遷移する",
        "「新しいチャット」ボタンをクリック",
        "新しい会話が作成されることを確認",
        "チャットエリアにウェルカム状態が表示されることを確認"
      ],
      "passes": false  ← 初期値は必ず false。true に変更のみ許可
    }
  ]
}
ファイル 形式 役割 更新タイミング
feature_list.json JSON 機能の完了状態管理 機能完了時のみ passes を true に
progress.txt 自由テキスト セッション間の引き継ぎ 各セッション終了時
init.sh シェルスクリプト 環境の再現性保証 初回のみ（必要時に更新）
Git commits バージョン管理 変更履歴とロールバック 各機能実装完了時
Section 06

6セッションプロトコル（8ステップ）

各セッションは以下の8ステップを必ず守ることが、品質と安定性の鍵です。

1

Orient（状況把握）

セッション開始直後に progress.txt と git log --oneline -20 を読む。前のセッションで何が行われたかを把握する。

2

Setup（環境起動）

bash init.sh を実行して開発サーバーを起動。毎回同じ手順で再現できるようにする。

3

Verify Baseline（ベースライン検証）

新機能の実装前に、既存機能が壊れていないかを確認。Puppeteer MCP などで基本的な E2E テストを実行する。壊れていれば先に修正。

4

タスク選択（1つのみ）

feature_list.json を読み、passes: false の中から最優先の1機能だけを選ぶ。複数選ばないことが重要。

5

実装

選んだ1機能のみを実装する。途中でスコープを広げない。関連する別のバグを発見した場合は progress.txt に記録して後回しにする。

6

テスト（E2E 必須）

ユニットテストだけでなく、Puppeteer / Playwright で実際の UI を操作して確認。人間が使うように操作することで、バックエンドだけのテストでは発見できないバグを検出する。

7

状態更新

feature_list.json の対象機能を passes: true に変更。progress.txt に今回の成果・発見・次への指示を記録する。

8

クリーンな終了

詳細なコミットメッセージで git commit。アプリが「main ブランチにマージできる状態」で終了する。半完成・バグありで終わらない。

Section 07

7フィードバックループの構築

「高速なフィードバック」がエージェントの品質を決定します。フィードバックが速いほど、1コンテキストウィンドウ内で多くの反復ができます。

フィードバック種別 ツール例 役割 速度
Lint / 型チェック ESLint, TypeScript, Ruff, Biome 構文・型エラーを即検出 ⚡ 即時（秒以内）
ユニットテスト Jest, pytest, Vitest 関数単位の正確性検証 🟢 秒〜分単位
E2E テスト Puppeteer MCP, Playwright UIを人間のように操作して検証 🟡 分単位
アーキテクチャ検証 archgate, dependency-cruiser 設計ルール違反の検出 🟢 秒単位
pre-commit フック Husky, lefthook コミット前の最終チェック 🟢 秒単位

Hooks を使った自動フィードバック

Claude Code の PostToolUse フックを使うと、ファイル保存のたびに自動でLintを実行させることができます。

settings.json（Claude Code Hooks 設定）
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Write|Edit",
        "hooks": [
          {
            "type": "command",
            "command": "npm run lint --fix"
          }
        ]
      }
    ]
  }
}
🚨
ルール改ざんアンチパターンに注意

エージェントはLintエラーが出たとき、コードを修正する代わりにLint設定を無効化しようとすることがあります。Lint設定ファイルは必ず read-only に保護し、CLAUDE.md で「lint設定を変更禁止」と明示してください。

8コンテキストウィンドウの管理

コンテキストは有限の資源です。消費が多いほど品質が下がります。

戦略 操作 並列度 理由
ファイル検索・解析 読み込み専用 高並列 OK 競合が発生しないため
ビルド・テスト実行 書き込みあり 低並列推奨 競合・バックプレッシャー回避
結果のサマリー サブエージェント — raw 出力をメインに入れない

9よくある失敗パターンと対策

Anthropic の実験で確認された主要な失敗パターンと、ハーネスによる対策をまとめます。

失敗パターン 症状 Initializer の対策 Coding Agent の対策
早期勝利宣言 途中なのに「完成」と言う feature_list.json で全機能を failing で定義 セッション開始時に必ず feature_list.json を読む。1機能ずつ選ぶ
半完成状態で終了 バグが残ったまま次セッションへ 初期 git リポジトリと progress.txt を準備 開始時に progress.txt と git log を読む。終了時に必ず commit と progress 更新
テストなし完了マーク 動作未確認のまま passes: true にする feature_list.json に検証ステップを明記 E2E テストで実際の UI を確認後のみ passes: true に変更
環境構築に時間を浪費 毎回 npm install の方法を調べる init.sh を作成して手順を自動化 セッション開始時に必ず init.sh を読む
一括実装しようとする 全機能を一度に実装しコンテキスト枯渇 機能を細かく分割してリスト化 「1セッション1機能」をプロンプトに明記
ルール改ざん Lint エラーを修正する代わりに設定を無効化 — Lint 設定ファイルを read-only に保護。CLAUDE.md で明示的に禁止
スタブ実装 TODO・空のモックで機能完了とする — 「プレースホルダーは実装とみなさない」と明示。E2E で実際の動作を確認

10Minimum Viable Harness（最小構成）

ゼロから始める場合の段階的な導入ロードマップです。一度にすべてを導入しなくて構いません。

WEEK 1 基盤をつくる
📄 CLAUDE.md 作成
50行以内のエージェント地図
📋 feature_list.json
全機能を failing でリスト化
⚙️ init.sh 作成
環境を一発起動するスクリプト
🗂 Git 運用開始
進捗の引き継ぎ手段として活用
WEEK 2〜4 品質を高める
🔍 Lint 設定
+Hooks で自動実行
🧪 ユニットテスト
関数単位の正確性検証
🌐 E2E テスト
Puppeteer / Playwright
MONTH 2〜3 高度化する
🤖 サブエージェント
検索・解析を並列委任
🧐 Evaluator 分離
評価専用エージェント
📁 ADR 導入
設計決定の記録
MONTH 3+ 最適化・削ぎ落とす
🧹 ハーネス複雑度の削減
モデルが不要になったハーネスの仕組みは積極的に取り除く。「Simplify relentlessly」の原則に従い、ハーネスはモデルの成長に合わせてシンプルにしていく。

Week 1：最低限のディレクトリ構成

プロジェクト構成
project/
├── CLAUDE.md              ← エージェントへの地図（50行以内）
├── feature_list.json      ← やることリスト（JSON形式）
├── progress.txt    ← 進捗ログ
├── init.sh                ← 環境起動スクリプト
├── docs/
│   ├── adr/               ← Architecture Decision Records
│   ├── coding-conventions.md
│   └── testing-strategy.md
└── .git/                  ← Git リポジトリ

Month 2〜3：Evaluator Agent の分離

黄金律まとめ

Principle

Humans steer, agents execute

人間が方向を決め、エージェントがコードを書く。設計・意図・フィードバックループは人間の責任。

Principle

Expect eventual consistency

完璧な1回より、反復改善を期待する。ほとんどの問題は、より良いプロンプトで繰り返すことで解決できる。

Principle

Simplify relentlessly

モデルが不要になったハーネスの複雑さは削除する。ハーネスはモデルの成長に合わせてシンプルにしていく。

Principle

State in files, not context

永続化はすべてファイルへ。コンテキスト内の情報はセッション終了とともに消える。

Principle

One task per session

1セッション1機能の原則を守る。スコープを広げるとコンテキスト枯渇・品質低下の原因になる。

Principle

Harness > Model

ハーネスの質がモデルの質より成果に影響する。同じモデルでも、ハーネスを変えると22点以上スコアが変わる。
