Ch.01 — K-Level: Advanced

内部アーキテクチャ深掘り

Hermesの中核は run_agent.py の AIAgent クラスです。CLI・Gateway・ACP・Batch Runnerなど全エントリポイントが同一クラスを共有し、プラットフォーム差異はエントリポイント側で吸収されます。

エージェントループ — データフロー

flowchart TD A["User Input (CLI / Gateway / ACP)"] --> B["HermesCLI .process_input()"] B --> C["AIAgent .run_conversation()"] C --> D["prompt_builder .build_system_prompt()"] D --> E["runtime_provider .resolve()"] E --> F{API Mode} F -->|chat_completions| G["OpenAI 互換 API"] F -->|codex_responses| H["Codex API"] F -->|anthropic_messages| I["Anthropic Messages API"] G & H & I --> J{Tool Call?} J -->|Yes| K["model_tools .handle_function_call()"] K --> L["tools/approval.py 危険コマンド検査"] L -->|承認| M["Tool 実行"] L -->|拒否| N["Error Return"] M --> J J -->|No| O["Final Response"] O --> P["SessionDB (SQLite + FTS5)"]

プロンプトの3層構造

システムプロンプトは安定度別に3層に分割されており、会話中は変更されません（プロンプトキャッシュ最適化のため）。

1
stable 層 — エージェントID / SOUL.md / ツールスキーマ(28ツールセット) / スキルインデックス
キャッシュ対象
2
context 層 — AGENTS.md / CLAUDE.md / .cursorrules / プロジェクトコンテキスト
プロジェクト固有
3
volatile 層 — MEMORY.md (~800 tokens) / USER.md (~500 tokens) / タイムスタンプ
セッション毎
⚠
/model コマンドを実行するとツールスキーマが変更され、プロンプトキャッシュが無効化されます。次の推論呼び出しで全トークンが再送されるためコストが増加します。

ツールレジストリの依存チェーン

run_agent.py / cli.py / batch_runner.py
model_tools.py（ツール発見・スキーマ収集）
tools/*.py（import時に自動登録）
tools/registry.py（依存なし）

tools/*.py ファイルはトップレベルで registry.register() を呼び出し、importと同時に自動登録されます。手動リスト管理不要で、新規ツールファイルを追加するだけで即利用可能です。

Ch.02 — K-Level: Advanced

セキュリティモデル全体像

Hermes Agentは 7層の多層防御（defense-in-depth） セキュリティモデルを採用しています。各レイヤーは独立して機能し、1層が突破されても次の層が守ります。

7層セキュリティスタック

1
User Authorization — allowlist / DM pairing でアクセス制御
バイパス不可
2
Dangerous Command Approval — 破壊的操作の人間ループ確認
YOLO で一部迂回可
3
Container Isolation — Docker/Singularity/Modal サンドボックス
コンテナが境界
4
MCP Credential Filtering — サブプロセスへの環境変数分離
allowlist のみ通過
5
Context File Scanning — AGENTS.md 等のプロンプトインジェクション検出
自動・無効化不可
6
Cross-Session Isolation — セッション間データアクセス不可、パストラバーサル対策
自動
7
Input Sanitization — working directory パラメータを allowlist で検証
自動
ℹ
Layer 3（コンテナバックエンド）を使用すると Layer 2（危険コマンド承認）はスキップされます。コンテナ自体がセキュリティ境界となるため、コンテナ内の破壊的コマンドはホストを傷つけられません。本番環境では docker / modal / daytona バックエンドを使用することで承認プロンプトなしの完全自動化が実現できます。

Ch.03 — K-Level: Intermediate→Advanced

コマンド承認システムの詳細設計

承認モード 3種類の比較

モード 動作 推奨環境
manual 【デフォルト】 危険コマンドは常にユーザーへ承認プロンプトを表示 個人開発・インタラクティブ利用
smart 補助LLMがリスク評価。低リスク→自動承認、高リスク→自動拒否、不明→手動プロンプト 本番ゲートウェイ（推奨）
off 全承認チェック無効。--yolo と同等 CI/CD・コンテナ内のみ

Hardline Blocklist — YOLO 時も機能する絶対防壁

以下のパターンは --yolo、approvals.mode: off、cron_mode: approve、「常に許可」クリックのいずれの状況でも 絶対にブロック されます。

パターン 理由 バイパス可否
rm -rf / とその変形 ファイルシステムルートを完全削除 ✗ 絶対不可
rm -rf --no-preserve-root / 明示的ルート削除フラグ ✗ 絶対不可
:(){ :|:& };: fork bomb 再起動まで全CPU占有 ✗ 絶対不可
mkfs.* マウント済みルートデバイス 稼働中システムのフォーマット ✗ 絶対不可
dd if=/dev/zero of=/dev/sd* 物理ディスクのゼロ埋め ✗ 絶対不可
信頼できないURLを sh へパイプ RCE攻撃ベクター ✗ 絶対不可

承認フロー — CLI vs Gateway

flowchart TD A["コマンド実行要求"] --> B{Hardline Blocklist?} B -->|Yes| BLOCK["無条件ブロック 設定に関係なく"] B -->|No| T["tirith スキャン"] T --> C{approvals.mode} C -->|manual| D["ユーザー承認プロンプト 60秒タイムアウト後 DENY"] C -->|smart| E["補助 LLM がリスク評価 低リスク→自動承認 高リスク→自動拒否 不明→手動"] C -->|off| G["全コマンド自動承認"] D --> H{"once / session / always / deny"} E --> H G --> RUN["実行"] H -->|allow| RUN H -->|deny| DENY["拒否"]

良い例 / 悪い例

✅ 本番ゲートウェイ推奨
approvals: mode: smart timeout: 120 cron_mode: deny destructive_slash_confirm: true
❌ 絶対にやらない設定
approvals: mode: off # 本番cronで # cron_mode: approve # も同様に危険
⛔
cron_mode: approve は headless で動作する cron ジョブが危険コマンドを自動承認します。本番環境では絶対に使用しないでください。デフォルトの cron_mode: deny を維持することを強く推奨します。
~/.hermes/config.yaml # 本番ゲートウェイ推奨設定 approvals: mode: smart # LLM リスク評価 timeout: 120 # 応答遅延を考慮 cron_mode: deny # headless cron は常に拒否 mcp_reload_confirm: true destructive_slash_confirm: true # 恒久許可パターン（慎重に追加） command_allowlist: - systemctl restart nginx # 特定サービスのみ

ユーザー認可 & DM Pairing

認可チェックの優先順位

flowchart TD A["受信メッセージ"] --> B{プラットフォーム別 allow-all? DISCORD_ALLOW_ALL_USERS} B -->|Yes| ALLOW["アクセス許可"] B -->|No| C{DM Pairing 承認済み?} C -->|Yes| ALLOW C -->|No| D{プラットフォーム別 allowlist? ⚠ allowlist が何も設定されていない場合、全ユーザーがデフォルトで拒否されます。ゲートウェイ起動時にログへ警告が出力されます。

DM Pairing — OWASP 準拠のセキュアなユーザー追加フロー

  • 1
    未知のユーザーが DM を送信
    Bot は 8文字ペアリングコードを返信（0/O/1/I を除く32文字、secrets.choice() で暗号学的生成）
  • 2
    Bot 運営者が CLI で確認・承認
    hermes pairing list で保留リスト確認後、hermes pairing approve telegram ABC12DEF で承認
  • 3
    ユーザーが永続承認リストへ追加
    ~/.hermes/pairing/{platform}-approved.json（chmod 0600）に保存。以降の全メッセージが処理される。

クレデンシャル安全管理

サンドボックス別フィルタリング動作

サンドボックス デフォルトフィルター パススルー上書き
execute_code KEY/TOKEN/SECRET/PASSWORD/AUTH を含む変数名をブロック スキル宣言 or config で許可
terminal (local) Hermes インフラ変数（プロバイダーキー・GW トークン）をブロック env_passthrough リストで許可

スキルによる自動パススルー（推奨）vs 手動設定（非推奨）

✅ スキル frontmatter で宣言（v0.5.1+）

❌ インフラキーをパススルーに追加
terminal: env_passthrough: # NG: プロバイダーAPIキーは絶対に追加しない - OPENAI_API_KEY - TELEGRAM_BOT_TOKEN
ℹ
v0.5.1 以降、スキルが宣言した環境変数は Docker/Modal コンテナへ 自動でフォワード されます。docker_forward_env への手動追加は不要です。

サブエージェント委譲と並列処理

delegate_task vs execute_code の使い分け

要素 delegate_task execute_code
推論 完全な LLM ループ なし（コード実行のみ）
コンテキスト 新規の独立した会話 会話なし・スクリプトのみ

トークンコスト 高い 低い

並列サブエージェントの深度制限

flowchart TD P["親エージェント (depth 0)"] subgraph D1["depth 1 (max_spawn_depth=1 デフォルト)"] L1["子1 role=leaf 委譲不可"] L2["子2 role=leaf"] L3["子3 role=leaf"] end subgraph D2["depth 2 (max_spawn_depth=2 設定時)"] O1["子1 role=orchestrator 委譲可能"] G1["孫1"] G2["孫2"] end P --> L1 & L2 & L3 P -->|"depth>=2"| O1 O1 --> G1 & G2

良い例 / 悪い例

✅ コンテキストを明示的に渡す

no-agent モード（ウォッチドッグ）

✅ コストゼロの監視タスク
hermes cron create "every 5m" \ --no-agent \ --script memory-watchdog.sh \ --deliver telegram \ --name "memory-watchdog" # stdout 空 = silent tick（通知なし） # stdout あり = Telegram へ配信 # non-zero exit = エラーアラート
❌ 文脈を前提とした cron プロンプト
cronjob( prompt="あのサーバー問題を確認して", # NG: cron は独立したセッション。「あのサーバー」を知らない schedule="every 5m" )
⛔
Cron ジョブのプロンプトは作成・更新時に プロンプトインジェクション と 資格情報窃取パターン がスキャンされます。不可視Unicode、SSHバックドア試行、機密情報窃取ペイロードを含むプロンプトはブロックされます。

Ch.12 — K-Level: Advanced

本番デプロイメント チェックリスト

必須 10 項目

  • 1
    明示的な allowlist を設定
    GATEWAY_ALLOW_ALL_USERS=true は本番禁止。TELEGRAM_ALLOWED_USERS=... 等を設定。
  • 2
    コンテナバックエンドを使用
    terminal.backend: docker。危険コマンドチェックが不要になり完全自動化が可能。
  • 3
    リソース制限を設定
    container_cpu / container_memory / container_disk に必要最小限の値を設定。
  • 4
    .env のパーミッション保護
    chmod 600 ~/.hermes/.env を必ず実行。バージョン管理に含めない。
  • 5
    DM Pairing を有効化
    ユーザーIDのハードコードの代わりにペアリングコードを使用。
  • 6
    command_allowlist を定期監査
    hermes config edit で定期的に確認し、不要なパターンを削除。
  • 7
    terminal.cwd を明示設定
    機密ディレクトリからの操作を防止。絶対パスで指定する。
  • 8
    root 以外のユーザーで実行
    専用の非特権ユーザーでゲートウェイプロセスを実行。
  • 9
    cron_mode: deny を維持
    デフォルト値のまま変更しない。headless cron での危険コマンド自動実行を防止。
  • 10
    定期アップデート & アドバイザリーチェック
    hermes update + hermes doctor を週次で実行。サプライチェーン汚染に素早く対応。

最小権限 Docker 本番設定

~/.hermes/config.yaml — 本番推奨terminal: backend: docker docker_image: "nikolaik/python-nodejs:python3.11-nodejs20" docker_forward_env: [] # 空 = コンテナに機密情報を渡さない container_cpu: 1 container_memory: 2048 # 2GB container_disk: 10240 # 10GB container_persistent: false # ephemeral cwd: "/workspace" approvals: mode: smart timeout: 120 cron_mode: deny destructive_slash_confirm: true security: tirith_enabled: true tirith_fail_open: false # 高セキュリティ allow_private_urls: false allow_lazy_installs: false
