Section 01

01.Google Antigravity とは — 従来IDEとの比較

Google Antigravityは2025年11月18日にGemini 3と同時発表されたエージェント・ファースト型IDEです。 VS Codeのフォークをベースに、Claude CodeのようなCLIツールではなくフルIDEとして「エージェントが主役」のパラダイムを実装しています。 人間の役割は「コードを書く人」から「アーキテクトとして指揮する人」へと変わります。

▸ 従来IDE vs Antigravity — パラダイムの違い
❌ 従来のIDE
👤 人間 — コードを書く主役
↓
🤖 AI補完 — 提案するだけ
↓
✅ コード完成（人間が判断・実行）
VS
✅ Google Antigravity
🤖 AIエージェント — 自律的に計画・実行・検証
↓
👤 人間（アーキテクト）— 方針を決め・レビューする
↓
📊 Artifact生成 — 証拠付きで結果を可視化

Antigravity vs Claude Code — 比較表

特性 Google Antigravity Claude Code
種別 Agent-First IDE（VSCode fork） CLI / ターミナルエージェント
主要AI Gemini 3.1 Pro / 3 Flash + Claude Sonnet 4.6 / Opus 4.6 / GPT-OSS 120B Claude Opus / Sonnet / Haiku
永続記憶 GEMINI.md + Knowledge Base + Rules CLAUDE.md + MEMORY.md
ナレッジ拡張 SKILL.md（進歩的開示） SKILL.md（同一規格）
手順自動化 Workflows (.md) Custom Commands (.md)
ブラウザ自動化 Browser Subagent（ネイティブ） なし（MCP経由）
エビデンス生成 Artifacts（自動） なし（手動）
価格 無料（Public Preview中） / Google AI Pro $19.99/月 Anthropicプランに依存
ℹ️
SKILL.mdは共通規格です。AntigravityとClaude Codeは同一のSKILL.md形式を採用しており、ファイルを両プラットフォーム間でほぼそのまま移植できます。 またv1.20.3（2026-03-05）以降はAGENTS.mdにも対応し、他のエージェントツールとのルール共有が可能になりました。

Section 03

03.GEMINI.md — グローバル永続メモリ

Claude CodeのCLAUDE.mdに相当するグローバル永続メモリファイル。 ~/.gemini/GEMINI.mdに配置し、全プロジェクト・全エージェントセッションに横断的に注入されます。 プロジェクト固有のルールは後述の.agent/rules/に分離します。

~/.gemini/GEMINI.md — ベストプラクティステンプレート 
# PROJECT: my-saas-app

## プロジェクト概要（2〜4文で）
Next.js 15 + Supabase のマルチテナント SaaS。
本番: Vercel / DB: PostgreSQL (pgvector付き)

## 技術スタック
- Frontend: Next.js 15 App Router, TypeScript
- Backend: Supabase Edge Functions
- Auth: Supabase Auth + Row Level Security

## ビルド・テストコマンド
- ビルド: `pnpm build`
- テスト: `pnpm test`
- Lint: `pnpm lint`

## ⚠️ 禁止操作（絶対厳守）
- `supabase db reset` は絶対に実行しない（本番データ消去）
- `.env.production` の読み書き禁止
- `--force` フラグは使わない

## エージェント委譲ルール
- タスクが3件以上かつ独立している → 並列実行
- タスクに依存関係がある → 順次実行
- 不確実な場合は実装前に確認すること

## @import（モジュール分割）
@./docs/architecture.md
@./docs/api-conventions.md
❌ やってはいけない例 
# My Project

# ❌ コードスニペットを大量に貼り付ける（トークン浪費）
function myUtil() {
  const data = fetch('/api/data');
  ... 300行のコード ...
}

# ❌ APIキーや機密情報を直接記載
SUPABASE_KEY=eyJhbGci...（シークレット）

# ❌ 変更履歴を蓄積させる（Gitで管理すること）
2026-01-15: ○○を修正した
2026-01-20: △△を追加した

# ❌ プロジェクト固有情報をグローバルGEMINI.mdに書く
本番DBのIP: 192.168.1.100
⚠️
機密情報は.geminiignoreで除外し、変更履歴はGitで管理。大きな資料は@./docs/guide.mdのように@importで分割参照しましょう。
⚠️
アンチパターン: GEMINI.mdにプロジェクト固有のDB設定やAPIエンドポイントを書かない。 グローバルファイルはプロジェクト固有情報を持つべきではありません。プロジェクト情報は.agent/rules/または.context/へ。
Section 04

04.SKILL.md — 進歩的開示ナレッジ（最重要）

AntigravityのSkillsは「Progressive Disclosure（進歩的開示）」という設計思想に基づきます。 全てのナレッジをコンテキストに常時注入するのではなく、エージェントがユーザーの意図を検知したときのみ対応するスキルをロードする仕組みです。 これによりコンテキスト消費を最小化し、応答速度と精度を向上させます。

3段階の段階的読み込み

LEVEL 1 — 常時常駐

メタデータのみ

セッション開始時に全スキルのnameとdescriptionのみがシステムプロンプトに注入される。スキル100個あっても約10,000トークン程度の軽量な状態。

LEVEL 2 — オンデマンド

SKILL.md本文の展開

ユーザーのプロンプトとdescriptionを意味論的に照合し、activate_skillツールで本文（5,000トークン以内推奨）をコンテキストに展開。

LEVEL 3 — 動的参照

リソースの動的実行

references/内のドキュメントはBashで動的読み取り。scripts/内のスクリプトはソースコードではなく実行結果のみがコンテキストに入る。実質無制限。

SKILL.md の完全テンプレート

🔑
descriptionが命： SKILL.mdのdescriptionフィールドはエージェントが「いつこのスキルを使うか」を判断するための意味的トリガーです。 「何を・いつ・なぜ」を具体的に書くほど精度が向上します。曖昧なdescriptionはスキルが全く呼ばれないか、誤った状況で呼ばれる原因になります。
.agent/skills/db-migration/SKILL.md — 完全テンプレート 
---
name: db-migration  # ← ハイフン区切り英語小文字・ディレクトリ名と完全一致させる

description: # ← 【最重要】AIのトリガー条件。具体的なキーワードを入れる！
  PostgreSQLのスキーマ変更（マイグレーション）を実行する。
  「テーブルを追加」「カラムを追加」「インデックスを作成」
  「DBのスキーマを変更して」という指示が出たら必ずこのスキルを使うこと。
---

# DB Migration Skill

## Goal（このスキルの目的）
PostgreSQL スキーマ変更をプロジェクト標準の手順で安全に実行する。

## Before Starting（前提条件）
- 変更対象のテーブル名・カラム名・データ型をユーザーから確認すること

## Step-by-Step Guide（具体的な手順）
1. `migrations/` 配下に `YYYYMMDD_HHMMSS_description.up.sql` を作成
2. ロールバック用 `.down.sql` を必ず同時に作成
3. 整合性チェックを実行: `python scripts/check.py`
4. レビュー後に適用: `python scripts/migrate.py --apply`
5. 完了後にtasks.mdの該当タスクをチェック済みにする

## Examples（Few-shot：具体的な入出力例）
# ここに例を書くとAIが期待する動作を模倣する
**Input**: "usersテーブルにlast_login_atカラムを追加して"
**Output**: migrations/20260101_add_last_login_at.up.sql を生成

## Rules（制約事項）
❌ 既存マイグレーションファイルを絶対に編集しない
❌ 本番環境への直接適用は人間確認なしに禁止
✅ ロールバックファイルは必ずセットで作成する
✅ NULL制約の後付けはデータ移行計画なしに行わない

descriptionの書き方 — 良い例 vs 悪い例

❌ アンダートリガー（呼ばれない） 
name: security-auditor
description: |
  セキュリティ監査を行うスキル。
→ AIがいつ使うか判断できない
✅ 正しいトリガー設計 
name: security-auditor
description: |
  コードのセキュリティ監査を実施する。
  脆弱性チェック、OWASP Top 10の検証、
  セキュリティ、認証、XSS、SQLi、
  という言葉が出たら必ず使うこと。
→ トリガーが明確で確実に呼ばれる

5つのスキルパターン

01

Basic Router

SKILL.mdのみ。指示と制約をテキストで記述。最もシンプル。

SKILL.md only
02

Reference Pattern

references/にドキュメントを置き、SKILL.mdから参照。静的知識の拡張。

SKILL.md + references/
03

examples/にInput/Outputのペアを複数用意。精度の向上。

SKILL.md + examples/
04

SKILL.md + scripts/
05

All-in-One Pattern

全ディレクトリを組み合わせた最強パターン。複雑な業務ロジックに対応。

Section 05

05.Rules (.agent/rules/) — 受動的制約

AntigravityのRulesは「常にバックグラウンドで動作するシステムプロンプト」です。 ファイル単位でルールを分割・個別に有効化できる点が特徴です。

Rules vs SKILL.md — 使い分け早見表

観点 Rules (.agent/rules/) SKILL.md (.agent/skills/)
起動タイミング 常時（または設定条件で） 意図が合致したときのみ
コンテキスト消費 高（毎回注入） 低（オンデマンド）
用途 コーディング規約・禁止事項 専門ワークフロー・知識
推奨内容 「〜してはいけない」ルール 「〜する方法」の手順
.agent/rules/always-on.md — 常時適用ルール 
---
activation: always
---

# Project Core Rules

## Architecture
- このプロジェクトはマイクロサービス構成（Go + gRPC）
- サービス間通信はgRPCのみ（REST禁止）

## Security Constraints
- 認証なしのエンドポイントを新規作成しない
- パスワードは必ずbcrypt (cost=12)でハッシュ化
- JWTの有効期限は1時間以内

## Forbidden Actions
- ORM使用禁止（raw SQLのみ） ← なぜ: パフォーマンス測定で20x遅延を確認した
- グローバル変数の新規追加禁止
- panic()の使用禁止（エラーを返せ）
.agent/rules/typescript.md — 言語別ルール（fileMatch使用） 
---
activation: fileMatch
pattern: "**/*.ts"  # ← TypeScriptファイル編集時のみ適用される
---

# TypeScript Coding Standards

- strict: true を必ず有効にする
- enum / namespace は使用禁止（erasableSyntaxOnly）
- エラーは throw ではなく Result<T,E> 型で返す
- never型を活用して網羅性チェックを徹底する
💡
activation: fileMatchで特定ファイルを編集するときだけルールを適用できます。 Goファイル編集時のみGoルールを、TypeScriptファイル時のみTSルールを注入することでコンテキストを節約できます。
Section 06

06.Workflows (.agent/workflows/) — 能動的手順書

WorkflowsはRulesの「受動的制約」とは対照的な「能動的手順書」です。 /workflow-name のようにスラッシュコマンドでトリガーし、エージェントが定義された手順を順番に実行します。

.agent/workflows/deploy.md — /deployコマンド 
# Deploy Workflow
# トリガー: チャットで `/deploy staging` と入力するとエージェントがこの手順を実行

## 引数
- `$ENV`: ターゲット環境 (staging | production)

## 手順
1. `git status` でコミット漏れがないことを確認する
2. テストを全件実行: `go test ./... -race`
3. テストが失敗した場合は即座に停止してユーザーに報告する
4. Dockerイメージをビルド: `docker build -t app:$(git rev-parse --short HEAD) .`
5. $ENV環境にデプロイ:
   - staging: `kubectl apply -f k8s/staging.yaml`
   - production: ユーザーに確認を求めてから実行すること
6. ヘルスチェック: `curl https://$ENV.myapp.com/health`
7. Artifacts（デプロイサマリー）を生成してユーザーに提示する
.agent/workflows/review.md — /reviewコマンド 
# Code Review Workflow

## 手順
1. `git diff main` で変更差分を取得する
2. 以下の観点でコードレビューを実施:
   - セキュリティ: SQLインジェクション・XSS・認証漏れ
   - パフォーマンス: N+1クエリ・不要なアロケーション
   - 設計: SRPの遵守・依存方向・インターフェース設計
   - テスト: エッジケースのカバレッジ
3. 問題があれば重大度(P0/P1/P2)付きでリストアップ
4. `docs/tasks.md` に未対応課題として追記する
5. Implementation Artifactとしてレビュー結果を出力
💡
ワークフロー内から別のワークフローを呼び出すことができます（チェーン化）。 例: "Ship Feature"ワークフローの中で/reviewを呼び出し、通過したら/deploy stagingを実行する構成が可能です。
Section 07

07.Artifacts — エージェント自動生成の証拠

AntigravityのArtifactsは、エージェントが自律的に作業を進める際の「信頼の証拠」として機能します。 「エージェントが何をしたか・なぜそう判断したか」をMarkdownや画像・動画で可視化することで「Trust Gap」を埋めます。

📋
Task List
エージェントがコードを書く前に生成する構造化計画。Research → Implementation → Verification のフェーズを明示。
✦ 自動生成
🏛
Implementation Plan
コードベースへの変更を事前に詳述する技術設計書。どのファイルをなぜ変更するかを人間がレビュー可能な形式で提示。
✦ 自動生成
📊
Walkthrough
完了した変更のサマリー。変更ファイル一覧・主要な設計判断の理由・テスト結果を含む。
✦ 自動生成
📸
Screenshots
Browser Subagentが撮影したUIの視覚的エビデンス。フロントエンド変更の検証に使用。
✦ Browser Agent自動撮影
🎬
Browser Recordings
UI操作のビデオ録画。E2Eフローの証明として機能。人間がレビュー後にフィードバックをインラインコメントで残せる。
✦ Browser Agent自動録画
⚠️
SDD最重要原則: Artifactを読め
多くのユーザーがTask ListやImplementation Planを「なんとなく承認」してコーディング速度を上げようとします。 しかしSDDではここが最重要レビューポイントです。 「コードは後で直せる。設計の誤りは高くつく」—— Artifactを厳密に検査することがバグを未然に防ぐ最善策です。

Section 09

09.横断ベストプラクティス 10則

01

descriptionがSKILL.mdの命

意味的トリガーとして機能するdescriptionを徹底的に具体化。「何を・いつ・なぜ」を英語・日本語問わず記述すると精度が向上する。

02

Rules ≠ SKILL.md: 用途を厳守

「常に守るルール」はRules、「特定タスクの知識」はSKILL.md。混在するとコンテキスト汚染が発生する。

03

Artifactレビューを省略しない

Task ListとImplementation Planは必ずレビュー。ここで設計ミスを検出することがバグコスト最小化の鍵。

04

fileMatchで言語別ルールを分離

activation: fileMatchを使い、Go/TS/Pythonなど言語別にルールを切り替え。全言語に共通ルールを書かない。

05

スキルは10〜15個以内に絞る

アクティブなスキルが増えすぎるとAIが混乱し精度が低下する。使わないスキルは/skills disableコマンドで無効化。

06

.context/でナレッジを蓄積する

Known GotchasをKnowledge Baseに記録。エージェントが同じ過ちを繰り返さず、ドキュメントの自動進化を促す。

07

Workflowsで繰り返し作業を撲滅

毎回同じ手順を口頭で指示するものはWorkflowに書く。/deploy、/review、/release-notesを整備する。

08

GEMINI.mdは「薄く・分割して」

@./docs/guide.mdのように@importでファイルを分割参照。1ファイルに全部詰め込まないことがコンテキスト節約の鍵。

09

Rulesには「Why」を書く

「ORM使用禁止」だけでなく「なぜ禁止か（パフォーマンス測定で20x遅延を確認）」を書くとエージェントの判断精度が上がる。

10

tasks.mdに並列情報を明示する

「Agent A担当」「Agent B担当（並列可）」を記述することでAgent Managerが最適にタスクを分配できる。

公開・運用前チェックリスト

name はハイフン区切りの英語小文字か（64文字以内）
description に具体的なトリガーキーワードが含まれているか
SKILL.md 本文は 500 行以内か（参照資料は references/ に分離済みか）
スクリプトはハードコードされた値ではなく引数で動作変更できるか
危険な操作のスキルに disable-model-invocation: true が設定されているか
Few-shot の Examples セクションが含まれているか（入出力の例示）
Rules セクションで ❌禁止事項 と ✅推奨行動 が明確に定義されているか
アクティブなスキルの合計数が 15 個以内に収まっているか
ワークスペーススキルはチームで共有できるようにGitにコミットされているか
AGENTS.md にクロスツール共有ルールを記述し他ツールでも動作確認したか（v1.20.3〜）
