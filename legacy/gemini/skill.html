# Global Developer Profile

## 🎯 My Working Style
# エージェントへの"個人プロファイル"を記述
- コードレビューはQuality > Speed
- TDD優先（テストを先に書く）
- コメントは英語、コミットはConventional Commits形式

## 🛠 Global Toolchain
# 全プロジェクト共通で使うツール
- フォーマッター: Prettier / gofmt / Black
- Lint: ESLint / golangci-lint / ruff
- テスト: Vitest / Go test / pytest

## ⚠️ Global Constraints
# 絶対禁止ルール（プロジェクト横断）
- 本番DBへのDELETE/DROPは必ず人間確認
- .envファイルをコミットしない
- console.log()はコードレビュー前に削除

## 🤖 Agent Behavior Preferences
- タスク開始前にImplementation Planを必ず生成すること
- 不確実な場合は実装前に確認（"Always Proceed"禁止）
- git commitはタスク単位で細かく行う
⚠️
アンチパターン: GEMINI.mdにプロジェクト固有のDB設定やAPIエンドポイントを書かない。グローバルファイルはプロジェクト固有情報を持つべきではありません。プロジェクト情報は.agent/rules/または.context/へ。

04.Rules (.agent/rules/) — 受動的制約・バックグラウンドシステムプロンプト

AntigravityのRulesは「常にバックグラウンドで動作するシステムプロンプト」です。Claude CodeのCLAUDE.mdとは異なり、ファイル単位でルールを分割・個別に有効化できる点が特徴です。

📜
Rules Files
.agent/rules/*.md

Rules vs SKILL.md — 使い分け早見表

観点 Rules (.agent/rules/) SKILL.md (.agent/skills/)
起動タイミング 常時（または設定条件で） 意図が合致したときのみ
コンテキスト消費 高（毎回注入） 低（オンデマンド）
用途 コーディング規約・禁止事項 専門ワークフロー・知識
推奨内容 「〜してはいけない」ルール 「〜する方法」の手順

ベストプラクティス: ルールファイルを機能別に分割

.agent/rules/always-on.md — 常時適用ルール 
---
activation: always
---

# Project Core Rules

## Architecture
- このプロジェクトはマイクロサービス構成（Go + gRPC）
- サービス間通信はgRPCのみ（REST禁止）
- 各サービスは独自のPostgreSQLスキーマを持つ

## Security Constraints
- 認証なしのエンドポイントを新規作成しない
- パスワードは必ずbcrypt (cost=12)でハッシュ化
- JWTの有効期限は1時間以内

## Forbidden Actions
- ORM使用禁止（raw SQLのみ）
- グローバル変数の新規追加禁止
- panic()の使用禁止（エラーを返せ）
.agent/rules/go-style.md — 言語別ルール（Go専用） 
---
activation: fileMatch
pattern: "**/*.go"
---

# Go Coding Standards

- エラーハンドリング: errors.Wrap()を必ず使用
- テスト: table-driven tests が標準
- インターフェース: 受け取る側（呼び出し元）で定義
- コンテキスト: 全関数の第1引数はctx context.Context
- ゴルーチンリーク: defer cancel()を必ず書く
💡
activation: fileMatch で特定ファイルを編集するときだけルールを適用できます。Goファイル編集時のみGoルールを、TypeScriptファイル時のみTSルールを注入することでコンテキストを節約できます。
Section 05

05.SKILL.md — 進歩的開示ナレッジ（最重要）

AntigravityのSkillsは「Progressive Disclosure（進歩的開示）」という設計思想に基づきます。全てのナレッジをコンテキストに常時注入するのではなく、エージェントがユーザーの意図を検知したときのみ対応するスキルをロードする仕組みです。これによりコンテキスト消費を最小化し、応答速度と精度を向上させます。

🎓
SKILL.md
.agent/skills/<skill-name>/SKILL.md

スキルディレクトリ構造

スキルの標準ディレクトリ構造
.agent/skills/
└── db-migration/
    ├── SKILL.md        # 必須: メタデータ + 指示（スキルの「脳」）
    ├── scripts/        # 任意: Python/Bash/Goスクリプト
    │   └── run_migration.py
    ├── references/     # 任意: ドキュメント・テンプレート
    │   └── migration-spec.md
    └── assets/         # 任意: 静的ファイル

SKILL.mdの完全テンプレート

.agent/skills/db-migration/SKILL.md 
---
name: db-migration
# ↑ スキルの一意識別子（ハイフン区切り）
description: >
  Executes database schema migrations for PostgreSQL using the
  project's standard migration protocol. Use when the user asks
  to add tables, columns, indexes, or modify the DB schema.
# ↑ 【最重要】意味的トリガーフレーズ。
#   「いつ使うか」を具体的に書くほど精度が上がる
---

# Database Migration Skill

## Goal
PostgreSQLスキーマ変更を、プロジェクト標準のマイグレーション
プロトコルに従って安全に実行する。

## Instructions
1. `migrations/`配下に `YYYYMMDD_HHMMSS_description.up.sql` を作成
2. ロールバック用 `.down.sql` を必ず同時に作成
3. スクリプトで整合性チェックを実行:
   `python scripts/run_migration.py --check --env staging`
4. レビュー後に適用: `python scripts/run_migration.py --apply`
5. 完了後にtasks.mdの該当タスクをチェック済みにする

## Examples
# Few-shot examples でエージェントの精度を向上させる

**Input**: "usersテーブルにlast_login_atカラムを追加して"
**Output**: 
```sql
-- 20250118_143022_add_last_login_at_to_users.up.sql
ALTER TABLE users ADD COLUMN last_login_at TIMESTAMPTZ;
CREATE INDEX idx_users_last_login ON users(last_login_at);
```

## Constraints
- 既存マイグレーションファイルを絶対に編集しない
- NULL制約の後付けはデータ移行計画なしに行わない
- 本番環境への直接適用は人間の承認が必要
- `DROP TABLE` は人間確認なしに実行禁止

5つのスキルパターン

01

Basic Router

SKILL.mdのみ。指示と制約をテキストで記述。最もシンプル。

SKILL.md only
02

Reference Pattern

references/にドキュメントを置き、SKILL.mdから参照。静的知識の拡張。

SKILL.md + references/
03

Few-shot Pattern

examples/にInput/Outputのペアを複数用意。精度の向上。

SKILL.md + examples/
04

Tool Use Pattern

scripts/に実行可能なスクリプトを置く。LLMが苦手な計算・DB操作を外出し。

SKILL.md + scripts/
05

All-in-One Pattern

全ディレクトリを組み合わせた最強パターン。複雑な業務ロジックに対応。

SKILL.md + all dirs

グローバルvs.ワークスペーススキル

🌐 グローバルスキル

~/.gemini/antigravity/skills/（Antigravity IDE）
~/.gemini/skills/（Gemini CLI）
「JSON整形」「UUID生成」「コードレビュー」など、全プロジェクトで使う汎用スキル。GitHubからスキルセットをクローンして配置可能。

📁 ワークスペーススキル

<project>/.agent/skills/
「このプロジェクトのデプロイ手順」「社内フレームワークのボイラープレート生成」など、プロジェクト固有のスキル。チームでgit管理。

🔑
descriptionが命: SKILL.mdのdescriptionフィールドはエージェントが「いつこのスキルを使うか」を判断するための意味的トリガーです。「Executes read-only SQL queries to retrieve user data for debugging」のように「何を・いつ・なぜ」を具体的に書くほど精度が向上します。曖昧なdescriptionはスキルが全く呼ばれないか、誤った状況で呼ばれる原因になります（[11]）。

06.Workflows (.agent/workflows/) — 能動的手順書

WorkflowsはRulesの「受動的制約」とは対照的な「能動的手順書」です。/workflow-name のようにスラッシュコマンドでトリガーし、エージェントが定義された手順を順番に実行します。繰り返し実施するタスク（デプロイ・レビュー・リリースノート作成等）を自動化するのに最適です。

⚡
Workflow Files
.agent/workflows/*.md
.agent/workflows/deploy.md — /deployコマンド 
# Deploy Workflow
# トリガー: `/deploy` と入力するとエージェントがこの手順を実行

## 引数
- `$ENV`: ターゲット環境 (staging | production)

## 手順
1. `git status` でコミット漏れがないことを確認する
2. テストを全件実行: `go test ./... -race`
3. テストが失敗した場合は即座に停止してユーザーに報告する
4. Dockerイメージをビルド: `docker build -t app:$(git rev-parse --short HEAD) .`
5. イメージをプッシュ: `docker push gcr.io/myproject/app:$(git rev-parse --short HEAD)`
6. $ENV環境にデプロイ:
   - staging: `kubectl apply -f k8s/staging.yaml`
   - production: ユーザーに確認を求めてから実行
7. ヘルスチェックエンドポイントを確認: `curl https://$ENV.myapp.com/health`
8. Artifacts（デプロイサマリー）を生成してユーザーに提示する
.agent/workflows/review.md — /reviewコマンド 
# Code Review Workflow

## 手順
1. `git diff main` で変更差分を取得する
2. 以下の観点でコードレビューを実施:
   - セキュリティ: SQLインジェクション・XSS・認証漏れ
   - パフォーマンス: N+1クエリ・不要なアロケーション
   - 設計: SRPの遵守・依存方向・インターフェース設計
   - テスト: エッジケースのカバレッジ
3. 問題があれば重大度(P0/P1/P2)付きでリストアップ
4. `docs/tasks.md` に未対応課題として追記する
5. Implementation Artifactとしてレビュー結果を出力

ワークフローのチェーン化

💡
ワークフロー内から別のワークフローを呼び出すことができます（チェーン化）。例: "Ship Feature"ワークフローの中で/reviewを呼び出し、通過したら/deploy stagingを実行する構成が可能です（[15]）。

07.Knowledge Base (.context/) — 自動注入コンテキスト

Antigravityのもう一つの強力な機能がKnowledge Base（ナレッジベース）です。エージェントが作業を通じて学習したパターン・プロジェクト固有の慣習・重要な洞察を.context/配下のMarkdownファイルに保存し、次セッションで自動的に参照します。

🧠
Knowledge Base
.context/*.md
🤖 自動更新

エージェントが作業中に「有用なパターン」を発見すると自動的にナレッジベースに追記。Claude CodeのMEMORY.mdに相当。

✍️ 手動追加も可

人間が直接.context/配下にMarkdownファイルを作成・編集できる。特定の知識を確実に記録したい場合に使用。

📊 セマンティック検索

ナレッジベースはベクター埋め込みで検索されるため、「DB」に関するタスクで自動的にapi-conventions.mdが参照される。

.context/architecture.md — アーキテクチャコンテキスト 
# System Architecture Context
# このファイルはエージェントと人間が共同で保守する

## Current Stack
- Backend: Go 1.23 + gRPC
- Frontend: Next.js 15 (App Router)
- DB: PostgreSQL 16 + Redis 8 (Valkey互換)
- Infra: GKE + Cloud Run + Cloud SQL

## Key Architectural Decisions
- サービス間通信はgRPCのみ（REST APIはクライアント向けのみ）
- DBコネクションプールはpgx v5を使用（database/sqlではない）
- キャッシュキーの命名規則: `{service}:{entity}:{id}`

## Known Gotchas
# エージェントが発見した落とし穴を自動記録
- Cloud SQLのUnix socket接続は/cloudsql/{conn-name}を使う
- gRPCのDeadlineはcontext経由で伝播させること（直接設定禁止）
- Cloud RunのCOLD START: 初回リクエストで500ms遅延が発生する
ℹ️
ナレッジベースはproduct-guidelines.mdのような重厚なファイルを人間が手動で維持する必要を大幅に削減します。エージェントが自律的に学習・記録するため「コードとドキュメントの乖離」問題が軽減されます（[13]）。
Section 08

08.Artifacts — エージェント自動生成マークダウン

AntigravityのArtifactsは、エージェントが自律的に作業を進める際の「信頼の証拠」として機能します。「エージェントが何をしたか・なぜそう判断したか」をMarkdownや画像・動画で可視化することで「Trust Gap」を埋めます。SDDの文脈では人間がレビューすべきゲートポイントとして重要です。

📋
Task List
エージェントがコードを書く前に生成する構造化計画。Research → Implementation → Verification のフェーズを明示。
✦ 自動生成
🏛
Implementation Plan
コードベースへの変更を事前に詳述する技術設計書。どのファイルをなぜ変更するかを人間がレビュー可能な形式で提示。
✦ 自動生成
📊
Walkthrough
完了した変更のサマリー。変更ファイル一覧・主要な設計判断の理由・テスト結果を含む。
✦ 自動生成
📸
Screenshots
Browser Subagentが撮影したUIの視覚的エビデンス。フロントエンド変更の検証に使用。
✦ Browser Agent自動撮影
🎬
Browser Recordings
UI操作のビデオ録画。E2Eフローの証明として機能。人間がレビュー後にフィードバックをインラインコメントで残せる。
✦ Browser Agent自動録画
⚠️
SDD最重要原則: Artifactを読め
多くのユーザーがTask ListやImplementation Planを「なんとなく承認」してコーディング速度を上げようとします。しかしSDDではここが最重要レビューポイントです。「コードは後で直せる。設計の誤りは高くつく」— Artifactを厳密に検査することがバグを未然に防ぐ最善策です（[15]）。

AntigravityのSDD（仕様駆動開発）では、spec.md / requirements.md / design.md / tasks.md を「エージェントがArtifactを生成する基礎資料」として活用します。エージェントが自律的にImplementation Planを作成するためにこれらのファイルを自動参照します。

📚

Antigravity流SDDの特徴: 「Rulesで仕様書を自動参照させる」

.agent/rules/always-on.md（仕様書参照ルール追加） 
## Spec-Driven Development Rules
# Rulesに仕様書の存在と読み込み順序を明示

新機能を実装する前に必ず以下を確認すること:
1. `docs/spec.md` でプロダクトのスコープを確認
2. `docs/requirements.md` で対象機能の要件を確認
3. `docs/design.md` でアーキテクチャ制約を確認
4. `docs/tasks.md` で依存タスクが完了済みか確認

実装後は `docs/tasks.md` の対象タスクをチェック済みにすること。
仕様書と実装が乖離した場合は、人間に確認を求めること。
docs/tasks.md — マルチエージェント対応版 
# タスクリスト（Agent Manager対応）

## 🟢 Phase 1: 認証基盤（Agent A担当）
- [x] Task 1.1: JWTミドルウェア実装

Section 10

10.横断ベストプラクティス 10則

01

descriptionがSKILL.mdの命

意味的トリガーとして機能するdescriptionを徹底的に具体化。「何を・いつ・なぜ」を英語で記述すると精度が向上する。

02

Rules ≠ SKILL.md: 用途を厳守

「常に守るルール」はRules、「特定タスクの知識」はSKILL.md。混在するとコンテキスト汚染が発生する。

03

Artifactレビューを省略しない

Task ListとImplementation Planは必ずレビュー。ここで設計ミスを検出することがバグコスト最小化の鍵。

04

fileMatchで言語別ルールを分離

activation: fileMatchを使い、Go/TS/Pythonなど言語別にルールを切り替え。全言語に共通ルールを書かない。

05

Global vs Workspace の使い分け

汎用スキル（JSON整形・git commit等）はグローバル。プロジェクト固有（内製フレームワーク等）はワークスペース。

06

tasks.mdに並列情報を明示

「Agent A担当」「Agent B担当（並列可）」を記述することでAgent Managerが最適にタスクを分配できる。

07

.context/でspec乖離を防ぐ

Known GotchasをKnowledge Baseに記録。エージェントが同じ過ちを繰り返すことを防ぎ、ドキュメントの自動進化を促す。

08

Workflowsで繰り返しを撲滅

毎回同じ手順を口頭で指示するものはWorkflowに書く。/deploy、/review、/release-notes を整備する。

09

少数精鋭のスキルから始める

スキルは5〜10個から始め、実際に使われたものを磨く。使われないスキルはdescriptionを改善するか削除する。

10

Rulesは「Why」を書く

「ORM使用禁止」だけでなく「なぜ禁止か（パフォーマンス測定で20x遅延を確認したため）」を書くとエージェントの判断精度が上がる。
