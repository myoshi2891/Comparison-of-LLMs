CH 01 K-Level: K1–K2

SonarQube の全体像とアーキテクチャ

1.1 定義：SonarQube とは何か

SonarQube は、静的解析（Static Analysis）によってソースコードの品質・セキュリティ・保守性を自動的に評価する 継続的インスペクション（Continuous Inspection） プラットフォームである。コードを実行せず、ソースコードを直接解析することで、バグ・脆弱性・コードスメルを早期に検出する。

75%Fortune 100 企業の採用率
750B+毎日解析されるコード行数
30+対応プログラミング言語数
6,000+組み込みルール数

1.2 理由：なぜ SonarQube が必要か

従来のコードレビューは人間のレビュアーに依存し、一貫性・速度・網羅性に限界がある。SonarQube は以下の課題を解決する：

人的レビューの限界見落とし・疲労・属人化 → SonarQube で自動化・標準化
技術的負債の蓄積Clean as You Code で新規コードの品質を常に維持
セキュリティリスクOWASP/CWE対応の Taint Analysis で脆弱性を早期検出
CI/CDとの断絶Quality Gate でパイプラインを自動ブロック・制御

CH 02 K-Level: K2–K3

コアコンセプト：CaYC / MQR / Quality Gate

2.1 Clean as You Code（CaYC）

定義： 既存のレガシーコード全体を一度に修正しようとせず、新規コード（New Code）に常に高い品質基準を課す戦略。開発者は自分が追加・変更したコードのみに責任を持つ。

理由： 技術的負債を段階的に解消しつつ、今日書くコードを常にクリーンに保つことで、時間をかけてコードベース全体の品質が向上する。

flowchart LR subgraph DEF["New Code の定義オプション"] A["直近 N 日間のコード変更"] B["特定ブランチとの差分"] C["前回バージョン以降の変更"] end subgraph GATE["Sonar way Quality Gate の条件"] D["新規 Issue: 0件"] E["新規脆弱性: 0件"] F["Hotspot 全件レビュー済み: 100%"] G["カバレッジ 80%以上"] H["重複率 3%以下"] end A & B & C --> D & E & F & G & H

2.2 インスタンスモード：Standard vs MQR

定義： SonarQube Server 2026 では2つのモードが存在する。Standard Experience はバグ・脆弱性・Code Smell という従来分類。MQR（Multi-Quality Rule）モードはルールを セキュリティ・信頼性・保守性 という品質特性の軸で評価し、1つのルールが複数の品質特性に影響するケースを正確に表現できる。

観点 Standard Experience MQR Mode（推奨）
ルール分類 Bug / Vulnerability / Code Smell Security / Reliability / Maintainability
Severity 段階 Blocker, Critical, Major, Minor, Info Blocker, High, Medium, Low, Info
1ルールの影響 単一カテゴリに1 severity 複数品質特性に別々の severity を付与
移行 Administration → General Settings → Mode 同左（admin 権限必要）
⚠️ MQR モードへの切り替え後、1つの Issue が複数品質特性に紐付くため Issue 数が増加することがある。カスタム Quality Gate の条件も更新が必要。

2.3 主要メトリクス定義

メトリクス 定義 良好値の目安 モード
Coverage テストが実行するコード行の割合 新規コード ≥ 80% 両モード
Duplications 重複コードブロックの割合 新規コード ≤ 3% 両モード
Reliability Rating バグの深刻度に基づく A〜E 評価 A（バグなし） Standard
Security Rating 脆弱性の深刻度に基づく A〜E 評価 A（脆弱性なし） Standard
Maintainability Rating Code Smell 修正工数率に基づく A〜E A（≤ 5%） Standard
Software Quality Severity 各品質特性への影響深刻度 Blocker/High: 0件 MQR
SQALE Index 技術的負債の推定修正工数 プロジェクト規模による 両モード

2.4 Issue 深刻度のピラミッド

Info — 参考情報（要対応なし）
Minor — 小規模なスタイル問題
Major — 保守性・品質に影響
Critical — バグ・セキュリティリスク
Blocker — 即時修正必須

3.2 sonar-project.properties の詳細設定

定義： プロジェクトルートに配置する設定ファイル。スキャナーの動作をコードとして管理できる（Infrastructure as Code 的アプローチ）。

PROPERTIES — sonar-project.properties 
# ===== プロジェクト識別子 =====
sonar.projectKey=com.example:my-service
sonar.projectName=My Service
sonar.projectVersion=1.0.0

# ===== ソース設定 =====
sonar.sources=src/main
sonar.tests=src/test
sonar.sourceEncoding=UTF-8

# ===== Java 固有設定 =====
sonar.java.binaries=target/classes
sonar.java.libraries=target/dependency/*.jar
sonar.java.test.binaries=target/test-classes

# ===== カバレッジ レポート =====
sonar.coverage.jacoco.xmlReportPaths=target/site/jacoco/jacoco.xml

# ===== 解析除外（自動生成・サードパーティ） =====
sonar.exclusions=**/generated/**,**/vendor/**,**/*.min.js,**/node_modules/**
sonar.coverage.exclusions=**/config/**,**/dto/**,**/entity/**
sonar.test.exclusions=**/testdata/**

# ===== New Code 定義（プロジェクトレベル） =====
# previous_version | NUMBER_OF_DAYS | reference_branch
sonar.newCode.referenceBranch=main

3.3 トークン管理：良い例 vs 悪い例

✅ 推奨：環境変数経由
# CI シークレットとして設定
# SONAR_TOKEN=sqp_xxxxx

sonar-scanner \
  -Dsonar.token="${SONAR_TOKEN}" \
  -Dsonar.host.url="${SONAR_HOST_URL}"
❌ NG：ファイルにハードコード
# sonar-project.properties に直書き
# → Git 履歴にトークンが残る！

sonar.login=sqp_ab12cd34ef56
sonar.host.url=https://sonar.example.com

3.4 トークン種別と最小権限の原則

トークン種別 スコープ 推奨用途
User Token ユーザー権限相当 ローカル開発・デバッグ
Project Analysis Token 単一プロジェクトの解析のみ ✅ CI/CD パイプライン（最小権限）
Global Analysis Token 全プロジェクトの解析 中央集権 CI（使用は最小限に）
⚠️ CI/CD では必ず Project Analysis Token を使用すること。万が一の漏洩時の影響範囲を最小化できる。

4.2 GitHub Actions

ℹ️

重要：fetch-depth: 0 は必須

浅いクローン（デフォルト fetch-depth: 1）では Git 履歴が不足し、New Code の検出精度が大幅に低下する。必ず fetch-depth: 0 を指定すること。

YAML — .github/workflows/sonarqube.yml 
name: SonarQube Analysis

on:
  push:
    branches: [main, develop]
  pull_request:
    types: [opened, synchronize, reopened]

jobs:
  sonarqube:
    name: SonarQube Scan
    runs-on: ubuntu-latest

    steps:
      - name: Checkout repository
        uses: actions/checkout@v4
        with:
          fetch-depth: 0  # ← 完全 Git 履歴（必須！）

      - name: Set up JDK 21
        uses: actions/setup-java@v4
        with:
          java-version: '21'
          distribution: 'temurin'
          cache: 'maven'

      - name: Cache SonarQube packages
        uses: actions/cache@v4
        with:
          path: ~/.sonar/cache
          key: ${{ runner.os }}-sonar
          restore-keys: ${{ runner.os }}-sonar

      - name: Build and run tests with coverage
        run: mvn -B verify -Pcoverage

      - name: SonarQube Scan
        uses: SonarSource/sonarqube-scan-action@v5
        env:
          SONAR_TOKEN: ${{ secrets.SONAR_TOKEN }}
          SONAR_HOST_URL: ${{ secrets.SONAR_HOST_URL }}

      - name: Quality Gate check
        id: sonarqube-quality-gate-check
        uses: SonarSource/sonarqube-quality-gate-action@v1
        timeout-minutes: 5
        env:
          SONAR_TOKEN: ${{ secrets.SONAR_TOKEN }}
          SONAR_HOST_URL: ${{ secrets.SONAR_HOST_URL }}

      - name: Fail if Quality Gate failed
        if: steps.sonarqube-quality-gate-check.outputs.quality-gate-status == 'FAILED'
        run: exit 1

4.3 Jenkins Declarative Pipeline

GROOVY — Jenkinsfile 
pipeline {
    agent any
    environment {
        SONAR_ENV = 'SonarQube'  // Jenkins 管理画面での名前
    }
    stages {
        stage('Build & Test') {
            steps {
                sh 'mvn -B clean verify -Pcoverage'
            }
        }
        stage('SonarQube Analysis') {
            steps {
                withSonarQubeEnv(env.SONAR_ENV) {
                    sh 'mvn sonar:sonar'
                }
            }
        }
        stage('Quality Gate') {
            steps {
                // Webhook 使用でノンブロッキング待機（推奨）
                timeout(time: 5, unit: 'MINUTES') {
                    waitForQualityGate abortPipeline: true
                }
            }
        }
    }
    post {
        failure {
            slackSend(color: 'danger',
              message: "❌ Quality Gate Failed: ${env.JOB_NAME} #${env.BUILD_NUMBER}")
        }
    }
}
⚙️

Jenkins Webhook の設定が必須

SonarQube 管理画面 → Administration → Configuration → Webhooks で http://<jenkins-host>/sonarqube-webhook/ を登録すること。Webhook がないと waitForQualityGate がタイムアウトする。

4.5 Quality Gate 待機方式の比較

方式 利点 欠点 推奨シーン
sonar.qualitygate.wait=true 設定シンプル、どのCIでも動作 スキャナーがポーリング（時間・CPU消費） 小規模・シンプルな構成
Quality Gate Action（GitHub） 非同期・効率的 GitHub Actions 依存 GitHub Actions
waitForQualityGate（Jenkins） Webhook 活用で軽量、再起動耐性あり Webhook 設定が必要 Jenkins
外部 API ポーリング（自作） CI ツール非依存 実装コスト高 独自 CI 基盤
CH 05 K-Level: K3

Pull Request 解析とデコレーション

定義： PR 解析は、プルリクエストが開かれた時点、およびプッシュがあるたびに自動実行される。PR ブランチとターゲットブランチを比較し、差分コードに導入された Issue のみ を報告する。Developer Edition 以上で利用可能。

5.1 PR 解析シーケンス

sequenceDiagram participant Dev as Developer participant SCM as GitHub or GitLab participant CI as CI Pipeline participant SS as SonarScanner participant SQ as SonarQube Dev->>SCM: PR 作成 / コミットプッシュ SCM->>CI: Webhook でパイプライン起動 CI->>CI: Build and テスト実行 CI->>SS: SonarScanner 起動 (PR パラメータ付き) SS->>SQ: 解析レポート送信 SQ->>SQ: PRブランチ vs targetブランチ差分解析 Note over SQ: 差分コード of Issue のみ報告 SQ->>SCM: Quality Gate 結果をデコレーション Note over SCM: PR ページにインラインコメント表示 SQ-->>CI: QG ステータスを返答 CI->>Dev: パイプライン結果通知

5.2 PR パラメータの手動設定

多くの CI 環境ではパラメータを自動検出するが、カスタム CI 環境では手動設定が必要な場合がある。

BASH — 手動 PR パラメータ 
sonar-scanner \
  -Dsonar.pullrequest.key=${PR_NUMBER} \         # PR の一意識別子
  -Dsonar.pullrequest.branch=${SOURCE_BRANCH} \  # feature ブランチ名
  -Dsonar.pullrequest.base=${TARGET_BRANCH} \   # マージ先（main 等）
  -Dsonar.pullrequest.provider=GitHub \
  -Dsonar.pullrequest.github.repository=org/repo

自動検出対応環境： GitHub Actions GitLab CI Jenkins（Multibranch） CircleCI Azure Pipelines Bitbucket Pipelines

5.3 PR 解析の重要な挙動と制約

1
差分コードのみを対象

PR ブランチで追加・編集されたコードの Issue のみ報告。既存コードの Issue は報告されない（Issue バックデーティングが適用外）。

2
マージ後の新規 Issue の可能性

マージ後の最初のメインブランチ解析で、PR 解析が未検出の旧コード上の Issue が出現することがある。

3
30日間の自動パージ

PR 解析結果はデフォルト30日間保持。Administration → Housekeeping で変更可能。

4
Issue 属性の同期

Issue のステータス・アサイン・コメントはターゲットブランチの対応 Issue と同期される。

✅ PR 解析では 差分コードに絞ることで解析時間を短縮するために差分インクリメンタル解析機構が採用されている。
CH 06 K-Level: K4

Quality Gates の高度な設計

定義： Quality Gate はコードを一連の条件（Condition）で評価し「プロジェクトはリリース可能か？」という問いに答える。各条件はメトリクスと閾値の組み合わせで構成される。

6.1 Quality Gate 評価フロー

flowchart TD ANALYSIS["解析完了"] --> EVAL["Quality Gate 全条件を評価"] EVAL --> C1{"新規 Issue = 0件?"} EVAL --> C2{"Security Hotspot 全件レビュー済み?"} EVAL --> C3{"カバレッジ >= 閾値%?"} EVAL --> C4{"重複率 <= 閾値%?"} C1 -->|OK| P1["OK"] C1 -->|NG| F1["ERROR"] C2 -->|OK| P2["OK"] C2 -->|NG| F2["ERROR"] C3 -->|OK| P3["OK"] C3 -->|NG| F3["ERROR"] C4 -->|OK| P4["OK"] C4 -->|NG| F4["ERROR"] P1 & P2 & P3 & P4 --> PASS["PASSED (デプロイ可能 / PR マージ許可)"] F1 --> FAIL["FAILED (パイプライン停止 / マージブロック)"] F2 --> FAIL F3 --> FAIL F4 --> FAIL

6.2 Sonar way Quality Gate の条件（デフォルト）

条件 メトリクス 演算子 閾値 変更可否
新規 Issue ゼロ Issues is greater than 0 🔒 固定
Hotspot 全件レビュー Security Hotspots Reviewed is less than 100% 🔒 固定
信頼性評価 Reliability Rating is worse than A 🔒 固定
セキュリティ評価 Security Rating is worse than A 🔒 固定
テストカバレッジ Coverage is less than 80% ✏️ 変更可
重複率 Duplicated Lines (%) is greater than 3% ✏️ 変更可

6.3 プロジェクト特性別 Quality Gate 設計

🏭 本番サービス（厳格型）
★★★★★
Coverage ≥ 85%、重複 ≤ 2%、Blocker Issue 0件絶対。セキュリティ評価 A 必須。リリース頻度が高く、ダウンタイムのコストが大きいサービス向け。
🚀 スタートアップ MVP（段階的型）
★★★☆☆
Coverage ≥ 60% から開始し、フェーズごとに締め付け。Blocker のみブロック、Major 以下は警告のみ。スピードと品質のバランスを取る。
🔬 実験的プロジェクト（緩和型）
★★☆☆☆
Hotspot レビューのみ必須。Coverage 要件なし。PoC・研究目的のプロジェクトで、品質よりも開発速度を優先するケース。

6.4 段階的締め付け戦略（レガシー移行期）

Phase 1（導入期）: Coverage ≥ 50%50%
Phase 2（安定期）: Coverage ≥ 70%70%
Phase 3（成熟期）: Coverage ≥ 80%80%
Phase 4（最終目標）: Coverage ≥ 85%85%
CH 07 K-Level: K4–K6

Quality Profiles とカスタムルール

定義： Quality Profile は解析時に適用するルールセットを定義する設定。プロジェクト単位・言語単位で割り当て可能。Sonar way は SonarSource が提供する読み取り専用の推奨プロファイル。

7.1 継承戦略：Sonar way を直接変更しない

flowchart TD SW["Sonar way (SonarSource 提供・読み取り専用)"] SW -->|extends| BASE["Org Base Profile (組織共通ルール追加)"] BASE -->|extends| BE["Backend Profile (Java / Go / Python 固有)"] BASE -->|extends| FE["Frontend Profile (TypeScript / JavaScript / CSS)"] BASE -->|extends| SEC["Security Profile (セキュリティ強化ルール追加)"] BE -->|プロジェクト割り当て| P1["Service A"] BE -->|プロジェクト割り当て| P2["Service B"] FE -->|プロジェクト割り当て| P3["Web App"] SEC -->|プロジェクト割り当て| P4["Payment Service"]
✅ 継承により、Sonar way が更新された際の新ルールを子プロファイルが自動的に受け継ぐ。Sonar way を直接コピーしてカスタマイズすると、この自動更新の恩恵を失う。

7.2 カスタムルールの追加（3 つの方法）

方法 1：XPath ルール（UIから即座に作成）

方法 2：Java Plugin API（フル機能・推奨）
JAVA — カスタムルール実装例 
@Rule(key = "NoSystemOutPrintlnRule")
public class NoSystemOutPrintlnRule
    extends IssuableSubscriptionVisitor {

  // 監視するAST ノード種別を宣言
  @Override
  public List<Tree.Kind> nodesToVisit() {
    return Collections.singletonList(Tree.Kind.EXPRESSION_STATEMENT);
  }

  @Override
  public void visitNode(Tree tree) {
    ExpressionStatementTree expr = (ExpressionStatementTree) tree;
    String text = expr.firstToken().text();
    if (text.contains("System.out.println")) {
      reportIssue(tree,
        "System.out.println を使用しないでください。" +
        "ロガーを使用してください。");
    }
  }
}

セキュリティ解析の徹底活用

8.1 Security Hotspot レビューワークフロー

定義： Security Hotspot は脆弱性ではなく「セキュリティに敏感なコード」。深刻度は付与されず、必ず人間によるレビューが必要。レビュー後に Safe または Vulnerability に分類される。

flowchart TD DETECT["SonarQube が Security Hotspot を検出"] --> ASSIGN["担当者にアサイン (自動割り当て)"] ASSIGN --> REVIEW["コードレビュー実施 (脅威シナリオを確認)"] REVIEW --> SAFE{"実際に脅威あり?"} SAFE -->|脅威なし| MARK_SAFE["Safe としてマーク (根拠を記録)"] SAFE -->|脅威あり| FIX["コードを修正 (脆弱性として扱う)"] SAFE -->|判断不能| ESCALATE["セキュリティ専門家へエスカレーション"] MARK_SAFE --> CLOSED["Hotspot クローズ (QG の条件に算入)"] FIX --> REANALYSIS["再解析 -> Vulnerability として追跡"] ESCALATE --> REVIEW

Security-Injectionインジェクション脆弱性を検出。Taint Analysis で汚染データの流れを追跡（SQL/XSS/SSRF 等）
Security-Configuration誤った設定を検出（無効な暗号化アルゴリズム・TLS バージョン・権限チェック漏れ等）

9.4 Context Augmentation の 4 カテゴリ

📖
ガイドライン (Guidelines)
エージェントのプロンプト内容に基づき、関連する Sonar ルールを LLM コンテキストに挿入。DB アクセス操作なら SQL インジェクション関連ルールを自動注入。
🔭
コードナビゲーション
テキスト検索ではなく、コールスタック・クラス階層・参照情報に基づきコードベースを意味的に探索。エージェントが正確な箇所を修正できる。
📦
依存関係ガイダンス
新規ライブラリ追加前に、既知の脆弱性・サプライチェーンマルウェア・ライセンス準拠を事前確認（SCA 機能）。
🧬
コーディング標準
プロジェクト固有の過去 Issue 傾向を分析し、エージェントが行おうとしている操作に関連する標準ルールを推奨。

9.5 Remediation Agent（Beta）

🤖

Remediation Agent とは

SonarQube Cloud の Team（年間）/ Enterprise プランで利用可能。PR の Issue を自動修正する AI エージェント。「Assign to Agent」ボタンをクリックするとエージェントが独立して Issue を解析・修正 PR を作成し、開発者がレビューして承認する。Reliability・Maintainability Issue に対応（Beta フェーズ）。

⚠️ Security Issue への自動修正は必ず人間の最終確認が必須。Remediation Agent の提案は参考情報として扱い、盲目的に適用しないこと。

10.3 Issue サマリーの自動生成（Python）

PYTHON — New Code Issue サマリー取得 
import requests
import os

SONAR_TOKEN = os.environ["SONAR_TOKEN"]
SONAR_HOST  = os.environ["SONAR_HOST_URL"]
PROJECT_KEY = "com.example:my-service"

def get_new_code_issues(branch: str = "main") -> dict:
    """新規コードの Issue サマリーを取得"""
    params = {
        "componentKeys": PROJECT_KEY,
        "resolved": "false",
        "sinceLeakPeriod": "true",  # New Code のみ
        "branch": branch,
        "facets": "types,severities",
        "ps": 1  # カウントのみ必要
    }
    resp = requests.get(
        f"{SONAR_HOST}/api/issues/search",
        params=params,
        auth=(SONAR_TOKEN, "")
    )
    resp.raise_for_status()
    return resp.json()

issues = get_new_code_issues()
print(f"New Code Issues: {issues['total']}")
for facet in issues.get("facets", []):
    if facet["property"] == "types":
        for v in facet["values"]:
            print(f"  {v['val']}: {v['count']}")

11.6 チーム文化への定着ポイント

1
「最初の分析は穏やかに」

導入直後は Issue 数が膨大になりがち。最初の QG は緩く設定し、チームの恐怖感を取り除く。

プルリクエスト文化と組み合わせる

ダッシュボードを可視化する

4
定期的なルールレビュー

Quality Gate の条件は四半期ごとにレビュー。ビジネス要件・チームの成熟度に応じて段階的に引き上げる。
